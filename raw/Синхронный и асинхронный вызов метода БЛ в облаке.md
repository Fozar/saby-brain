Оглавление

При асинхронном вызове [метода БЛ](https://link.sbis.ru/article/b5386ab7-fe67-4f3c-8da5-0f9ac822bb0b) программа не дожидается выполнения метода, а лишь запрашивает его результат, обработку которого производит другая функция. Применение асинхронных вызовов позволяет рационально распределить использование ресурсов системы в тех случаях, когда результат выполнения метода не требуется для дальнейшего функционирования программы, например, фиксация вызова метода в логах сервиса.

В зависимости от сервиса облака, к которому происходит обращение для вызова метода бизнес-логики, выделяют два: локальные и удалённые.

Вызов любых методов осуществляется посредством протоколов AMQP, RBC или HTTP/HTTPS. Какой протокол будет использован для вызова зависит от используемого.

## Список модулей для работы с асинхронными вызовами

Подключите сервисный модуль BL Async в состав тех сервисов (на вызываемой и вызывающей сторонах), где планируется использование механизма асинхронных вызовов.

Конфигурация, необходимая для работы модуля, будет получена автоматически из облака вместе с остальной конфигурацией для работы сервиса.

- AMQP Client — реализация работы с "брокером" асинхронных вызовов RabbitMQ. Если используется протокол AMQP.
- RequestBrokerClient - реализация работы с платформенным "брокером" асинхронных вызовов [request-broker](https://online.sbis.ru/article/7b3bb4dd-d382-4f1c-afa0-bd9f9de1d80b). Если используется протокол RBC.
- BL Async — основной модуль по работе с асинхронными вызовами.
- Berkeley DB — библиотека для работы с Berkeley DB.
- Berkeley Storage — локальное временное хранилище асинхронных вызовов на основе Berkeley DB.
- MessageBroker — базовый модуль для работы с произвольным "брокером".

## Типы вызовов метода

## Локальный вызов

При локальном вызове выполнение метода БЛ производится в том же сервисе облака, из которого он был вызван. Объект класса EndPoint ([Python](https://wi.sbis.ru/docs/py/sbis/EndPoint/) / [C++](https://wi.sbis.ru/docs/cpp/sbis/blcore/EndPoint/)) не используется, либо передан как пустая строка.

Для работы с удалёнными вызовами установлены требования, описанные в статье [Необходимые условия для удалённого вызова метода бизнес-логики](https://link.sbis.ru/article/1e1d4daf-4349-493d-9eca-6d4e9c8646a8).

### Аутентификация при удалённом вызове

В объект можно явно передать сессию последним параметром sid. Если сессия не установлена, то при внешнем вызове будет использоваться текущая сессия (если таковая имеется).

### Вызов метода по виртуальному каталогу сервиса с предварительной аутентификацией

- Она будет произведена под служебным пользователем при условии, что сервис не расположен в корне приложения и не является сервисом персональных данных клиентов.
- Если сервис "персональный", то при каждом вызове требуется дополнительно устанавливать пользователя или клиента, под которым должен выполняться вызов. Это необходимо, чтобы получить доступ к данным клиента. Если сервис входит одновременно в сайты online/inside/private, то вызов будет производиться через [online-private.sbis.ru](http://online-private.sbis.ru/) (сайт для межсервисных обращений на сервисы с системным именем "online").

Конфигурацию пользователя/клиента производят в объекте Endpoint, примеры использования которого вы можете прочитать в. В объекте передают второй параметр auth\_data:

- AuthByClientID — по прямому идентификатору клиента в облаке;
- AuthByExtID — по внешнему идентификатору клиента из биллинга;
- AuthByUserID — по прямому идентификатору пользователя клиента в облаке;
- AuthByLogin — по логину пользователя клиента в облаке.

## Приоритет обработки вызова метода БЛ

Вызовы методов БЛ обрабатываются Брокером, который осуществляет обмен сообщениями между компонентами программной системы. Для веб-приложений СБИС в качестве брокера применяется [RabbitMQ](https://ru.wikipedia.org/wiki/RabbitMQ), либо платформенный [request-broker](https://online.sbis.ru/article/7b3bb4dd-d382-4f1c-afa0-bd9f9de1d80b), который обрабатывает вызовы в соответствии с их приоритетами.

- rpNORMAL (по умолчанию) — нормальный приоритет.
	Используйте такой приоритет для "быстрых" методов БЛ, результат выполнения которых требуется моментально возвращать пользователю.
- rpLOW — пониженный приоритет. Вызов отправляется на служебный пул.
	Используйте такой приоритет для фоновых задач или "долгих" методов БЛ (например, при работе с файлами).

У всех асинхронных вызовов приоритет rpLOW и изменить его нельзя, т.е. все вызовы падают только на служебный пул. Также для асинхронных вызовов можно задать.

## Типы доставки асинхронного вызова методов

## Вызов с негарантированной доставкой

Вызов с негарантированной доставкой будет обработан не более одного раза. Это означает, что он может быть утерян, например, если сервис-исполнитель будет остановлен до исполнения запроса.

Запрос отправляется по протоколу HTTP/HTTPs. Адрес и другие реквизиты удаленного сервиса диспетчер берет из конфигурации. Соответственно, если мы отправляем запрос на сервис "Администрирование", то в конфигурации должна быть секция "Администрирование", в которой должен быть указан URL данного сервиса, и, опционально, реквизиты для аутентификации на нем. При отсутствии этой секции в настройках будет генерироваться исключение.

Вызов осуществляется посредством существующих возможностей библиотеки sbis-сommunication прямо из рабочего процесса БЛ (sbis-rpc-worker) с минимальным таймаутом. Управление клиентскому коду возвращается сразу после получения запроса исполнителем, ещё до передачи его в обработку.

## Вызов с гарантированной доставкой

Запрос отправляется по протоколу AMQP или RBC. Вызов с гарантированной доставкой будет обработан не менее одного раза. Это означает, что он может быть повторен, например, если сервис-исполнитель во время исполнения будет остановлен. Число повторных вызовов задаётся через [SetRepeatCnt](https://wi.sbis.ru/docs/py/sbis/EndPoint/methods/SetRepeatCnt/).

Обработка поступивших запросов ведется в общем порядке приоритетов, которые задаются при помощи метода [SetPriority](https://wi.sbis.ru/docs/py/sbis/EndPoint/methods/SetPriority/).

Вызов производится непосредственно из рабочего процесса БЛ (sbis-rpc-worker). Управление клиентскому коду возвращается в тот момент, когда от Брокера будет получено подтверждение приема сообщения.

### Выбор протокола

- AMQP. Запрос будет доставлен посредством RabbitMQ. Данный протокол используется по умолчанию.
- RBC. Запрос доставляется через платформенный [брокер асинхронного обмена](https://online.sbis.ru/article/7b3bb4dd-d382-4f1c-afa0-bd9f9de1d80b).

### Специализированный брокер

Для определенного класса запросов можно выделить отдельный изолированный контур доставки. Специализированный выделенный брокер будет обслуживать только такие запросы, что повысит отказоусточивость. В случае аварий на основном брокере, данный контур продолжит работать, и наоборот. Это необходимо прикладным механизмам, для которых крайне критична потеря запросов, либо продуцирующим большой поток запросов, чтобы уменьшить влияние на остальных.

💡

#### Повышенные гарантии доставки

Запросу, проходящему через специализированный брокер также можно повысить гарантию доставки. Все запросы будут храниться в резервном хранилище и будут повторяться, если по какой то причине были потеряны самим брокером (например, во время аварий). Опция необходима для особо критичных запросов, потеря которых крайне нежелательна.

Асинхронные вызовы доставляются до сервисов бизнес-логики через брокер, в котором образуется очередь. По умолчанию, все новые вызовы встают в конце очереди. Для каждого асинхронного вызова можно задать расширенные настройки приоритета в общей очереди с помощью метода [SetAsyncPriority](https://wi.sbis.ru/docs/py/sbis/EndPoint/methods/SetAsyncPriority/). ### Ограничения на размер тела запроса

💡

Размер тела запроса не должен превышать 100 Кбайт! Когда размер тела запроса находится в диапазоне от 100 Кбайт и до 1 Мбайт, в логи записывается сообщение с типом Предупреждение.

"Асинхронный вызов метода "ОбработкаPDF.ОбработкаФайлов" с сервиса " [private.sbis.ru/pdfservice/service/](http://private.sbis.ru/pdfservice/service/) " имеет слишком большой размер тела запроса (4677805 байт). Идентификатор запроса: 8c78ef0c-9a91-4935-b3d0-69fa5f42d961".

По этой причине запрещена передача файлов асинхронным вызовом с гарантией доставки.

![](https://link.sbis.ru/previewer/r/1600/782/site-builder/service/?id=0&method=SDocument.StreamResource&protocol=6&params=eyJmaWxlX2lkIjoiNWZlZDczMjYtOTBmMC00YmRmLThlMzQtYWY4YjA0ZGZkOTNlX2I1MTJmMmQxLTkxZGUtNDBkOS1iYzgyLTI3N2Q2NjMwNjM0YiIsInJlc291cmNlX2lkIjoiM2FiYjBhNGUtMzNlMi00YmFiLWEwN2EtMTk1ZWZjM2ZjYzVhIiwicmVzb3VyY2VfbmFtZSI6ImltYWdlLnBuZyIsInJlc291cmNlX29mZnNldCI6MTgzMDI0LCJyZXNvdXJjZV9zaXplIjoxOTczNzZ9)

Причины ограничения:

1\. Брокер реализован как мульти-мастер: каждый запрос сохраняется на диск на каждой из нод, что для файлов "дорого".

2\. Большие запросы увеличивают время синхронизации между нодами, что увеличивает время ответа Брокера.

3\. Брокер не рассчитан на хранение данных и у него выделен небольшой диск под хранилище. Если в асинхронных запросах передавать много данных, а целевой сервис не будет успевать обрабатывать запросы, то хранилище переполнится, что приведет к нестабильности всего облака.

### Приоритет обработчиков результата асинхронного вызова

В большинстве случаев, если результат выполнения метода ожидает пользователь (например, когда он загружает файлы), приоритет должен быть нормальным (значение 5), а если никто не ожидает, то — пониженный (значение 2).

"Расширенные" значения приоритетов асинхронных вызовов требуются для отдельных специализированных сервисов, где их определяет ответственный за сервис.

При асинхронном вызове внутри вызванного метода иногда требуется узнать приоритет, с которым он был вызван. Это нужно например для того, чтобы последующие вызовы из этого метода сделать с таким же приоритетом. Приоритет можно получить через вызов:

```text/x-sh
xxxxxxxxxx
```

```text/x-sh
Session.GetHeader( "X-AsyncPriority" )
```

Для синхронного запроса вернется None, для асинхронного — строка с приоритетом.

Если при вызове были указаны обработчики асинхронного вызова, то они будут выполнены с тем приоритетом, что и сам вызов.

## Передача больших данных в асинхронных вызовах. Huge Payload Protocol

- Если размер данных запроса не превышает пороговое значение (по умолчанию 100 Кбайт), то вызов происходит по обычной схеме (см. и доставка).
- Если превышено пороговое значение, то:
	- на вызывающей стороне: данные выгружаются на [Сервис временных файлов](https://link.sbis.ru/article/582e260b-6576-40ec-9d59-3ce80c17c583) (file-transfer), а в запросе передаётся только ссылка на данные.
		- на вызываемой стороне: перед вызовом исходного кода метода БЛ данные загружаются из сервиса file-transfer и передаются в метод так, как будто они были получены напрямую.

Протокол оформлен в виде сервисных модулей HugePayload и HugePayload-Py, которые поставляются в составе [SBIS SDK](https://link.sbis.ru/article/ab3eaab0-00b0-4e92-94fa-46c7c07241cb).

## Обработка результатов асинхронного вызова

Во многих сценариях требуется обработать результат асинхронного вызова метода БЛ. Всего существует четыре сценария:

| Сценарий | Поведение |
| --- | --- |
|  | Результат никак не обрабатываться. Является штатным сценарием при асинхронном вызове с негарантированной доставкой. |
|  | Является аварийным сценарием и представляет опасность для сервера приложений. В случае критической ошибки при выполнении задачи (падение рабочего процесса с дампом) делается соответствующая запись в лог, а задача помещается в специальную очередь ошибок Брокера для группы серверов (называется она как основная очередь группы с добавлением постфикса #error). Повторная обработка сбойной задачи не предусмотрена (после исправления ошибки разработчик, ответственный за данную предметную область, должен будет переместить необработанные задачи обратно в основную очередь). |
|  | Для обработки результата используется специальная функция — обработчик асинхронного вызова, который создают на обработку ошибки. |
|  | Для обработки результата используется специальная функция — обработчик асинхронного вызова, который создают на обработку результат нормального выполнения метода. |

Для C++ обработчики могут быть заданы любым параметром при помощи [bl::Callbacks](https://wi.sbis.ru/docs/cpp/sbis/blcore/Callbacks/). ## Обработчик асинхронного вызова

💡

Платформой СБИС гарантируется только то, что указанные функции-обработчики будут вызваны ровно один раз. Никакой обработки результата вызова данных функций не предусмотрено.

Обработчик асинхронного вызова — это специальная функция, предназначенная для обработки либо успешного выполнения асинхронного запроса, либо его ошибки. Использовать обработчики следует в том случае, если требуется выполнить некоторую дополнительную работу на сервисе-инициаторе запроса по завершении выполнения данного запроса исполнителем.

1. Обработки результата успешного асинхронного запроса (Callback). Функции-обработчики результата нормального выполнения метода принимают на вход три параметра: "Метод", "Параметры" и "Результат". Callback вызывается асинхронно.
2. Обработки ошибочного выполнения асинхронного запроса (Errback). Функции-обработчики ошибок принимают на вход три параметра: "Метод", "Параметры" и "Ошибка".

## Асинхронные очереди Брокера

Под каждый сервис бизнес-логики на Брокере создается несколько очередей. Их названия формируются сервисом при старте.

## Основная очередь

Название основной очереди имеет вид <UUID\_Облака>.<ИдГруппыСерверов>. Все символы в нижнем регистре.

- UUID\_Облака хранится в базе управления облаком в таблице ServiceGuid и присылается на сервис вместе с конфигурацией в параметре Ядро.Асинхронные сообщения.ИдентификаторОблака;
- ИдГруппыСерверов – идентификатор группы серверов из структуры облака в шестнадцатеричном виде. Присылается на сервис вместе с конфигурацией в параметре \_\_sbis\_\_group\_id.

Название основной очереди сохраняется в параметре Ядро.Асинхронные сообщения.НазваниеОчереди и имеет следующий вид:

```text/x-ini
xxxxxxxxxx
```

```text/x-ini
Ядро.Асинхронные сообщения.НазваниеОчереди=4704ea8e-22b9-40e0-96c9-c18a4ec6ceb2.00000008
```

```text/x-sh
xxxxxxxxxx
```

```text/x-sh
tsd-abramovvi:1002/admin/service/
```

## Прочие очереди

- очередь ошибок – <ИмяОсновнойОчереди>#error
- отложенные очереди – <ИмяОсновнойОчереди>#delay<ВремяПовтора>
