# ReferralProgram.GetLeadPeriodList — LeadCount: источник данных

Navigation: [[SabyBank-RKO-Referral]] | [[ReferralProgram-Data-Model]]

Метод возвращает список периодов (месяц/год) с агрегированной статистикой по рефпрограмме партнёра. Колонка `LeadCount` — количество лидов за период.

---

## Проблема: цикл по источникам через marketing-API

Исходная реализация получала идентификаторы всех источников (партнёров) программы, затем для каждого источника обращалась в marketing-API (`get_sales_sources_stats`) — по одному вызову на источник в цикле.

При 50 партнёрах в программе = 50 межсервисных вызовов. При 200 мс каждый ≈ 10 с. **Не масштабируется.**

> [!key-insight] Архитектурная догма
> В цикле по строкам нельзя делать межсервисные вызовы. Это не рекомендация — это запрет (Мусохранов А., 2026-06-24).

---

## Решение: единственный источник — ВидЦеныДокумент

**Принятое решение (2026-06-24, Мусохранов + Тимошенко):**

Считать `LeadCount` и `RewardSum` из `ВидЦеныДокумент` в **одном** запросе — без цикла по источникам, без обращения к marketing-API в рамках этого метода.

### Источник по типу программы

| Тип программы | Источник LeadCount |
|---|---|
| SabyBank / табмод | `ВидЦеныДокумент` по `ВидЦены` + период |
| Стандартная | `ВидЦеныДокумент` (лиды = корешки) |

Ранее для стандартных программ LeadCount брался из CRM/маркетинга (`get_sales_sources_stats`). Это неверно: `ВидЦеныДокумент` — таблица вознаграждений, не лидов; стандартные программы создают записи только через `SetLeadPrice`. Подробнее — в истории ошибки ниже.

### Оптимизация: один запрос вместо двух

Было: два подзапроса к ВЦД — один для `COUNT(корешков)`, другой для `SUM(бонус)`.

Стало: **один** запрос, считающий одновременно:
```sql
SELECT
    date_trunc('month', "ДатаВремя") AS period,
    COUNT(*) AS lead_count,
    SUM(COALESCE("Бонусы", 0)) AS reward_sum
FROM "ВидЦеныДокумент"
WHERE "ВидЦены" = ANY($1)   -- все ВидЦены программы/партнёра
  AND "ДатаВремя" BETWEEN $2 AND $3
GROUP BY 1
ORDER BY 1
```

Фильтрация по партнёру: если запрос «по одному партнёру» — фильтруем по `"Карта"` (идентификатор карты партнёра в ВЦД). Если «по всей программе» — по списку `ВидЦены`.

## История: ошибка LeadCount (задача 04307081, 2026-06-16)

`LeadCount` изначально считал только лиды с вознаграждением (`Бонусы > 0` в ВЦД). Корень: `ВидЦеныДокумент` — таблица вознаграждений; для стандартных программ записи создаются только через `SetLeadPrice` (UPDATE). Лиды без вознаграждения не имеют записи в ВЦД.

Исправление: стандартные программы → `get_sales_sources_stats('Лид', source_ids, ...)` из маркетинга per-period в Python. SabyBank/стабы → ВЦД по `ДатаВремя` (дата создания корешка), без фильтра `Бонусы > 0`.

Решение 2026-06-24 **отменяет** маркетинговый запрос для стандартных программ в пользу ВЦД (корешки). Это корректно только после того, как стандартные программы начнут создавать корешки через `CreateLead → CreateStub`.

---

## Что вырезается

- Получение идентификаторов источников всех партнёров программы (`get_all_referral_codes`)
- Цикл `for source in sources: get_sales_sources_stats(source, ...)`
- Отдельный подзапрос к ВЦД только для `LeadCount`

Остаётся один SQL-запрос к ВЦД, покрывающий и `LeadCount`, и `RewardSum`.

## Будущее

При переходе сделок (лидов) на корешки интеграция с marketing-API вернётся для стандартных программ — но уже одним batch-запросом, не в цикле.
