# Bonus Programs Architecture

Бонусная программа (БП) — правила начисления бонусных баллов. Списание бонусов осуществляется через **правила списания бонусов** (отдельная сущность).

---

## Концепт

- **Начисление** — настраивается в бонусных программах (ВидЦены, Тип=5).
- **Списание** — настраивается в правилах списания бонусов (ВидЦены, Тип=40).
- В рамках одной продажи возможно накопление и/или списание по **множеству** БП и правил списания.
- Offline: списание бонусов требует интернет-соединения (для получения баланса). Накопление — без связи.

Типы БП:
- Бонусы за покупки (% от суммы или за конкретные товары)
- Приветственные бонусы (за регистрацию)
- Праздничные бонусы (день рождения, праздники)
- [[Referral-Bonus-Program|Реферальная программа]] (ВидЦены.Тип=9; бонусы за покупки приглашённых, до 3 уровней иерархии)
- Внешние программы (интеграция с внешними БП)

---

## Начисление бонусов

### Праздничные начисления (scheduler)

1. Планировщик СДК ежедневно выбирает аккаунты с настроенными праздничными начислениями.
2. Асинхронно вызывает метод онлайна для инициации начисления.
3. Онлайн выбирает клиентов по критерию праздника (CRM), начисляет бонусы.
4. Онлайн публикует событие `online.loyalty-card.card-data.changed` → СДК обновляет данные.
5. Онлайн уведомляет сервис рассылок → рассылка уведомлений.

### Начисление по выборке клиентов

1. Пользователь делает выборку → заполняет диалог начисления.
2. Вызов метода начисления по условиям → онлайн → CRM → список @Лиц → начисление.
3. Событие `online.loyalty-card.card-data.changed` → СДК обновляет данные.
4. Одноразовая рассылка с ид документа начисления.

### Накопительные бонусные программы (offline)

- Режим "в сети": расчёт идентичен онлайну.
- Режим "без сети": только начисление. Сумма накоплений = 0 (кеш не используется).
  - Если первый порог > 0 — порог не срабатывает без связи.
- При первом заказе — доначисление "за первый заказ" не работает.

---

## Расчёт бонусного баланса

- Рассчитывается **динамически** из таблицы `ВидЦеныДокумент`.
- Учитываются только завершённые (полностью оплаченные) продажи.
- С учётом отложенного действия и даты сгорания.
- По всем бонусным программам.

**Максимальная сумма списания** = min из:
1. Максимально возможная для скидок (стоимость позиций − скидки − МРЦ)
2. Баланс бонусов (сумма по всем БП)
3. Вручную введённое значение списания

Механика "размазывания": сумма списания распределяется по позициям с учётом приоритетов правил.

> [!key-insight] Превышение баланса
> В редких сценариях (долгая задержка offline-sync или одновременные чеки) возможно списание больше накопленного. Реализация не защищает от этого намеренно.

## Доначисление и перерасчёт (закрытые продажи)

Метод: `RetailSaleDoc.IncCustomerBonus`

| Режим | Поведение |
|-------|-----------|
| Обычный | Дополняет ранее рассчитанные бонусы (напр., условие по типу оплаты) |
| Force (`force_recalc=True`) | Полный перерасчёт, перезаписывает все старые результаты |

Сценарии:
- **Продажа в кредит**: начисление сразу + доначисление после погашения (force).
- **QR-код в чеке**: доначисление при привязке покупателя (обычный режим).
- **Ручное доначисление**: `Bonus.AccrueForClosedSales` за период (обычный или force).

> [!note] Оффлайн
> Доначисление вызывается только в онлайне. Записи не попадают в оффлайн даже после закрытия кредита.
