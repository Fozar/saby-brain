# Анализ выполнения работ — ежедневный алгоритм

Задача: распределить все невыполненные работы за предыдущий день.

Перенос работ осуществляется на текущий день или на дальние даты с учётом цветных дней календаря.

## Правила переноса работы на «красный» день

При переносе на «красный» день, если сервис входит в Соглашение — необходимо проконтролировать наличие согласований:

| Ответственный | Сервисы |
|--------------|---------|
| Пустовалов И. | Задачи по сервисам семейства Оператор связи и приложению СбисДиск |
| Семилетов Д. | Задачи по сервисам семейств Отчётность, ФЭД |
| Субботин А.В. | Инфраструктурные задачи |
| Уваров С.В. и Новиков Д.В. | Все остальные задачи разработки (достаточно одного из двух) |

**Если согласование получено** — перенести работы в текущий план.
**Если согласований нет** — попросить дежурных ЦОД остановить выполнение, предупредить автора, перенести в ближайший «белый день».

## Алгоритм ежедневного переноса работ

### 1. Отбор работ (фильтры)

- Период: **«Вчера»**
- Состояние работ по стенду: **«prod»**
- **«Скрывать выполненные работы»** и **«Все кроме завершённых»**

Результат: список работ по всем prod-облакам, которые НЕ находятся в состоянии «Завершена» и не выполнены в РУ-облаке.

### 2. Правила при отборе

- Если текущая дата — «красный» день → следовать правилам переноса на красный день
- Если у работы статус «Выполняется» и текущая дата «белый» день → проверить согласование Новикова Д. на безостановочное выполнение

### 3. Порядок действий

1. Пометить работы через ПМО (кроме работ со статусом «Запускается») → нажать «Изменить дату»
2. Сменить вехи у переносимых работ (Excel-выгрузка или новые вкладки)
3. Указать даты планов согласно регламенту и соглашению → Подтвердить
4. До выпуска — сбросить статус у работ «Запускается» **через план выпуска** → поставить нужный ручной статус → вернуться в реестр и изменить дату
5. Региональные работы «Только КЗ/УЗ» — также отображаются; для просмотра статуса входить в карточку или переключать стенд
6. В новом плане снять «кирпичи» по prod-стендам у работ, не требующих дополнительных действий

> [!warning] Важно
> Работы, которые ещё НЕ сделаны в других облаках, закрывать на данном этапе НЕ нужно (кроме ТМ).

## Алгоритм контроля невыполненных работ

После переноса работ — контроль на наличие пропущенных или незакрытых работ за прошлые дни.

### Фильтры

- Период: **с понедельника предыдущей недели до даты последнего закрытого плана**
- Состояние работ по стенду: **«prod»**
- **«Показывать выполненные работы»** и **«Все кроме завершённых»**

Результат: список НЕЗАВЕРШЁННЫХ работ, требующих ручной обработки.

### Принципы обработки работ

| Ситуация | Действие |
|----------|----------|
| Внешние приложения / мобилки, ошибочно отображающиеся для регионов | Проставить «Завершена» |
| Работа на проверке у автора, и фаза уточнения положительно закрыта | Руками «Завершена» |
| Работа на уточнении, автор просит «отправить в свободное плавание» | Проставить «Планируется» |
| Работа по закрытому облаку (или осталось только в закрытом, например «ТМ») | Сразу «Завершена» |
| Стандартные пользовательские работы «Согласование плана», «Журнал работ ЦОД» | «Завершена» или статус «Выполнено» |
| Работа потеряна при переносах | Добавить на первый разрешённый для выполнения день (по согласованию с автором) |

> [!note] Исключение
> НЕ обрабатывать работы в папках «Планируется» — они в свободном плавании.

## Еженедельный отчёт (пятница)

До совещания с Новиковым Д.:
1. Пробежаться по работам в состоянии «Планируется»
2. Проверить что нет удалённых документов или документов, закрытых положительно как выполненных
3. Такие работы пометить как «Завершена»

**Фильтры при демонстрации экрана:**
- Период: с понедельника предыдущей недели до среды текущей недели
- «Показывать выполненные работы» и «Все кроме завершённых»

**Формат устного отчёта:**
> «На текущий момент незакрытых работ по реестру работ НЕТ»
> 
> — или —
> 
> «На текущий момент незакрытые работы по реестру работ есть — автор пока не уточнил по срокам выполнения работ на бою»
