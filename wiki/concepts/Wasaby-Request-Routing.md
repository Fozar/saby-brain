# Wasaby — Маршрутизация запросов к web-сервисам

## Классы сервисов

| Класс | Примеры |
|-------|---------|
| Простые | dwc и т.п. |
| Multiversion | specifications-ps, messageservice |
| Multitenancy | online, online-ps |

## Компоненты маршрутизации

- **http-балансировщик** — nginx-кластер для синхронных запросов; определяет upstream → выбирает узел → добавляет заголовки → проксирует
- **web-service** — кластер обработки http и amqp запросов, определяет область данных
- **Хранилище маршрутов** — тип сервиса + настройки кэширования

## URL синхронного запроса

```
<protocol>://<domain-name><url-путь><params>
```

Важные параметры:
- `x_version` — версия вызывающего сервиса
- `x_module` — дата/время сборки модуля (только для сервисов представления, при обновлениях)
- `srv=1` — направить в **служебный пул** (фоновые задачи)

### Алгоритм маршрутизации синхронных запросов

1. Из URL исключаются protocol и domain_name
2. По URL-пути подбирается наиболее подходящий upstream
3. Проверяется тип upstream (простой/multitenancy/multiversion)
4. Проверяется блокировка сервиса (→ 502 если заблокирован)
5. Запрос → координатор узла web-сервиса → основной или служебный пул (по `?srv=1`)

Если узел недоступен: временно исключается, запрос пробрасывается на следующий, до успеха или 502.

## Кэширование GET-запросов

Varnish-кластер кэширования:
- Upstream: узлы кэша (обычные) + узлы web-сервиса (резервные)
- Алгоритм: балансировщик → Varnish → проверка области data → (кэш или web-сервис)

## Резервные сервисы

Если все узлы web-сервиса недоступны → запросы перенаправляются на резервный сервис. Реализовано на уровне nginx upstream (резервные узлы помечены как `backup`).

## Маршрутизация асинхронных запросов (AMQP)

```
Клиент → обменник (GUID облака $ системное имя сервиса) → Основная очередь → узел web-сервиса
```

Обменник именуется: `<GUID облака>$<системное имя web-сервиса>`

При отсутствии очереди — сообщение пропадает. Брокер (RabbitMQ) — посредник между клиентом и web-сервисом.

## Multitenancy-маршрутизация

За одним URL стоят сервисы разных клиентов. Задача — определить узел сервиса конкретного клиента. Усложняется при обновлениях (для минимального простоя).

Параметры `x_version` и `x_module` актуальны **только при обновлении** (для multiversion/multitenancy).

## Конфигурация диспетчера

Если конфигурация устарела (мигающий флажок в cloud-ctrl → Приложения), необходимо обновить через кнопку "Обновить конфигурацию" для нужных сайтов. Иначе маршрутизация может быть некорректной.
