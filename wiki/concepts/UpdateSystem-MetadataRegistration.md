# Регистрация метаданных

Процесс получения данных из дистрибутива и регистрации их в целевом middleware-сервисе. Выполняется в процессе обновления приложения или сразу после него.

## Сервис metadata-registrator

Приложение-сервис, содержащее всю логику регистрации. Обновляется независимо от релизного цикла Хоттабыча — это позволяет проводить долгие регистрации без блокировки Хоттабыча.

**Git:** `https://git.sbis.ru/middleware/metadata-registration`

Хоттабыч при запуске обновления запрашивает у сервиса все объекты с префиксом `DistrReg_` и создаёт логики для каждой регистрации. Зависимости между модулями переносятся на порядок исполнения задач.

**Кэширование:** повторное обновление на ту же версию возвращает сохранённые данные без повторного скачивания дистрибутива — значительно ускоряет обновления других приложений на том же дистрибутиве.

## API Хоттабыча к сервису регистрации

| Метод | Когда вызывается |
|-------|-----------------|
| `MetadataRegistratorParseDistribution(DistrName, Version)` | Фаза доставки — разбор дистрибутива, заполнение `DistrRegData` |
| `MetadataRegistratorPrepare(...)` | Подготовительная фаза |
| `MetadataRegistratorUpdate(...)` | Основная фаза |
| `MetadataRegistratorRollback(...)` | Фаза отката |
| `MetadataRegistratorClean(...)` | Фаза очистки |
| `MetadataRegistratorCommit(...)` | Фаза подтверждения |

На фазе подготовки: проверяет наличие данных в `DistrRegData` → если есть, возвращает без скачивания; если нет — скачивает дистрибутив, распаковывает, вызывает `CollectData` для каждого регистратора, сохраняет в `DistrRegData`.

## Способы регистрации

### Гибкий (полный) способ

Разработчик реализует BL-модуль `DistrReg_[НазваниеФункционала]` (в раздел «РегистрацияМетаданных» в Genie). Внутри — объект с тем же именем.

**Метод `Info()` — без аргументов, возвращает Record:**

| Поле | Описание |
|------|----------|
| `Name` | Человекочитаемое имя на русском (отображается в Хоттабыче) |
| `SystemName` | Системное имя сервиса для регистрации |
| `DependsOn` | Массив системных имён зависимых регистраторов |
| `IsCritical` | `true` = критичная, `false` = отложенная |
| `Type` | Для каких типов приложения нужно выполнить регистрацию |
| `AfterUpdate` | До или после обновления сервиса (только для критичных) |

**Метод `CollectData`** — принимает путь до распакованного дистрибутива, возвращает данные для регистрации.

**Для критичной регистрации дополнительно:**

| Метод | Фаза | Описание |
|-------|------|----------|
| `Prepare` | Подготовка | Вызывает `FeatureRegistratorPrepare` на сервисе (данные подготовлены, но не используются) |
| `Update` | Обновление | Активирует новые данные |
| `Rollback` | Откат с основной фазы | Вызывает `FeatureRegistratorRollback` |
| `Clean` | Очистка (при откате с Prepare) | Отменяет подготовленные данные |
| `Commit` | Подтверждение | Подтверждает использование новых данных (`FeatureRegistratorCommit`) |
| `Discard` | После отката | |

Методы принимают: `AppName`, `DistrName`, `Version`, `Services` (массив системных имён сервисов).  
Возвращают (опционально) RecordSet: поля `type: Int` (0=info, 1=error, 2=warning), `msg: String`.

**Для отложенной регистрации дополнительно:**
- `RegisterAtOnce(AppName, DistrName, Version, Services)` — всё в одном методе, вызывается **только после фазы Commit** (приложение уже полностью обновлено). На Prepare и Update ничего не происходит.

**Хелперы для сохранения состояния между фазами:**
```python
RegistrationHelpersSaveState(...)     # сохранить промежуточные данные
RegistrationHelpersGetSavedState(...) # получить сохранённые данные
```

### Упрощённый способ (ExtCollector)

Не требует реализации BL-модуля. Достаточно:
1. Создать файл `[расширение_файла].extregistration` в репозитории Core / модуль ExtCollector / папка `registration`
2. Реализовать методы приёма данных на целевом сервисе со строго заданной сигнатурой

**Поля `.extregistration` файла:**

| Поле | Описание |
|------|----------|
| `extension` | Расширение регистрируемых файлов |
| `name` | Название регистрации (для карточки обновления) |
| `description` | Описание |
| `responsible` | Ответственный |
| `is_critical` | Признак критичности |
| `service` | Системное имя сервиса-получателя |
| `object` | Название объекта для вызова методов регистрации |
| `format` | Формат данных |

В упрощённом режиме `MetadataRegistratorPrepare/Update/Rollback/Clean` вызывают не методы модуля, а собственную реализацию с сохранением состояния на стороне сервиса регистрации. Сценарий:
- `Prepare` → `GetUnregister()`, сохранить в `SavedState`
- `Update` → `Register()` (новый список)
- `Rollback` → `Register(SavedState)` + `PrepareUnregister(новый)`
- `Commit` → `Unregister(старый)` при успехе или `Unregister(новый)` при откате

Перед регистрацией проверяется наличие данных через `ExtCollector`; если данных нет — вызывается `DataExists()` на целевом сервисе. Если возвращает false — дальнейшие вызовы не выполняются.

## Типы регистрации по критичности

### Критичная (`IsCritical = true`)
Без выполнения сервис не запустится или работает некорректно. При недоступности целевого сервиса — повторные попытки до успеха. При таймауте ожидания — обновление падает с ошибкой, блокируя обновление других узлов.

### Отложенная (`IsCritical = false`)
Для некритичных сервисов. Хоттабыч принимает статус «отложена» как выполненную — обновление может завершиться без ожидания. В таблице `ИсторияОперацийПродукта` ставится флаг незавершённых отложенных регистраций. Обновление размещается в разделе «Отложенные регистрации» в реестре.

Очередь в `RegistrationQueue`, задача планировщика обрабатывает её. При ошибках — повторные попытки до определённого времени с момента постановки.

## Схема базы данных

**Таблицы:**

| Таблица | Поля | Назначение |
|---------|------|------------|
| `DistrVersion` | `Name`, `Version` | Версии дистрибутивов |
| `DistrRegData` | `VersionId Int`, `RegName String`, `RegData String` | Данные регистраций по версиям |
| `SavedState` | по `DistrVersionId, RegName, AppName` | Промежуточное состояние между фазами |
| `RegistrationQueue` | `State`, `StartTime`, `RegistratorName`, `AppName`, `DistrName`, `Version`, `Services`, `Callback`, `Method` | Очередь отложенных регистраций |

**Индексы:**
```
DistrVersion(Name, Version)
DistrRegData(DistrVersionId)
DistrRegData(DistrVersionId, RegName)
SavedState(DistrVersionId)
```

**Состояния RegistrationQueue:** в очереди / отправлено / ошибка / выполнено.

## Сценарий создания нового модуля

1. Разработчик получает push-доступ к репозиторию регистраторов
2. В Genie в разделе «РегистрацияМетаданных» создаёт модуль `DistrReg_[ОсмысленноеНазваниеCamelCase]` + одноимённый объект
3. Реализует `Info()` и `CollectData()`
4. Для критичной → реализует `Prepare`, `Update`, `Commit`, `Rollback`, `Discard`, `Clean` (все делают вызов на middleware-сервис внутри облака по системному имени)
5. Для отложенной → реализует `RegisterAtOnce`
6. В middleware-сервисе реализуется ответная часть методов

## Особые режимы

### Обновление в других облаках
Если приложение обновляется в другом облаке — все вызовы идут на сервис регистраторов **того** облака. Сам сервис вызывает middleware-сервисы внутри того же облака по системному имени.

### Режим при ошибке чтения дерева проекта
Если metadata-registrator не может прочитать дерево проекта:
1. Отвечает Хоттабычу с минимальными данными для проведения обновления
2. Хоттабыч пропускает все логики регистрации, проводит обновление БЛ и БД
3. В конце фазы обновления — вызывается батчевый метод: разбор дистрибутива + все регистрации в одном вызове
4. При неудаче — исключение с предложением повторить лёгкое обновление вручную

### Требования к middleware-сервисам
Нежелательны обновления в блокирующем режиме — подходит лёгкая конвертация. Если без полного обновления не обойтись — обновлять отдельно, тогда регистрация произойдёт до блокировки сервиса.

## Подсистема прав

Методы сервиса доступны только **администратору Хоттабыча** (для регистраций в других облаках).

## Облачные параметры

| Параметр | Описание |
|----------|----------|
| `Управление версиямиОбновленияЧерныйСписокРегистраторов` | Системные имена middleware-сервисов, регистрации которых исключаются из всех обновлений (если сервис нестабилен) |
| `Управление версиямиОбновленияОтключитьРегистрациюДляПриложений` | Список приложений без регистрации метаданных |
| `Управление версиямиОбновленияРегистрацияВрежимеОтладки` | Список регистраций, для которых ошибки игнорируются (формат: `DistrReg_Name` или `SimplifiedRegistration<расширение>`) |
| `Управление версиямиОбновленияТаймаутВызоваКритичныхРегистраций` | Таймаут критичных регистраций |
| `Управление версиямиОбновленияТаймаутВызоваОтложенныхРегистраций` | Таймаут отложенных регистраций |
| `Управление версиямиОбновленияМинимальноНеобходимоеСвободноеМестоНаРегистраторе` | Мин. свободное место (MB) в `ApplicationDataPath` для скачивания дистрибутива |
