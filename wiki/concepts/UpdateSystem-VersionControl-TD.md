# ТД Управления версиями

Техническая документация сервиса «Управление версиями» (Хоттабыч). Git: `https://git.sbis.ru/orchestration/admin.git`. Модуль: `Version manager`.

## Архитектура управления обновлением

### Запуск обновления

RPC-вызовы:
- `ПродуктОбновить` — все типы обновлений
- `ПродуктInstall` — первичный разворот
- `DeployPackageStart` — разворот данных

Перед запуском проверяется:
- Доступность облака
- Права пользователя
- Зависимости (`dependencies.xml`) — запрашиваются версии зависимых сервисов у облака
- Наличие дублирующих серверов БЛ и статик
- Наличие реплик у всех БД (для типа мастер-реплика)
- Возможность запуска обновления или скрипта (политика параллельности)

### Менеджер асинхронного обновления (МАО)

Управление обновлением через ориентированный ациклический граф:
- Вершины = задачи (таблица `Task`)
- Дуги = зависимости (таблица `TaskRelationship`)

Граф хранится в БД — позволяет продолжить управление на любом равноправном узле БЛ при отказе одного.

**Статусы задач:**
- `rsPENDING = 0` — ожидает
- `rsQUEUE = 1` — взята на исполнение
- `rsRUNNING = 2` — исполняется
- `rsFINISHED = 4` — завершена → разрешает зависимости, запускает следующие

### Типы задач

| Тип задачи | Класс (C++) | Описание |
|------------|-------------|----------|
| Формирование графа фазы | `PhaseFormationTask` (PFT) | Строит граф задач для фазы |
| Подфаза | `SubPhaseTask` (SPT) | Информирует о ходе, выставляет режимы |
| Загрузка утилиты | `DownloadUtilityTask` | Загружает sbis-update-exec на сервер |
| Запуск утилиты | `RunUtilityTask` (RUT) | Запускает утилиту с параметрами |
| Задание контейнеру | `ContainerTask` | Асинхронный вызов `ContainerAsyncAction` (Datacenter API) |
| Обновление агента | `UpgradeAgentTask` | `UpdateAgentUpgrade` + проверка версии |
| Запуск скрипта | `ScriptTask` | RPC вызов `UpdateScriptExecute` |
| Сброс кэша | `BlCacheTask` | RPC вызов `NginxResetCache` |
| Регистрация планировщика | `SchedulerTask` (ST) | Регистрирует задачи планировщика |
| Регистрация кластера | `ClusterTask` | Регистрирует кластеры в облаке |
| Регистрация параметров | `RegParamValuesTask` (RPVT) | `SettingsRegisterParameter` для strict и flexible |
| Завершение фазы | `FinalizerTask` (FT) | |

Агент отправляет отчёты раз в 5 секунд через `ПродуктОтчетыАгента`.

### Фазы обновления

| Фаза | Что происходит |
|------|---------------|
| **Доставка** | Загрузка `sbis-update-exec` + `*_db.zip` из ХД на БЛ Хоттабыча; извлечение параметров приложения, данных планировщика; формирование задач загрузки утилиты на серверы |
| **Обновление агентов** | Проверка актуальности версии агентов, обновление при необходимости |
| **Подготовка** | Загрузка/распаковка файлов БЛ и jinnee_lite; подготовка БД; регистрация параметров; регистрация планировщиков |
| **Обновление** | Перемещение файлов сервиса, конвертация БД, перезапуск и прогрев сервиса, регистрация |
| **Откат** | Восстановление файлов из бэкапа (только БЛ и статика; конвертация БД откату не подлежит) |
| **Очистка** | Удаление временных файлов и бэкапов; после запуска нельзя перейти на другую фазу |
| **Очистка папок** | Удаление временных папок |

## Алгоритмы типов обновлений

### Лёгкое обновление (umLite = 2)

Zero-downtime. Пользователь работает во время обновления. Объекты: серверы статики и БЛ.

```
CloudGroupSetUpdateMode(mode=umLite, node_mode=update_first)  # перенаправить на 2-ю часть БЛ
→ обновить 1-ю часть БЛ
→ CloudGroupSetUpdateMode(node_mode=update_both)               # обе части
→ CloudGroupSetUpdateMode(node_mode=update_second)             # перенаправить на 1-ю часть
→ обновить 2-ю часть
→ снять режим (FT)
```

### Лёгкое последовательное (umSequentialLite = 4)

Для патчей в дневное время. Режим ставится **один раз** в начале фазы. Не влияет на диспетчер (для него приложение в необновляемом состоянии). Узлы БЛ обновляются по одному в цепочку. При прерывании и возобновлении — цепочка продолжается с прерванного места.

При наличии СП (сервисов представления): БЛ обновляется поочерёдно (sequential), СП — двумя частями с маршрутизацией. Параметр `Управление версиямиОбновленияРазбросУведомленийИнтерфейса` — диапазон случайного времени уведомления в moment `update_both`.

### Полное обновление

Блокировка + остановка всего продукта. Парковочная страница через `ДиспетчерУстановитьПарковочнуюСтраницу`. Обе половины БЛ обновляются одновременно. Конвертация БД.

### Полное последовательное

Для продуктов, которым нужна непрерывная запись в БД. Флаг `full_consistent_update` на сервисе в Genie (вкладка Extended: «Full consistent update allowed»). Алгоритм:
1. БЛ и БД СНЗ (сервиса непрерывной записи) распределяются по равноправным группам в облаке
2. СНЗ перезагружает БЛ — применяет настройки из облака (БЛ работает только со своей БД)
3. Все остальные сервисы блокируются и обновляются как при полном обновлении
4. Параллельно каждый СНЗ обновляется последовательно по группам:
   - БЛ группы выключаются → запросы уходят на другие сервера
   - Все БД группы конвертируются параллельно
   - БЛ группы обновляются параллельно
   - Группа включается
5. Снятие режима обновления → разблокировка → перезапуск БЛ СНЗ

### Посхемное обновление (без остановки)

Для продуктов онлайна. С конвертацией, но без полной остановки. На фазе обновления:
- Половина БЛ и статики обновляется, половина остаётся на старой версии
- Конвертация БД посхемно: клиент работает на старой БЛ → блокируется в момент своей конвертации → схема конвертируется → разблокируется → направляется на новые БЛ
- По завершении конвертации всех клиентов → вторая половина БЛ и статик обновляется

Особенности: задание списков конвертируемых/игнорируемых клиентов; возможность изменить список игнорируемых в процессе; возможность «дообновлять» только упавшие сервисы.

### Лёгкая конвертация

Конвертация без остановки. БЛ работает с конвертируемой базой. Jinnee не разрывает установленные соединения. Операции с ALTER TABLE могут временно блокировать таблицу. Отчёт «Анализ изменения БД» в Хоттабыче показывает критичность изменений для этого типа.

### Мастер-реплика

БД: у каждой БД должна быть настроена реплика. Алгоритм:
1. Все БЛ переключаются на реплику
2. Отключается репликация
3. Конвертируется мастер-база
4. БЛ и статика обновляются (БЛ настраиваются на работу с мастером; режим umLite для маршрутизации)
5. Возобновляется репликация, Хоттабыч ждёт нормализации лага
6. Все БЛ переключаются обратно (мастер + реплика), режим снимается на очистке

> В процессе обновления (шаги 1-5) все запросы на запись в базу выдают ошибку.

### Подготовка конвертации БД

Отдельный тип обновления для разделения обновления на два этапа. На фазе подготовки передаётся параметр `Управление версиямиКонвертерыВремяХраненияХешаСхем` в jinnee_lite (удлинённое время жизни хэша). Фазы обновления и подтверждения пропускаются.

При следующем обновлении с конвертацией: проверяется история → если хэши актуальны → jinnee переиспользует их, получая ID предыдущего обновления (как будто продолжает его). Если хэши устарели — конвертация в обычном режиме.

Флаг «Принудительно запустить подготовку конвертации» игнорирует историю (если была добросла критичная правка конвертации).

### Параллельное обновление приложений с общей БД

Параметр: `Управление версиямиОбновленияПриложения с общей БД`. Конвертацией занимается master-обновление, slave-обновления ждут в точке конвертации. Подробно: easy conv → master конвертирует и ставит флаг → slave ждёт флага; full → синхронизация режимов → master конвертирует → slave ждут.

## Выполнение скриптов (технические детали)

### Фаза подготовки скрипта

Не выполняется.

### Фаза обновления: внедрение скрипта

Утилита обновления:
1. Находит целевой сервис, создаёт копию в отдельной папке
2. Скачивает архив модуля выполнения скрипта (`DeveloperScript_sbis-script-module`) и архив прикладного скрипта (`DeveloperScript_%ИмяАрхива%`)
3. Если в архиве скрипта нет `.smod` — генерируется автоматически с зависимостью от Python Core
4. Запускает копию сервиса

Хоттабыч запрашивает клиентов из облака → сохраняет в `ClientDatabase` (связь с `ИсторияОперацийЛогики`). Для неперсональных: клиент = сам сервис.

### Генерация сценариев DWC

По каждой БД сервиса — 1 или несколько DWC-сценариев. Количество = настройка «Число потоков конвертера для фаз исполнения скриптов».

Каждый сценарий содержит 2 задачи на клиента:
1. `DeveloperScriptExecuteAll` — выполняет скрипт, возвращает результат
2. `ClientDatabaseScriptDone` — Хоттабыч сохраняет результат

### Параметры DeveloperScriptExecuteAll

| Параметр | Описание |
|----------|----------|
| `UpdateID` | Идентификатор обновления |
| `ClientDatabaseID` | ID записи в таблице `ClientDatabase` |
| `VersionManagerService` | Системное имя сервиса Управления версиями |
| `Guid` | Идентификатор запуска скрипта (проверяется `ClientDatabaseCheckScript`) |
| `Personal` | Признак персонального сервиса |
| `DoneMethods` | Уже выполненные методы скрипта |

Перед выполнением вызывается `ClientDatabaseCheckScript` (проверка Guid). Методы объекта `DeveloperScript` без параметров, не в `DoneMethods`, запускаются через `FutureInvoke` поочерёдно (одновременно выполняется только один). Основной поток периодически вызывает `ClientDatabaseReportAlive`.

### Состояния ClientDatabase

| Статус | Описание |
|--------|----------|
| 0 | Не выполнен (начальный) |
| 1 | Выполняется (при получении `ClientDatabaseReportAlive`) |
| 2 | Выполнен |
| 3 | Ошибка |

### Хранение результатов

- `ScriptMessage` — очередь ответов (WAL отключён для производительности); после обработки удаляются
- `ScriptMethod` — информация о выполненных методах
- Задача планировщика `ClientDatabaseProcessScriptResponses` (каждую минуту): берёт пачку из `ScriptMessage` → вычисляет статусы логик → публикует в журнал обновления → удаляет обработанные записи

Размер пачки: `Управление версиямиСкриптыРазмерПачкиСообщений`.

Задача также следит за зависанием: если клиент в статусе «Выполняется» дольше таймаута (`Управление версиямиСкриптыОжиданиеОтчета` минут) → публикует ошибку таймаута.

### Прерывание скрипта

1. Хоттабыч отменяет DWC-сценарии (отменяет не сделанные асинхронные вызовы)
2. Останавливает копию сервиса
3. Перед остановкой проверяет наличие асинхронной очереди по скриптовой БЛ в облаке → при наличии до таймаута прерывание не считается успешным

### Фаза подтверждения скрипта

1. Перезапустить узел скриптовой БЛ
2. Анализ очередей: через облако + через `?load`-запрос к узлу
3. Завершить процесс скриптовой БЛ

В `ScriptHistory` сохраняется результат выполнения. Для персональных сервисов при неполном выполнении — список обработанных клиентов (для продолжения после завершения обновления).

## Аварийный режим (RabbitMQ недоступен)

**Параметры:**
```
ЯдроАсинхронные сообщенияТранспортныйПротоколПоУмолчанию = HTTP
  (для сервисов: version-manager, datacenter, metadata-registrator, cloud-control-api)
Управление версиямиОбновленияАварийный режим работы = ДА
  (для сервиса version-manager)
```

Первый параметр переключает все async-вызовы в платформе на синхронные (без гарантии доставки, без RabbitMQ). Второй включает обходные пути:
- **Локальный планировщик**: активирует монитор фоновых операций для периодического вызова задачи обхода графа обновлений (вместо облачного планировщика)
- **Самодостаточность неперсональных скриптов**: исключает DWC из цепочки запуска → прямой вызов на скриптовой БЛ с прямой публикацией отчётов

## Агент обновления

Исполняет команды от БЛ Хоттабыча. Команда = путь до утилиты `sbis-update-exec` + аргументы.

**Основные методы агента:**

| Метод | Описание |
|-------|----------|
| `UpdateAgentExecuteAsync` | Асинхронный запуск утилиты. Порождает потоки: ExecuteCommandThread (запуск процесса), SendReportThread (чтение stdout), WaitInterruptionThread (проверка прерывания каждую секунду). Отчёты рассылаются каждые 3 секунды |
| `UpdateAgentExecuteSync` | Синхронный запуск, возвращает stdout |
| `UpdateAgentInterrupt` | Запрос на прерывание процесса (по host + `ИсторияОперацийЛогики` + фаза) |
| `UpdateAgentDownload` | Загрузка утилиты на сервер перед выполнением |
| `UpdateAgentUpgrade` | Обновление самого агента (процесс отделяется от агента, чтобы мог его перезапустить) |
| `UpdateAgentCheckServiceInstalled` | Проверка возможности установки сервиса |
| `UpdateAgentOperatingSystem` | ОС сервера |
| `UpdateAgentServerInfo` | Информация о сервере (диск, RAM, CPU) |
| `UpdateAgentVersion` | Версия агента |

Аргументы утилиты передаются через файл в `file-storage` или `file-transfer` (выбор через параметр конфигурации Хоттабыча).

## Утилита обновления (sbis-update-exec, C++)

### Алгоритм обновления БЛ (части сервиса)

Фаза подготовки:
- Получает список частей сервиса с ссылками из ХД
- Скачивает в `/opt/sbis` (не в корневой раздел → без расхода места там)
- Фиксирует успешное скачивание части → при возобновлении не перекачивает

Фаза обновления:
- Если список частей передан: только устаревшие части → в бэкап, новые части из подготовки → в сервис
- Если «обновить все»: все файлы → в бэкап, все из подготовки → в сервис
- После перемещения: проверка md5 через `ServiceHash`; при несовпадении — ошибка с анализом расхождения по частям

### Алгоритм конвертации БД (jinnee + утилита)

Jinnee в цикле спрашивает утилиту: «можно ли конвертировать схему?»

Возможные ответы (для посхемного):
- `sccPROCEED` — конвертировать
- `sccPERMANENTLY_SKIP` — пропустить (клиент в списке игнорируемых или вне конвертируемых); передаётся в Хоттабыч через `PerSchemaParamsAppendSkippedSchemas`
- `sccTEMPORARY_SKIP` — отложить (ожидание данных из облака)
- `sccMARK_AS_ERROR` — пропустить и занести в несконвертированные (не удалось заблокировать клиента)

Проблемные клиенты хранятся в контейнере (сбрасывается на диск после каждого цикла). Сообщение «FAILED_CLIENTS» с количеством → в отчёты агента.

Параметр «таймаут конвертации схемы» — если конвертация схемы превышает таймаут → предупреждение в журнал.

### Серверы конвертации

Список активных конвертеров — из кластера в Управлении облаком. Назначение конвертера: выбирается наименее загруженный (суммарная сложность задач = сумма числа потоков jinnee). Перед отправкой задачи: проверка LA, RAM хоста ЦОД, загрузки CPU → если превышен порог → задача ждёт (раз в минуту), при таймауте — ошибка.

Облачные параметры: `Управление версиямиКонвертерыЛимитЗадач`, `Управление версиямиКонвертерыЛимитПроцессора`, `Управление версиямиКонвертерыЛимитПроцессораХоста`, `Управление версиямиКонвертерыПиковоеПотреблениеПамятиДжинном`, `Управление версиямиКонвертерыВремяОжиданияРесурсов`.

## Разворот пакетов данных

Events-based подход:
1. Хоттабыч публикует событие `versionmanager:applypackage` (данные пакета + ссылка на дистрибутив)
2. Хоттабыч ждёт `versionmanager:packagestarted`
3. Хоттабыч ждёт `versionmanager:packagefinished`
4. Сервис управления облаком ловит `versionmanager:packagefinished` → обновляет версию пакета в облаке

## Необлачные приложения (update.saby.ru)

update.saby.ru: мастер-сервер + слейвы + балансировщик (Node.js). Хоттабыч:
1. Получает адрес балансировщика из облака → `GET /status/` → адрес мастер-сервера и путь загрузки
2. Агент скачивает дистрибутив во временную папку мастер-сервера
3. Основная фаза: проверяет lock-файл (папка первого уровня) → проверяет синхронизацию (`GET /api/sync/id`) → подменяет файлы → `POST /api/sync/id` (запуск синхронизации) → периодически `GET /api/sync/id` (% синка)
4. Подтверждение: удаление бэкапов, временных файлов, утилиты

**Сценарии выкладки:** набор правил (тип правила, значение, комментарий, время действия). Формируется json-конфиг с описанием поддерживаемых версий и правил → выкладывается на мастер-сервер. Фаза «Применение правил» (особая, только для необлачных со сценарием) — ждёт окончания времени жизни версии.

```typescript
// afterMount точки входа:

// Получить версию:
ServiceUpdateNotifier/notifier.getVersion() → { version: ... }

// Проверить актуальность:
ServiceUpdateNotifier/notifier.isVersionOutdated() → boolean
```

Уведомление: после получения события о мажорном обновлении → предложение обновить страницу. После 3 откладываний или N ошибок → окно без возможности отложить. «Через 10 минут» → таймер → автоперезагрузка.

> При SabyPage подключение не требуется.

## Обновление агентов из Управления облаком

### VersionManagerCommandToAgent

```
VersionManagerCommandToAgent(Agents: Json, Operation: Json) → Record
```

Метод Хоттабыча для вызова операции на агентах. Вызывается из Управления облаком.

**Agents Json** — список агентов:
```json
{
  "Host1": [
    { "Port": "String", "Directory": "String", "SystemName": "String", "ServerPool": "Int32", "AgentPort": "String" }
  ],
  "Host2": [...]
}
```

**Operation Json** — описание операции:
```json
{
  "<операция>": "String",   // bl_reboot | bl_stop | bl_start | bl_invoke | agent_update
  {
    "update_id": "Int32",   // опционально
    "id": "String",          // уникальный идентификатор запроса
    "modificator": "String", // synchronically | sequentially (при обновлении агентов игнорируется — всегда sequential)
    "Object": "String",      // для bl_invoke — название объекта БЛ
    "Method": "String",      // для bl_invoke — название метода
    "Arguments": "Record"    // для bl_invoke — аргументы метода
  }
}
```

**Result Record:**
```json
{
  "Agents": "Json",    // то же что на вход + @AgentOperationsServer (ID записей из таблицы)
  "Operation": "Json"  // то же что на вход + @AgentOperationsHistory (ID задачи)
}
```

**Алгоритм:** создаётся запись в `AgentOperationsHistory` и записи в `AgentOperationsServer` → параллельно загружается утилита на все серверы → последовательно выполняется операция обновления агентов.

### События обновления агентов

**`versionmanager:agent_operation_changed`**
- `Visibility = evSERVER_ONLY`, `PublicationPolicy = evIMMEDIATELY`, `Delivery = evGLOBAL`
- `message` (HashTable): `@AgentOperationsHistory`, `Identifier` (из Operation.id), `Status`, `Error`

**`versionmanager:agent_operation_server_changed`**
- `message`: `@AgentOperationsHistory`, `@AgentOperationsServer`, `Identifier`, `LogicIdentifier`, `Host`, `Port`, `Directory`, `DaemonName`, `Status`, `Error`, `Message`

**`versionmanager:agent_operation_journal_changed`** — формат аналогичен `agent_operation_server_changed`.

## Обновление кластеров

Хоттабыч управляет обновлением кластеров (Kubernetes) через API датацентра. Запуск возможен только на **папках** в справочнике кластеров, не на единичных экземплярах.

### Датацентр API

Датацентр принимает: `идентификаторы кластеров`, `версия`, `метаданные дистрибутива из ХД`. По идентификатору кластера читает его параметры, формирует helm-чарт, запускает в Kubernetes. По завершении — callback в Хоттабыч.

### Поддержка новых типов кластеров

Чтобы добавить поддержку нового типа кластера:
1. В Kubernetes описать оператора и Custom Resources (как Kubernetes будет обновлять кластер)
2. В Datacenter указать в коде имя Custom Resources для этого типа
3. В Управлении облаком: добавить настройки helm-чарта, выставить флаг «Развернут в Kubernetes»
4. Доработать сборку дистрибутива кластера: выкладывать в ХД json-файл со списком образов `{"helm": {"registry", "repository", "tag"}, "images": {...}}` — чтобы при копировании дистрибутива между хранилищами вызывалось копирование образов контейнеров

## Утилита обновления: вспомогательные классы

Библиотека `sbis-update-generic`:

**`AbstractSvcPartsManager`** — базовый класс обхода сервиса по частям. Инициализируется путём до корня сервиса и списками игнорируемых файлов/папок. Два коллбэка: `разрешить обход части` и `действие с частью`.

**`ServiceHash`** — вычисление хэша сервиса для проверки корректности обновления. При несовпадении общего хэша выдаёт хэши всех частей для анализа расхождения.

**`ServiceFilesMover`** (наследник `AbstractSvcPartsManager`) — перемещение файлов сервиса любого типа дистрибутива. Метод `FastMove`:
1. Исходный путь не существует → выход с успехом
2. Путь назначения существует → `std::filesystem::remove_all(dest)`
3. Попытка `std::filesystem::rename(src, dest)` (работает для вложенных путей без ручного обхода)
4. Если `rename` не прошёл (разные логические разделы) → `std::filesystem::copy(src, opts=recursive)` + `remove_all(src)`

## Анализ модулей при обновлении сервиса (устаревшее)

На фазе подготовки утилита получает файл `modules.json` из ХД:
```json
{
  "bl_modules": {
    "AMQP Client": {
      "link": "http://fix-distr-storage.sbis.ru/api/v1/distribution_files/<uuid>",
      "revision": "018b3cac893fb9bec887cd7d6e18535d",
      "version": "19.610-853"
    }
  },
  "ui_modules": {
    "Application": {
      "link": "...",
      "revision": "40d25a5e1c618409a7730eac881a361d",
      "version": "99.00-2419"
    }
  }
}
```

Алгоритм разворота:
1. Скачать `db-` и `core-` архивы (безусловно)
2. Сравнить ревизии установленных модулей с `modules.json` → список изменённых и неизменённых
3. Неизменённые — скопировать из существующего сервиса в папку подготовки
4. Изменённые — перекачать из ХД
5. Для изменённых модулей поднять версию зависимых модулей в `modules.json`
6. Запустить `jinnee_lite` — проставляет версии из `modules.json` в модулях

`jinnee_lite` при деплое подкладывает в корень каждого модуля файл `filechecksum` с ревизией модуля.
