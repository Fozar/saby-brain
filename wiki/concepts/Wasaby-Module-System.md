# Wasaby Module System

Модульная архитектура Wasaby Framework: плагинная система, порядок загрузки, события модулей, API-доступность, бинарные библиотеки, структура проекта.

## Плагинная архитектура

Wasaby — **плагинная архитектура**: функционал поставляется модулями. Модули описываются файлами `.s3mod`. Один модуль может зависеть от других.

Поиск модулей при загрузке:
- Параметр `Модули` не задан → папка `modules/` рядом с `sbis-lib`
- Параметр = путь к директории → поиск `*.s3mod` в подкаталогах первого уровня
- Параметр = путь к `.s3srv` → список модулей из этого файла

## Фазы загрузки модулей

1. **Анализ зависимостей** — дерево зависимостей, порядок загрузки
2. **Загрузка библиотек C++** — `.so`/`.dll` в порядке зависимостей (платформа не готова к сложной логике!)
3. **Настройка платформоспецифичного окружения** — iOS/Android (Objective-C/Kotlin инициализаторы)
4. **Инициализация ядра** — события `OnEndLoadKernelModules`/`OnKernelCleanup` (только платформа!)
5. **Инициализация прикладных модулей** — `OnEndLoadModules`/`OnCleanup`

> [!warning] `OnEndLoadKernelModules` и `OnKernelCleanup` — только для разработчиков платформы. Прикладным запрещено.

## События сервисного модуля

| Событие | Когда | Язык |
|---------|-------|------|
| `OnEndLoadModules` | После загрузки всех модулей сервиса (прямой порядок зависимостей) | C++, Python |
| `OnEndAllLoadModules` | То же, но срабатывает позже (после всех) | C++, Python |
| `OnCleanup` | Освобождение ресурсов (обратный порядок) | C++, Python |
| `OnCheck` / `OnCheckModules` | Проверка состояния сервиса (опционально) | C++, Python |
| `OnUnload` | Начало выгрузки — **при типовой конфигурации не вызывается** (процессы завершаются через `QuickExit()`) | C++, Python |

### Обработчики на Python

Файл `on_event.py` в корне директории сервисного модуля. Функции именуются как события:

```python
# Регистрация точки входа
def OnEndAllLoadModules(answer):
    try:
        RegisterEntryPoint(TestPyEntryPoint("TestCasePy"))
    except Exception:
        pass
    return True

# Подписка на событие в облаке
import sbis, event
def event_subscription():
    subscription = event.Subscription()
    subscription.channel = 'versionmanager.applypackage'
    subscription.durable = True
    subscription.user_login = 'online_service'
    subscription.SetCallback("RegulationService.PackageCallback", event._MsgBody_)
    subscription.Add()

def OnEndAllLoadModules(answer):
    event_subscription()
    return True
```

**Требование**: установить зависимость сервисного модуля от **Python Core** (из SBIS SDK).

### Обработчики на C++

Наследование `ld::ModuleInitializerHandle`, регистрация через `REGISTER_MODULE_INITIALIZER`. Метод `Name()` должен **точно** совпадать с именем модуля в s3mod-файле:

```cpp
struct MyModuleInitializer : ld::ModuleInitializerHandle {
    StringView Name() const noexcept override { return L"MyModule"_sv; }
    bool OnEndLoadModules(EndLoadEventAnswer& answer) const override {
        InitMyModule();
        return true;
    }
    bool OnCleanup(CleanupEventAnswer&) const override {
        CleanupMyModule();
        return true;
    }
};
REGISTER_MODULE_INITIALIZER(MyModuleInitializer)
```

## Бинарные библиотеки

Файлы: `*.dll` (Windows) / `lib*.so` (Linux), расположены в **корне директории сервисного модуля**.

Атрибуты в Genie:

| Атрибут | Описание |
|---------|----------|
| **Library** | Основная библиотека (одна). Указывать без префикса `lib` и расширения. Если не найдена — модуль сообщает о неработоспособности. |
| **Resources** | Дополнительные библиотеки (несколько), от которых зависит основная. |

> [!note] При создании нового модуля — всегда указывайте `Library`. Это правило хорошего тона.

## API сервисного модуля

Тип доступности методов БЛ (`Accessibility` в Genie):

| Тип | Часть API | Внешний доступ | Описание |
|-----|-----------|----------------|----------|
| **Internal** | — | — | Только внутри сервиса |
| **Trusted Application Contract** | — | только доверенные | Обработчики событий; защита от вызова извне (кроме доверенных точек входа) |
| **Internal Application Contract** | — | + | API для взаимодействия сервисов внутри одного приложения. Без гарантии стабильности. |
| **Service Contract** | + | + | Стабильное API между сервисами внутри облака. Поддерживается между мажорными версиями. |
| **Public Service Contract** | + | + | Крайне устойчивое API (мобильные приложения, внешние клиенты, Sbis Plugin). Менять только по договорённости. |

**External calls** — зависимости от методов других сервисов, описываются в Genie разделе `External calls`:
```xml
<externals>
  <external cloud="." method="ДокументАтрибут.СписокПоФормату" num_args="1"
    service_name="Сервис форматов"
    signature="ДокументАтрибут.СписокПоФормату(ФорматДок: Число целое): Таблица[...]"/>
</externals>
```

## Конфигурация сервисного модуля

Источники конфигурации (в порядке возрастания приоритета):

1. **Метаописание** — значения по умолчанию из s3mod, затем из s3srv, затем из s3cld
2. **Облачная конфигурация** — Оркестратор облака / реестр "Настройки" (только облачные сервисы)
3. **ini-файлы** — дистрибутивные или по известным путям

Формат ini-файла (имя параметра: `секция.ключ`):
```ini
[Базовая конфигурация]
ПаузаПередЗагрузкойМодулей = 30

[Ядро.Права]
Проверять = Нет

[Ядро.Сервер приложений]
ИмяСлужбы = sbis-cfg
```

Поддерживается `#include base.ini` для включения файлов (аналог C/C++).

## Структура проекта приложения (s3cld / s3srv / s3mod)

Трёхуровневая иерархия:

```
s3cld — облако/проект/приложение
  └── s3srv — сервис (БЛ или Представление)
       └── s3mod — сервисный модуль
```

### Атрибуты s3cld

| Атрибут | Описание |
|---------|----------|
| `name` | Уникальное имя приложения (→ стандарт именования) |
| `family_name` | Семейство приложений для группировки (согласовать с Новиковым Д.В.) |
| `sdk_version` | Совместимая версия SBIS SDK |
| `kaizen_zone` | UUID участка Кайдзен |

### Атрибуты s3srv

| Атрибут | Описание |
|---------|----------|
| `name` | Имя в Genie и реестре "Приложения" |
| `system_name` | Уникальное системное имя в облаке = имя директории при деплое |

## Стандарт именования

| Тип | Формат | Пример |
|-----|--------|--------|
| Приложение | `<app_name>[-country]` | `online-ru`, `spp-kz` |
| Сервис (файл) | `<service_name>[-ps][-country]` | `online-kz`, `online-ps-ru` |
| Системное имя | `<service_name>[-ps]` | `online`, `auth-internal` |
| Миксин BL | `<service_name>-bl[-country]` | `online-bl-ru` |
| Миксин UI | `<service_name>-ui[-country]` | `online-ui-kz` |

**Правила:**
- Только ASCII строчные, части через дефис (`-`)
- Не добавлять `service`, `app`, `application` — лишняя нагрузка
- Постфикс страны: нужен если разные дистрибутивы в разных облаках; не нужен если только в ru или единый дистрибутив везде

## Типы сущностей в s3srv-файлах

Один формат файла описывает разные сущности (нет явного признака типа):

| Сущность | Описание |
|---------|----------|
| **Сервис** | Законченное решение в облаке, имеет собственные модули и БД |
| **Шаблон (миксин)** | Набор модулей для переиспользования (наследование). Не является самодостаточным сервисом. |
| **Микросервис** | Аналог сервиса в облаке — для мобильных/десктоп приложений. Имеет БД и модули. |
| **Пул** | Для десктопных решений (Presto, Розница, Sbis Plugin). Описывает политику запуска процессов. Нет БД, нет своего кода. |

## Схема дистрибутива (мульти-страна)

Разделение глобального и регионального функционала через служебные s3srv-файлы:

```
<svc>           — общие модули для всех
<svc>-ps        — общий UI для всех
<svc>-bl-ru     — BL-модули только для России
<svc>-ui-ru     — UI-модули только для России
<svc>-bl-kz     — BL-модули только для Казахстана
<svc>-ui-kz     — UI-модули только для Казахстана
<svc>-ru        — сервис БЛ для России (наследует <svc> + <svc>-bl-ru)
<svc>-ps-ru     — сервис UI для России (наследует <svc>-ps + <svc>-ui-ru)
```
