# Loyalty Desktop Broker Migration

Проект перевода синхронизации лояльности в Desktop Розница/Presto со старого scheduled-механизма на [[Sync-Broker-Architecture|брокер синхронизации]].

---

## Проблема и цель

На текущий момент в online поддерживаются **два механизма** синхронизации лояльности:

| Механизм | Кто использует | Проблемы |
|---|---|---|
| **Новый** — через брокер синхронизации | Мобильная розница | — |
| **Старый** — безусловные регулярные вызовы облачных методов | Desktop Розница/Presto | Трафик избыточен, нет событийности, 2 кодовые базы |

Цель: единый механизм синхронизации через брокер для ~**29 000 копий** Desktop Розница/Presto.

Критерий: переход всех копий (или основной массы — обновляют постепенно ~полгода), сокращение вызовов облачных методов, удаление старого механизма.

> [!key-insight] Старый механизм: `PFSync.SyncLoyalty` вызывается из Каталога по расписанию и всегда отдаёт все сущности. Новый: брокер хранит только факты изменений, клиент тянет только изменённые UUID.

---

## Область сущностей

Перевод включает 5 типов сущностей:

1. **Типы карт** (ВидКарты) — методы `PFSync.CardType(First|Regular)Sync`
2. **Акции** (ВидЦены) — методы `PFSync.PriceEntity(First|Regular)Sync`
   - включая: Бонусные программы, Промокоды, Подсказки, Наценки
3. **Настройки бонусов** (BonusSettings) — методы `PFSync.BonusSettings(First|Regular)Sync`
4. **Настройки лояльности** (Settings) — новые методы `PFSync.Settings(First|Regular)Sync`
5. **Промокоды** — `PromoCode.Sync`, `PromoCode.ReSync` (переносятся из `PFSync.SyncLoyalty`)

Дополнительно: **отказ от синхронизации прайс-корней** (`__сбис__Прайс`, `__сбис__ВидЦены`).

Полный список типов ВидЦены с целевой поддержкой:
`PERCENT_DISCOUNT, SUM_DISCOUNT, RETAIL_GIFT_DISCOUNT, BONUS, GLOBAL_BONUS, REFERRAL_BONUS, SPECIAL_PRICE, HOLIDAY_BONUS, MANUAL_BONUS, PROMOCODE, BONUS_DEC_RULE, PROMPT, PERCENT_MARKUP, MANUAL_MARKUP`

## Архитектура решения

### Общая схема (новый механизм)

```
[Online: изменение сущности]
    ↓ BrokerBuilder.send()
[sync-broker: сохранение UUID]
    ↓ (по запросу offline)
[SyncBroker.GetObjectsWithFilter → изменённые UUID]
    ↓
[PFSync.*FirstSync / *RegularSync → данные по UUID]
    ↓
[Desktop offline: upsert в локальную БД]
```

Первичная синхронизация: итеративно порциями по `@ВидКарты`.
Инкрементальная синхронизация: по UUID.

### Модули

- **PriceFormation.Online** — облачная серверная часть (методы *FirstSync, *RegularSync)
- **PriceFormation.Offline** — клиентская часть Desktop (реализация тех же методов)
- Git: `warehouse/price-formation`

---

## Этапы реализации (на примере ВидКарты)

Аналогичные 5 этапов применяются к каждой сущности:

1. Убедиться что сущность отправляется в брокер (если нет — доработать)
2. Доработать **облачные** методы FirstSync/RegularSync: поддержать `anchor`, `limit`, фильтр-признак Desktop
3. Реализовать/доработать **клиентские** (offline) методы FirstSync/RegularSync
4. **Внедрить** в места запуска синхронизации Desktop (под фичей)
5. **Удалить** старый механизм (после обновления всех клиентов)

### Места запуска синхронизации Desktop

- Авторизация при старте приложения
- Открытие смены
- Расписание: раз в сутки (полная), раз в 10 мин, раз в час

## Целостность данных

**Проблема**: если ВидЦены (акции) синхронизируются раньше ВидКарты — ошибка FK (связь акции с несуществующим типом карты).

**Решение 1 (выбранное)**: соблюдать порядок синхронизации на клиенте:
1. Синхронизировать ВидКарты
2. Синхронизировать ВидЦены (при ошибке FK — дозатянуть нужный тип карты)

**Решение 2**: сложный объект синхронизации (ВидКарты + ВидЦены + ВидЦеныВидКарты). Тяжёлая транзакция, сложнее.

## Bugfix: лимиты промокодов

**Проблема**: при большом числе партнёрских промокодов новые промокоды не доходят в оффлайн.

Причины:
1. Лимит применяется отдельно к наборам `add/upd/del` → обрезает новые данные
2. Партнёрские промокоды возвращались **всегда** (даже без изменений)
   → при лимите 20: 2 новых + 20 партнёрских → в выборку попадают только партнёрские

Исправление:
1. Партнёрские промокоды возвращать только при изменении
2. Не резать `add/upd/del` по отдельности — доверить лимит серверному механизму
3. Гарантировать доставку новых промокодов
