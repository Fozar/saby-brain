# Bonus.GetSaleList — дубли W-записей при итеративной загрузке

## Симптом

`Controls/list` выдаёт ошибку:
```
Сырые данные содержат дубли записей с ключом "Id"="W37328".
Списочный метод: "Bonus.Bonus.GetSaleList"
keyProperty: "Id"
```

Воспроизводится при: реестр «Бонусы/Покупки», период апрель 2026, аккаунт с большим объёмом данных.

## Root Cause

Используется `BonusSaleListIterative` (флаг `LOYALTY_IT_NAV`). Итеративная загрузка разбивает `ВидЦеныДокумент` на блоки по `EffectiveDate`.

Складской документ (например, DocumentId=37328) имеет **две записи** `ВидЦеныДокумент` с разными `EffectiveDate` — например, начисление бонусов (`T_high`) и корректировка/списание (`T_low`). При большом объёме данных за апрель 2026 граница блока разрезает эти строки:

- **Блок N**: видит строку с `T_high` → `BonusWarehouseSales GROUP BY DocumentId` → одна запись с `DateWTZ=T_high` → Id=`W37328` в ответе
- Курсор устанавливается на `DateWTZ` последней отображённой записи → `T_low < cursor`
- **Блок N+1**: загружает строки с `EffectiveDate < cursor` → видит `T_low` для того же документа → Id=`W37328` снова
- Фронтенд накапливает записи из обоих ответов → дубль

При малом объёме данных обе строки попадают в один блок и агрегируются в одну запись (GROUP BY DocumentId). Ошибка не проявляется.

## Связанные файлы

- `www/service/Модули/PriceFormation.Online/priceformationonline/loyaltyprograms/bonus/get_sale_list.py` — класс `BonusSaleListIterative`
- `tests/tests_priceformationonline/loyaltyprograms/bonus/get_sale_list.py`

## Фикс

Добавлены два новых CTE в SQL-запрос `BonusSaleListIterative` — после `UnfilteredRecords`:

```sql
"SuspectWarehouseDocs" AS (
    SELECT DISTINCT ur."Документ"
    FROM "UnfilteredRecords" ur
    WHERE ur."Документ" IS NOT NULL
      AND EXISTS (
          SELECT 1
          FROM "ВидЦеныДокумент" vcd
          JOIN "BonusPriceEntities" ON vcd."ВидЦены" = "BonusPriceEntities"."@ВидЦены"
          WHERE vcd."Документ" = ur."Документ"
            AND vcd."EffectiveDate" < (SELECT MIN("EffectiveDate") FROM "UnfilteredRecords")
            AND vcd."EffectiveDate" >= COALESCE(!PeriodStart::TIMESTAMP, '-INFINITY')
            AND vcd."EffectiveDate" < COALESCE(!PeriodEnd::TIMESTAMP + interval '1 day', 'INFINITY')
      )
),
"SuspectSaleIds" AS (  -- аналогично для розничных продаж
    ...
)
```

**Смысл**: если у документа есть строки `ВидЦеныДокумент` в пределах периода, но **ниже минимального `EffectiveDate` текущего блока** — документ будет виден снова в следующем блоке → исключаем его сейчас. Документ появится в более позднем блоке, когда его самая ранняя строка окажется внутри блока.

В `BonusWarehouseSales` добавлен WHERE:
```sql
AND "UnfilteredRecords"."Документ" NOT IN (SELECT "Документ" FROM "SuspectWarehouseDocs")
```

В `BonusRetailSales` аналогично:
```sql
AND "UnfilteredRecords"."Sale" NOT IN (SELECT "Sale" FROM "SuspectSaleIds")
```

## Индексы — проверено безопасно

EXISTS-подзапросы используют:
- `AutoGeneratedIndexDokument (Документ)` — lookup по DocumentId, фильтр EffectiveDate в heap
- `Retail (Sale, ДатаВремя) WHERE Sale IS NOT NULL` — lookup по SaleId

Составного индекса `(Документ, EffectiveDate)` нет, но документы имеют 1–5 бонусных строк → производительность приемлема без нового индекса в `.dicx`.

## Тест

`test_8_no_duplicate_when_warehouse_doc_has_two_effective_dates` в `tests_priceformationonline/loyaltyprograms/bonus/get_sale_list.py`:
- Создаёт СУ-документ с двумя `PriceEntityWarehouseSaleDoc` при разных `EffectiveDate`
- Патчит `BonusSaleListIterative.get_result` чтобы не переопределять block_size в 2B (обход `WithoutResults=True` → 2_000_000_000`)
- Устанавливает `block_size=1` — обе строки гарантированно в разных блоках
- Листает страницы через FORWARD-навигацию (cursor из `Metadata().Get('nextPosition')`)
- Проверяет, что W-Id не повторяется
