# Wasaby Service Node Architecture & Configuration

## Структура процессов

```
sbis-daemon (systemd)
  └─ Монитор (корневой)
       └─ Координатор (основной)
            ├─ Эталонный процесс (pool 0 / основной)
            │    ├─ Рабочий процесс 1
            │    ├─ Рабочий процесс 2
            │    └─ ...
            └─ Эталонный процесс (pool 1 / служебный)
                 ├─ Рабочий процесс 1
                 └─ ...
```

| Процесс | Роль |
|---------|------|
| **Монитор** | fork() → запуск координатора; перезапуск при аварийном завершении |
| **Координатор** | Приём запросов, балансировка, рассылка ответов, управление пулами |
| **Эталонный процесс** | Предзагрузка всех модулей; отвечает только на fork() для быстрого рождения рабочих |
| **Рабочий процесс (pool 0)** | Приоритетные запросы: БЛ, интерфейс |
| **Рабочий процесс (pool 1)** | "Тяжёлые" запросы, зачастую async |

> На Linux — эталонный процесс + рабочие; на Windows — только рабочие (без эталонного).

## Файловая структура сервиса

```
<service_root>/
  *.so                  # бинарные компоненты ядра
  sbis-rpc-service.ini  # конфигурация
  version.xml           # версии ядра и сервиса
  modules/              # модули сервиса
  logs/
  asynccall/            # сохранённые async-вызовы
  resources/            # статические ресурсы
```

`sbis-rpc-service.ini` минимальный вид:
```ini
[Базовая конфигурация]
Конфигурация=Адрес=http://test-cloud-ctrl-private.sbis.ru/cfg
```

## Последовательность запуска (14 шагов)

1. `sbis-daemon` → fork() → **монитор**
2. Монитор → fork() → **координатор**
3. Координатор загружает `sbis-rpc-service.so` (ядро)
4. `sbis-loader.so` находит и загружает бинарники модулей
5. Фабрика конфигураций даёт провайдер (INI или Cloud Interaction)
6. Координатор вызывает функции инициализации модулей
7. Строится дерево зависимостей из `<depends>` .s3mod-файлов
8. `OnCleanup()` для каждого модуля (обратный порядок)
9. `sbis-daemon` запускает эталонные процессы (сначала pool 0, затем pool 1)
10. Эталонный процесс загружает модули, вызывает `OnEndLoadKernelModules()`, ждёт fork()
11. Координатор последовательно шлёт команды fork() эталонному процессу
12. Эталонный → fork() → рабочий процесс; вызовы `OnEndLoadModules()`, `OnCleanup()`
13. Рабочий процесс сигнализирует готовность координатору
14. Координатор открывает точки входа

## Ключевые параметры конфигурации

### Количество процессов

| Параметр | Умолч. | Пул |
|----------|--------|-----|
| `Ядро.Сервер приложений.ЧислоРабочихПроцессов` | 8 | основной |
| `Ядро.Сервер приложений.ЧислоСлужебныхРабочихПроцессов` | 2 | служебный |

**Рекомендация**: 1 процесс на ядро CPU; 1 ядро резервировать координатору.

### Количество потоков

| Параметр | Умолч. |
|----------|--------|
| `Ядро.Сервер приложений.ЧислоПотоковВРабочихПроцессах` | 1 |
| `Ядро.Сервер приложений.ЧислоПотоковВСлужебныхРабочихПроцессах` | 1 |

На практике на основных сервисах — 2 потока. Увеличивать, если метод часто ходит в БД или делает внешние вызовы. Python и JS — однопоточные интерпретаторы; для CPU-задач потоки не помогут (только процессы).

### Тип очереди

`Ядро.Сервер приложений.Очередь` (по умолчанию: FIFO)

| Очередь | Поведение |
|---------|-----------|
| `FIFO` | Первым пришёл — первым обслужен; риск монополизации пула одним методом |
| `PQ` | Приоритетная; не занимает весь пул одними методами |
| `SESSION_BALANCED_QUEUE` | Балансировка по сессиям; не занимает пул одним пользователем |

### Предел памяти

`Ядро.Сервер приложений.ПределИспользованияПамяти` (Мб, по умолч.: 603)

```
limit = (RAM - 0.5) * 0.8 * 1024 / (workers + 1)
```
где RAM — в Гб, workers — число рабочих процессов.

Проверка корректности: `workers * limit <= RAM * 1024`

## QoS — система автоматического перезапуска

### Контроль памяти

1. Процесс периодически опрашивает память (Linux: RES−SHR; Windows: RES)
2. Превышение лимита → координатор перестаёт давать новые задачи
3. Если после завершения всех задач память всё ещё выше — процесс завершается
4. **Обслуживание запросов не прерывается** (graceful)

Координатор НЕ перезапускается — его превышение лимита (408 944 640 байт) означает утечку, нужно ответственного.

### Контроль времени выполнения (Execution timeout)

- Задаётся в Genie атрибутом `Execution timeout (мс)`
- Применяется **только к внешним вызовам** (не к `BLObject.Invoke` внутри сервиса)
- Если задача просрочена и все активные задачи просрочены → процесс завершается
- Если есть задачи в норме — процесс продолжает после завершения просроченных

### Контроль утечек транзакций

- Незакрытая транзакция по окончании метода → **немедленный аварийный перезапуск**
- Прерывает ВСЕ текущие запросы в данном процессе
- Защита от десинхронизации данных в кластере

### Контроль зависания

| Параметр | По умолчанию |
|----------|-------------|
| `Ядро.Сервер приложений.ПроверятьЗависаниеРабочегоПроцесса` | включён |
| `Ядро.Сервер приложений.ДампЗависшегоПроцесса` | выключен |

Координатор пингует каждый рабочий процесс. Таймаут ответа — **2 минуты**. Нет ответа → процесс немедленно завершается.

В логах при срабатывании:
```
Дочерний процесс (poolX) (pidY) (state 6) был завершен по запросу координатора
```
