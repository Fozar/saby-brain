# Типы работ — Реестр работ

Реестр работ состоит из папок, объединённых по типу работ, которые внутри разделяются по версиям и датам.

> [!key-insight] Автозаполнение
> Для всех типов работ действует алгоритм автоматического заполнения полей карточки работы из документа на online-master-ru. Набор полей различается по типу работы. Автоматически заполненные поля выделяются синим.

## Таксономия типов работ

| Тип | Назначение | Ключевые поля |
|-----|-----------|---------------|
| **Обновление** | Обновление облачных приложений через Хоттабыч | Тип*, Приложение/Дистрибутив, Версия, Ссылка, Тип обновления |
| **Скрипт** | Автоматический запуск скриптов Хоттабыча | Приложение*, Ссылка (ВНР), Скрипт (архив из ВНР), Режим запуска |
| **Разворот данных** | Разворот пакетов данных на облачных приложениях | Приложение*, Версия*, Название пакета, Ссылка |
| **Пользовательская работа** | Любая ручная работа сотрудников ЦОД | Название*, Ссылка, Версия |
| **Регистрация значений параметров** | Автоматическое изменение параметров облака | Ссылка* (на документ регистрации) |
| **Патч** | Применение патчей к облачным приложениям в авторежиме | Приложение*, Ссылка* (ВНР), Патч (архив), Волна |
| **Кластеры** | Обновление кластеров только в Kubernetes | Кластеры* (из справочника), Версия*, Ссылка* |
| **Включение нового функционала** | Автоматическая активация фич, настроенных тестированием | Ссылка* (ВНР с идентификатором фичи) |
| **Внешнее приложение** | Обновление необлачных приложений, мобильных приложений и расширений браузера | Приложение*, Версия*, Ссылка*, Магазины, Сценарий |

*— обязательное поле

## Этап (Stage)

Этапы — папки для группировки работ в плане. Могут быть вложенными.

Ключевые поля ограничений запуска этапа:
- **Работы до** — обязательные условия разрешения запуска
- **Время запуска** — дата и время старта; наследуется от родительских этапов если не задано
- **Количество одновременно запущенных задач** — ограничение для всей вложенности
- **Ограничение запуска фазы обновления** — отдельно для фазы обновления
- **Работы после** — запускаются после всех работ этапа

## Детали по типам

### Обновление

Используется для облачных приложений через Хоттабыч.

**Порядок добавления:**
1. Если документ создан вручную — прогнать через сборку `http://cis.sbis.ru/view/utility/job/set_fields_on_os/`
2. Указать ссылку на ОС на online-master-ru
3. Поля автоматически выгружаются
4. Проверить что ВСЕ УСЛОВИЯ НАСТРОЙКИ, ОГРАНИЧЕНИЯ и ЗАВИСИМОСТИ из комментария соблюдены

**Добавление нового приложения/сервиса:**
- Пользовательская работа (разворот)
- Обновление (Тип "Полное" если новая БД, "Лёгкое" если без БД)
- Для каждого типа работ должна быть указана версия

### Скрипт

Автоматический запуск скриптов. Архив скрипта автоматически подтягивается из ВНР по ссылке.

**Режим запуска:**
- `Копия сервиса` (по умолчанию) — создаётся полная копия БЛ со всеми модулями
- `Минимальный сервис` — копия с минимально необходимым набором модулей (снижает нагрузку)

**Поле "Ограничение одновременного выполнения"** — количество клиентов (только для приложений основного сервиса). Значение указывает автор ВНР.

**Внимание:** для скриптов по основным сервисам (online-master-ru) — отбираются приложения online для ВСЕХ регионов, НЕ смотря на флаги.

### Патч

**Важные правила:**
- Запуск патчей возможен **только от этапа** (на самой работе кнопка скрыта)
- Настройка зависимостей — только на этапе
- Все патчи по одному приложению/дистрибутиву должны быть в **одном этапе** — тогда все патчи применятся, и запустится обновление на этот дистрибутив
- Дополнительные флаги: «Принудительная замена файлов», «Остановить сервисы отсутствующие в дистрибутиве», «Не показывать уведомления об обновлении» (обязателен для дневных патчей по основным)

### Регистрация значений параметров

Используется автоматически если:
- Все регистрируемые значения принадлежат одному приложению
- Можно указать значение "По всем приложениям"
- Задаваемое значение параметра переписывается полностью
- Число регистрируемых значений ≤ лимит (иначе разбить на несколько документов)

> [!warning] Ограничение
> Нельзя автоматически регистрировать параметры типа «Таблица паролей» — добавляется как «Пользовательская работа».

### Кластеры (Kubernetes)

Автозаполнение работает только для ОС. Для ВНР — кластеры заполняются руками.

**Таблица ключей автозаполнения** (для ботa Макса Сорокина в @ReleaseManagerTenBot):

| Путь | Ключ |
|------|------|
| Балансировщики-HTTP nginx_ks | `nginx_ks` |
| Системы кэширования/Varnish | `varnish` |
| Балансировщики-HTTP/Общие/sbis.ru | `sbisru` |
| Системы логирования/Graylog | `graylog` |
| Брокеры сообщений/RabbitMQ/Асинхронные вызовы/rabbitmq-async-main | `rabbitmq-async-main` |
| Брокеры сообщений/RabbitMQ/Асинхронные вызовы/rabbitmq-async-proxy | `rabbitmq-async-proxy` |
| Брокеры сообщений/RabbitMQ/Серверные события/rabbitmq-server-event-main | `rabbitmq-server-event-main` |
| Брокеры сообщений/RabbitMQ/Серверные события/rabbitmq-server-event-proxy | `rabbitmq-server-event-proxy` |
| Брокеры сообщений/RabbitMQ/STOMP/eventmq-route | `eventmq-route` |
| Брокеры сообщений/RabbitMQ/STOMP/eventmq-web | `eventmq-web` |
| Брокеры сообщений/RabbitMQ/STOMP/eventmq-web-guest | `eventmq-web-guest` |

Версия и билд — по версии helm (вкладка Helm на карточке приложения в Хоттабыче или в облаке через Домены → Балансировщики HTTP).

### Внешнее приложение (необлачные, мобильные, расширения браузера)

- Распространяется через сторонние магазины и update.sbis.ru
- Сценарии обновления — свод правил, формируется в файл, загружается с дистрибутивом на зеркала
- Если в пути загрузки стоит только `{zip_name}` — сценариев быть не должно

### Включение нового функционала

1. Тестировщик в облаке настраивает фичу и сохраняет значение **без применения**
2. В ВНР автором указывается идентификатор фичи + условия и зависимости
3. При выполнении работы — план просто применит сохранённое значение

### Пользовательские работы — что НЕ умеет Хоттабыч (добавляются вручную)

- Скрипты, не относящиеся к Хоттабычу
- Визуализации
- Работы на серверах (БД и БЛ) облачных приложений
- Работы на диспетчерах или RabbitMQ
- Разворот кластеров (если ВНР, а не ОС)
- Настройки УЦ
- Обновление форматов
- Импорт классификаторов
- Обновление окружения (RPM) — название обязательно должно содержать «rpm»
- Другие ручные работы сотрудников ЦОД

## Запрет дублирования работ

| Тип работы | Ключевые параметры сравнения |
|------------|------------------------------|
| Обновление | Тип + Ссылка + Версия + Билд + Приложение |
| Разворот данных, Кластеры, Внешнее приложение | Тип + Ссылка + Версия + Билд + Приложение/Пакет/Кластер + Волна |
| Скрипт, Патч | Тип + Приложение(я) + Ссылка |
| Фича, Параметр | Тип + Ссылка |

Исключение: разрешено добавление работ с одной версией но разными билдами (если в текущей работе указаны оба поля). Пользовательские работы не ограничены.

## Автозаполнение и синхронизация полей

- При вставке/замене ссылки на документ — автоматически выгружаются поля
- При принудительной синхронизации
- При открытии карточки — расхождения помечаются жёлтым треугольником
- Поля, заполненные вручную, помечаются флагом «Поле изменено вручную» — принудительная синхронизация их НЕ перезаполняет

## Переименование приложений

Специальный процесс через план выпуска:
1. ВНР — документ с описанием переименования и регионами
2. ОС — обновление с НОВЫМ именем, завязанное на выполнение ПОСЛЕ ВНР
3. Теги «ПЕРЕИМЕНОВАНИЕ» на оба пункта плана
4. В пункте с ОБНОВЛЕНИЕМ — флаги «Принудительная полная замена файлов» + «Остановить сервисы отсутствующие в дистрибутиве» + флаги регионов как в ВНР
5. Поручение на Абакова С., Давыдова А. и Сорокина М. с указанием версии и даты нового имени
