# Wasaby Service Architecture

Внутренняя архитектура узла web-сервиса Wasaby Framework: процессная модель, запуск, файловая структура.

## Процессная модель

Типовой узел web-сервиса состоит из **4 типов процессов**:

```
sbis-daemon
└── Монитор (fork от sbis-daemon)
    └── Координатор (fork от монитора)
        ├── Эталонный процесс Pool 0 (основной)
        │   ├── Рабочий процесс 1
        │   └── Рабочий процесс 2 ...
        └── Эталонный процесс Pool 1 (служебный)
            ├── Рабочий процесс 1
            └── Рабочий процесс 2 ...
```

### Роли процессов

| Процесс | Роль |
|---------|------|
| **Монитор** | Первый процесс. fork → координатор. Перезапускает координатор при аварийном завершении. |
| **Координатор** | Принимает запросы, балансирует нагрузку, хранит точки входа. Запросы сам не исполняет. |
| **Эталонный** | Быстрое воспроизведение рабочих процессов через fork(). Не выполняет запросы, минимально потребляет память. |
| **Рабочий** | Обрабатывает запросы. Результат — в разделяемую память → уведомление координатора. |

> [!note] На Windows эталонных процессов нет.

### Пулы процессов

- **Pool 0 (основной)** — приоритетные запросы: БЛ, интерфейс.
- **Pool 1 (служебный)** — тяжёлые/асинхронные запросы.

### Многопоточность

По умолчанию — **однопоточно**. Параметры для включения многопоточности:
- `Ядро.Сервер приложений.ЧислоПотоковВРабочихПроцессах`
- `Ядро.Сервер приложений.ЧислоПотоковВСлужебныхРабочихПроцессах`

> [!warning] Python и JS — однопоточные интерпретаторы. Если метод CPU-bound — потоки не помогут, только процессы.

## Файловая структура сервиса

```
<service_root>/
├── *.so                    # бинарные компоненты ядра платформы
├── sbis-rpc-service.ini    # базовая конфигурация
├── version.xml             # версия ядра + сервиса
├── modules/                # сервисные модули
├── logs/                   # логи
├── asynccall/              # сохранённые асинхронные вызовы
└── resources/              # статические ресурсы
```

**version.xml** содержит `<version_core>` (версия/дата/номер сборки ядра) и версию сервиса.

**sbis-rpc-service.ini** — минимальная конфигурация:
```ini
[Базовая конфигурация]
Конфигурация=Адрес=http://test-cloud-ctrl-private.sbis.ru/cfg
```

## Последовательность запуска (14 шагов)

1. `sbis-daemon` стартует → fork → **монитор**
2. Монитор → fork → **координатор**
3. Координатор загружает `sbis-rpc-service300.so` (всё ядро)
4. `sbis-loader.so` ищет и загружает бинарные файлы модулей в адресное пространство
5. Ядро обращается к фабрике конфигураций (ini-файл ИЛИ Cloud Interaction → "Управление облаком")
6. Координатор получает конфигурацию → вызывает функции инициализации каждого модуля
7. Строится дерево зависимостей по `<depends>` в s3mod-файлах → порядок загрузки
8. Вызов `OnCleanup()` для освобождения ресурсов загрузки (обратный порядок)
9. `sbis-daemon` запускает эталонные процессы (args: тип пула, разделяемая память)
10. Эталонный получает конфигурацию от координатора, загружает модули, вызывает `OnEndLoadKernelModules()`, ждёт команд на fork
11. Координатор последовательно посылает команды fork эталонному
12. Эталонный порождает рабочий процесс, вызывает `OnEndLoadModules()` и `OnCleanup()`
13. Рабочий процесс готов → уведомляет координатора
14. Все пулы запущены → координатор **открывает точки входа**
