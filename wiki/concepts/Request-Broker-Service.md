# Сервис request-broker

Специализированный брокер асинхронного обмена для DWC и Scheduler. Ключевое отличие от RabbitMQ: **возможность запросить статус конкретного запроса по ID** в любой момент.

## Проблема, которую решает

RabbitMQ не позволяет идентифицировать отдельный запрос. Если DWC/Scheduler отправил задачу и она потерялась — слот занят до таймаута. При большом таймауте (~часы) это ведёт к длительному простою.

request-broker предоставляет:
- Гарантированную доставку асинхронных вызовов
- API для опроса текущего статуса любого запроса по ID
- Корректирующие процедуры: потерянная задача → признаётся сбойной → слот освобождается

## Архитектура (3 компонента)

```
БЛ → [Управляющий сервис] ← → [Исполнительные узлы (request-broker-node)]
                                         ↕
                              [Backup-сервис (request-broker-backup)]
```

| Компонент | Роль |
|---|---|
| **Управляющий сервис** | Распределение очередей по исполнительным узлам; реагирование на смену конфигурации; API статусов |
| **Исполнительные узлы (node)** | Приём и передача сообщений; 3 пула потоков: приём, диск I/O, сеть I/O |
| **Backup-сервис** | Холодное хранилище всех запросов; компенсирующая процедура (каждые 5 мин) |

## Алгоритм отправки

1. Первая отправка сервису X → запрос управляющего сервиса → список исполнительных узлов (кэш 5 мин)
2. Случайный узел → синхронный HTTP-запрос → ID запроса в ответе
3. При ошибке "не туда" → перезапрос конфигурации и повтор

## Алгоритм приёма (подписчик)

1. Старт → список исполнительных узлов
2. TCP-соединения с узлами; узлы отправляют задачи "по заявке" (pull-модель)
3. Heartbeat: раз в **1 минуту** сервис подтверждает текущие задачи
4. Перезапрос конфигурации: раз в **5 минут**
5. Завершение → сигнал на узел

## Механизм повышенной гарантии (backup)

```
1. Клиент регистрирует копию запроса в backup
2. Клиент отправляет запрос на node
3. Node доставляет исполнителю
4. Исполнитель завершает → node удаляет из backup + пишет ID в Redis
```

Компенсирующая процедура (5-минутный интервал):
- Проверяет запросы с `LastCheckedAt` > 5 мин
- Если нет в broker и нет в Redis → помечает "подозрительным"
- Если уже "подозрительный" → повторная публикация в broker

## Распределение сервисов по узлам

```
hash(service_name) mod N → базовый узел
следующие узлы: (base + k*(N/n)) mod N, где n — кол-во узлов для сервиса
```

Минимум 2 узла на каждый сервис. DWC может получить больше через конфиг.

## База данных исполнительных узлов (berkeley-db)

| Каталог | Содержимое |
|---|---|
| `queues/` | Оперативные очереди заданий (key=ID, value=тело+заголовки) |
| `limbo/` | Список отправленных задач (для восстановления после рестарта) |
| `overflow/` | Неоперативное хранилище при переполнении; обратная загрузка раз в мин |

## База данных backup-сервиса

Шардирование: 64 виртуальных корзины → физические шарды. Ключ шардинга = ID запроса.

Таблицы в каждом шарде:
- `RequestMeta` — заголовочная (фиксированная структура)  
- `RequestData` — данные, секционированные по времени поступления
