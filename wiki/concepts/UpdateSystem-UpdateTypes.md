# Типы обновлений

Таксономия типов обновлений, которые поддерживает Хоттабыч.

## Облачные обновления

### Без конвертации БД

#### Лёгкое обновление (umLite = 2)

Нет изменений схемы БД — пользователи продолжают работать. Обновляются: BL-узлы, presentation-сервисы, CDN-модули, регистрация метаданных.

Алгоритм:
1. Узлы каждого сервиса делятся на 2 группы
2. Облако → Диспетчер: все запросы на 2-ю группу
3. Обновляется 1-я группа, прогревается, запускается
4. Устанавливается режим двух версий: запросы идут к версии, с которой был отправлен запрос
5. Аналогично обновляется 2-я группа
6. Облаку сообщается о полной готовности

> Требует минимум **2 узла** для каждого BL и СП.

#### Лёгкое последовательное обновление (umSequentialLite = 4)

Для снижения влияния на производительность; основное использование — патчи в дневное время. Изменения между версиями должны быть незначительными и совместимыми.

Алгоритм:
- Узлы BL (не СП) обновляются **по одному** (последовательно)
- Узлы СП — по двум частям (как при лёгком)
- После завершения 1-й части СП → клиенты переходят на новые узлы → обновляется 2-я часть СП

#### Патчи

Быстрое внесение изменений без полного цикла обновления. Создаёт новую версию дистрибутива с добавлением номера патча. Применяется на **все сервисы** с изменённым модулем. Подробно: [[UpdateSystem-Patches]].

#### Скрипты

Разовое выполнение произвольного кода в контексте существующего сервиса. Хоттабыч разворачивает копию сервиса, внедряет модуль `DeveloperScript`, выполняет через DWC. Подробно: [[Wasaby-Scripts]].

#### Разворот пакетов данных

Доставка архива с данными к сервису (без фаз подготовки и очистки).

Алгоритм:
1. Разработчик собирает дистрибутив пакета данных → выкладывает в ХД
2. Указывает дистрибутив на карточке приложения в облаке
3. Хоттабыч публикует сервисное событие → сервис скачивает архив и обрабатывает

**Сервисные события:**

| Событие | Направление | Ключевые поля |
|---------|-------------|---------------|
| `versionmanager.apply.package` | Хоттабыч → сервис | `Id`, `Product`, `Package`, `Version`, `Link: String`, `Services: StringArray` |
| `versionmanager.package.started` | сервис → Хоттабыч | `Id`, `Product`, `Version`, `Link`, `Error: bool`, `Message` |
| `versionmanager.package.finished` | сервис → Хоттабыч + cloud-ctrl | `Id`, `Product`, `Package`, `Version`, `Error: bool`, `Message` |

> Событие `apply.package` рассылается всем продуктам — сервис определяет релевантность по полям `Product`, `Package`, `Services`. `package.finished` обрабатывается и сервисом управления облаком: версия пакета прописывается на карточке продукта (если `Error != True`).

#### Развертывание приложения

Начальный деплой или повторное развёртывание приложения в облаке.

### С конвертацией БД

#### Полное обновление

Полная блокировка и остановка приложения. Парковочная страница с новостью об обновлении.

Алгоритм:
1. Облаку: установить режим полного обновления
2. Облако блокирует все сервисы → Диспетчер отдаёт парковочную страницу
3. Хоттабыч останавливает все узлы сервиса
4. Конвертация БД
5. Все узлы обновляются одновременно, прогреваются, стартуют на новой БД
6. Облаку: снять режим полного обновления → разблокировка

#### Лёгкая конвертация

Конвертация БД в **неблокирующем режиме**: приложение не блокируется, jinnee-utility не разрывает соединения.

Применимо, если изменения не повредят текущей работе сервисов. Для оценки — отчёт «Анализ изменения БД» в Хоттабыче.

Алгоритм:
1. Узлы каждого сервиса делятся на 2 группы
2. Запускается конвертация БД
3. Параллельно: обновляется 1-я группа, затем 2-я (аналогично лёгкому)

**Можно делать через лёгкую конвертацию:**
- Создание триггера / триггерной функции (если не мешает старой версии)
- Любые операции с таблицами малого размера (суммарно < 3 мин)
- Разворачивать пакеты данных
- Создавать новые поля/таблицы без заполнения

**Нельзя делать через лёгкую конвертацию:**
- Добавить и заполнить столбец в большой таблице (от N записей)
- Модификация ограничений целостности для большой таблицы
- Построение индекса на большой таблице

> Удаление полей — через лёгкую конвертацию, но в 2 обновления: сначала убрать использование, потом удалить.

#### Посхемное обновление

Для **персональных приложений** с tenant-группами. Простой только у клиента, чья схема конвертируется в данный момент.

Алгоритм:
1. Фаза подготовки: доставка дистрибутивов, подготовка данных в БД (без остановки)
2. Фаза обновления: BL и статика делятся пополам; одна половина обновляется
3. Конвертация посхемно: клиент работает на узлах старой версии → при наступлении его очереди блокируется → конвертируется его схема → разблокируется → направляется на новые узлы
4. После конвертации всех схем: 2-я группа BL и статики обновляется

#### Мастер-реплика

Конвертация БД без простоя сервиса, но с **ограничением записи** в момент конвертации. Требует read-реплики у каждой БД приложения.

Алгоритм:
1. Все BL-серверы работают с master + replica
2. Все BL переключаются только на replica
3. Репликация отключается
4. Конвертируется master-база
5. BL и статика обновляются; конфигурация предписывает работать с master
6. Репликация возобновляется; Хоттабыч ждёт, пока replica «догонит» master
7. Все BL переключаются обратно на master + replica

> В пп. 2–5 — серверы в режиме "только чтение". Все запросы на запись возвращают ошибку.

**Модуль Seamless Update Support** (опционально, не в стандартных шаблонах):

Добавляется в сервис для кастомизации поведения в режиме мастер-реплики:
- Асинхронные вызовы откладываются; после завершения обновления re-queue по сигналу от Хоттабыча
- Синхронные модифицирующие методы → ошибка "Сервис обновляется..."
- Read-only методы пропускаются на исполнение

Облачные параметры модуля:

| Параметр | Описание |
|---------|---------|
| `ЯдроБесшовноеОбновлениеРаботатьВРежимеОбновления` | Включить режим вручную (без реальной мастер-реплики) |
| `ЯдроБесшовноеОбновлениеСписокИсключенийМодифицирующихМетодов` | Модифицирующие методы, которые не блокируются |
| `ЯдроБесшовноеОбновлениеСписокИгнорируемыхМетодов` | Методы без каких-либо проверок |
| `ЯдроБесшовноеОбновлениеРазрешитьМодифицирующиеМетоды` | Разрешить все модифицирующие методы |

#### Подготовка конвертации БД

Разделяет обновление с конвертацией на **два запуска** для снижения нагрузки.

1. Первый запуск (тип «Подготовка конвертации БД»): затрагиваются только узлы БД; выполняются только подготовительные действия (новые таблицы, индексы и т.п.); хэши схем сохраняются с увеличенным временем хранения
2. Через некоторое время — основное обновление с конвертацией: Хоттабыч определяет что подготовка выполнена и пропускает выполненные действия

> Флаг «Принудительно запустить подготовку конвертации» — принудительно повторить подготовку (если в дистрибутив пришла критичная доброска с правкой конвертации между запусками, что делает хэши невалидными).

## Обновление приложений с общими базами

Для семейства приложений, использующих общие БД: `inside`, `online-master-ru`, `online-master-ca`, `online-master-mili`, `online-master-vip`.

Запускается обновление master-приложения → Хоттабыч автоматически запускает обновление slave-приложений с тем же типом. Конвертацию выполняет только `online-master-ru`; остальные синхронно выполняют необходимые действия. В истории запусков slave-обновления отмечены специальной иконкой.

> Из планов выпуска: одна работа по дистрибутиву корректно запускает и master, и всех slaves.

## Обновление внешних приложений

Типы: офлайн-приложения, мобильные приложения Тензора, браузерные расширения.

Управляются через сервис `apps`: Обновления → Приложения → вкладка Внешние. Запуск — через мастер обновления (как для облачных).

**Создание внешнего приложения:**
1. В разделе «Дистрибутивы» создать дистрибутив (название = название приложения)
2. Указать тип приложения, создать версию, загрузить zip-архивы (имена архивов = имена директорий для распаковки)
3. В «Настройки → Облака» добавить балансировщик (приложение «Загрузка обновления СБИС (Голубев АФ)»)
4. В разделе «Приложение» нажать «+ Приложение» (название = название дистрибутива)

Вид загрузки:
- **С заменой** — старые файлы удаляются, создаётся бэкап для отката
- **С дополнением** — добавляются/заменяются файлы, бэкап не создаётся, откат невозможен

Во время обновления офлайн-приложения загружаются на update.sbis.ru.

## Обновление кластеров

Справочник кластеров в Хоттабыче строится из API-метода облака. Иерархический: типы кластеров → папки (несколько уровней) → экземпляры.

**Запуск обновления** доступен только на уровне **папок** (не на отдельных экземплярах). В диалоге запуска — выбор конкретных кластеров из папки.

Структура истории операций при обновлении кластеров:
- `ИсторияОперацийПродукта` — идентификатор обновления
- `ИсторияОперацийСервиса` — группировка по разделам (папкам)
- `ИсторияОперацийЛогики` — конечные экземпляры кластеров

> Экземпляр кластера = единица обновления для Хоттабыча, хотя реально за ним может стоять несколько серверов. Фазы подготовки, очистки и т.д. формально присутствуют, но содержательно выполняются только фазы обновления и отката.

## Облачные параметры обновления (группа «Управление версиями.Обновления»)

| Тип обновления | Параметр | Описание |
|----------------|---------|---------|
| Лёгкое | `ПоследовательныйРазворотДанных` | Последовательный (а не параллельный) разворот данных внутри сервиса |
| Лёгкое | `РазворотДанных` | Включить разворот данных при лёгком обновлении |
| Лёгкая конвертация | `ПроверятьДублирующиеСервера` | Проверка дублирующих серверов при лёгкой конвертации (по умолчанию включена) |
| Полное | `ЧислоЗадержекРазблокировкиПользователей` | Коэффициент: сколько интервалов ждать перед ошибкой если % разблокированных не меняется |
| Посхемное | `ИнтервалУведомлений` | Интервал между уведомлением клиента и началом обновления; он же интервал между группами клиентов |
| Посхемное | `КоличествоУведомляемыхКлиентов` | Число клиентов на хост БД, уведомляемых за один интервал |
| Посхемное | `ТаймаутОжиданияМетодовКлиента` | Ожидание завершения методов под сессией клиента после блокировки; по истечении конвертация запускается безусловно |
| Посхемное | `УведомлятьКлиентов` | Нужно ли отсылать уведомление с возможностью отложить |
| Посхемное | `ЧислоПотоковКонвертераНаХост` | Потоков конвертера на хост при посхемной конвертации |
| Мастер-реплика | `ДополнительныйТаймаутНормализацииЛага` | Доп. время ожидания нормализации лага реплики после конвертации мастера (сверх времени основной фазы) |
| Разворот данных | `ТаймаутЗапуска` | Время, за которое сервис должен ответить что разворот запущен |

## Настройки обновления приложения (карточка в Хоттабыче)

| Поле | Описание |
|------|---------|
| Статус | `В работе` / `Не используется` — скрытые из общего реестра |
| Дистрибутив | Название дистрибутива из ХД (если не задано — совпадает с именем приложения) |
| Обратный вызов | URL, вызываемый по завершении обновления (напр. для автозапуска тестов) |
| Шаблон развертывания БД | Имя шаблона из дистрибутива для сверки конфигурации перед конвертацией |
| Запрет полного обновления | Запрещает тип «Полное» |
| Только лёгкое обновление | Разрешает только лёгкое (применимо для приложений на «чужой» базе) |
| Останавливаться только на ошибках | Обновление не ждёт подтверждения между фазами |
| Разворачивать dpack по умолчанию | Включает флаг «Разворот dpack» в диалоге запуска |

### Шаблон пути загрузки (внешние приложения)

| Переменная | Значение |
|-----------|---------|
| `${distr_name}` | Имя дистрибутива |
| `${zip_name}` | Имя архива в дистрибутиве |
| `${app_name}` | Имя офлайн-приложения |
| `${version}` | Базовая версия |
| `${build_number}` | Версия, билд и дата в формате Хоттабыча |
| `${build}` | Номер билда |
