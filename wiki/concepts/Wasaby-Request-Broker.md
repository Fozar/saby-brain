# Wasaby Request Broker

**request-broker** — специализированный платформенный брокер асинхронных вызовов с гарантированной доставкой и отслеживанием статуса запросов.

> [!note] Для прикладных разработчиков
> Request-broker — инфраструктурный компонент. Прикладной код использует AsyncInvoke ([[Wasaby-BL-Calls]]), а платформа сама выбирает между RabbitMQ и request-broker на основе конфигурации.

## Зачем нужен (проблема RabbitMQ)

RabbitMQ не позволяет идентифицировать конкретный запрос и узнать его статус. При аварии брокера сервисы DWC и Scheduler не могли понять, какие задачи потерялись, и держали их "в обработке" до таймаута.

**Request-broker решает**: гарантированная доставка + статус каждого запроса по его ID.

## Архитектура

```
Клиент → [Управляющий сервис]
              ↓ (получить список узлов)
Клиент → [Исполнительный узел (request-broker-node)]
              ↓
         [Сервис-исполнитель]
              ↓ подтверждение
[backup-сервис] ← резервная копия запроса
```

Три компонента:
- **Управляющий сервис** — размещает очереди по исполнительным узлам; один "главный" координатор (Redis-lock)
- **Исполнительные узлы (request-broker-node)** — принимают и доставляют сообщения; БД: berkeley-db (key-value)
- **Backup-сервис (request-broker-backup)** — холодное хранилище; компенсирующая процедура повторяет потерявшиеся запросы

## Размещение очередей

Для каждого сервиса выделяется ≥2 исполнительных узла (для отказоустойчивости). Для DWC — больше.

Размещение по хэшу имени сервиса: `узел = hash(имя_сервиса) % N`.

## Жизненный цикл запроса

1. Клиент запрашивает список исполнительных узлов для сервиса X (кэш 5 мин)
2. Регистрирует резервную копию в backup-сервисе
3. Отправляет запрос на случайный исполнительный узел (HTTP, sync)
4. Исполнительный узел держит задачу в памяти (+ запись на диск с лагом)
5. Сервис-исполнитель забирает задачи (pull) — не более чем количество рабочих процессов × AsyncScale
6. Каждую минуту: сервис-исполнитель подтверждает список задач в обработке
7. По завершении: исполнитель уведомляет узел → узел уведомляет backup → backup удаляет резервную копию

## Гарантии

Backup-сервис каждые 5 минут сканирует незавершённые запросы:
- Запрос есть в брокере → обновить метку, пропустить
- Запрос отсутствует в брокере, есть в Redis (выполнен) → удалить из backup
- Запрос отсутствует, не в Redis → пометить "подозрительным"
- "Подозрительный" при следующей проверке → повторная публикация в брокер

## Перебалансировка

При изменении состава узлов: главный управляющий узел заморозит старые очереди и перекидывает трафик на новые. Клиенты получают актуальный список при следующем запросе.
