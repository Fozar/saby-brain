# ТЗ — Рефералка в Заявках в банк на РКО по API

Navigation: [[SabyBank-RKO-Referral]] | [[SabyBank-RKO-WorkPlan]] | [[price-formation/_index]]

> **Подписано**: Свешников А.В. — 02.04.26, Мусохранов А.В. (Ведущий инженер-программист) — 07.04.26

---

## 1. Цели проекта и критерии их достижения

Целью проекта является внедрение реферальной системы на Заявки, отправляемые в банк на РКО по API.

**Что даёт:**
- Просмотр статистики Тензором (и Банками) по Заявкам
- Возможность начислять и просматривать вознаграждения в сторону Тензора

**Критерии достижения:**
1. Реализовано SabyNet-приложение SabyBank, в котором:
   - Тензор создал реф. программу "Заявки на РКО", банки — участники
   - Тензор и Банки видят статистику Заявок в рамках реф. программы
   - Менеджер Тензора вручную начисляет вознаграждения (в т.ч. через ПМО)
2. Возможность отправки в конкретный Банк определяется участием Банка в реф. программе

---

## 2. Потенциальный круг пользователей

Функционал реализуется **только для сотрудников Тензора** — мониторинг статистики и проставление вознаграждений.

- **За 1 месяц и 6 месяцев после внедрения**: компания Тензор (1 аккаунт)
- **Менеджеры Банков**: пользоваться не будут — все заявки уходят в банк по API, менеджеры ведут учёт в своих банковских системах

---

## 3. Концептуальное описание решения

### Понятийная модель

| Понятие | Описание |
|---------|----------|
| **Тензор** | Владелец схемы, монетизирует отношение Клиента с Банком |
| **Клиент Saby** | Пользователь Saby, инициатор создания Исх. Банк. Заявок |
| **Банк** | Получатель Вх. Банк. Заявок, Партнёр Тензора по реф. сети |
| **Приложение SabyBank** | SabyNet-приложение с реф. системой, формализует партнёрство Тензора и Банков |
| **Банковская услуга (РКО)** | Задаётся регламентами "Заявка на открытие счёта" и "Заявка на РКО" + экземпляром реф. программы типа "Заявка на РКО" |
| **Реферальная программа** | Тип "Заявка на РКО" — задаёт отражение Банк. услуги в реф. системе; позволяет задавать Банк-Участника и тарификацию |
| **Корешок** | Запись `ВидЦеныДокумент` в акк. Тензора, фиксирует факт создания Вх. Заявки |
| **Вознаграждение** | Фиксирует величину вознаграждения от Банка в сторону Тензора |

> [!key-insight] Точки интеграции
> Реф. система взаимодействует с системой банковских заявок через **два API метода**. Каким образом Исх. Заявка попадает в акк. Банка — Реф. системе не важно.

### Архитектура подсистем

**Система [Банк. Заявки] / Аккаунт Клиента:**
- Выбор Банка и тип заявки
- Создание Исх. заявки

**Система [Банк. Заявки] / Аккаунт Банка:**
- Создание Вх. заявки → Обработка Вх. заявки → Обработка через API Банка

**Система [Реф. система - SabyBank] / Аккаунт Тензора:**
- API получения Банков-участников реф. программ
- Оформление участия в реф. программах и уровень вознагр.
- API Создания и Обновления Корешка
- Создание реф. программы для Типа Банк. Услуги
- Начисление и расчёт вознаграждения

**Система [SabyNet]:**
- ЛК Владельца / Партнёра

## 4. Интерфейс

### 4.1 Содержимое приложения SabyBank

Приложение содержит: **Офферы** (Программы), типовые страницы SabyNet — **Партнёры**, **Участники**, **Настройки**.

### 4.3 Карточка создания / редактирования реф. программы

Доступно редактирование: название, описание, изображение программы.

### 4.4 Детализация программы. Страница "Лиды"

При провале в программу на вкладке "Лиды":

- Список заявок
- Кнопка начисления вознаграждений вручную для конкретной заявки

### 4.5 Страница "Партнёры"

На вкладке "Партнёры":
- Фильтр по периоду со статистикой вознаграждений

### 4.6 Виджеты

На главной странице можно добавить виджеты **"Программы"** и **"Партнёры"**, аналогично виджетам в реф. программе.

## 6. Техническое решение

### 6.1 Архитектура данных и связь понятий

#### Модель данных (связь с таблицами лояльности)

- Название, описание, картинка, иконка → атрибуты `ВидЦены`
- ~~Базовая величина вознаграждения~~ — решили **не автоматизировать** расчёт вознаграждения
- Для каждого типа банк. услуги ("Открытие РКО", "Кредит" и т.д.) создаётся свой подтип `ВидЦены.Тип`

**Участник реф. программы** (Банк-партнёр) = запись в таблице `Карта`, поле `Лицо`:
- ~~Персональная величина вознаграждения~~ — решили **не автоматизировать**

**Корешок** = запись `ВидЦеныДокумент`, связана с Участником и Реф. программой:
- Для поиска корешка: новое **индексируемое поле** `ВидЦеныДокумент.РефЗаявка::UUID`
- Вознаграждение по Заявке → `ВидЦеныДокумент.Бонусы`

### 6.1.1 Реф. сеть SabyBank

**Создание новой реф. сети:**
- Новый служебный компонент `SabyBank` (только для Тензора, аналог "Биллинг Тензор")
- Новая реф. сеть `SabyBank` (на бою и тестовых) с включённым компонентом
- Все новые страницы привязываются к этому компоненту → доступ только у Тензора

**Страница "Офферы":**
- Создаётся новая страница (`navx`, `pagex`) для владельца
- Кнопка "+" вызывает `ReferralProgram.Create` с доп. параметром: тип "Заявки на РКО"
- Обработчик: `LoyaltyOnline/ReferralLeadNew/Owner/registry:openCreateDialog` с доп. параметром
- Добавляется в конфигурацию владельца

**Карточка создания / редактирования реф. программы:**
- Доработать `ReferralProgram.Read/Update` — добавить поле "тип реф. программы"
- Доработать `LoyaltyOnline\ReferralLeadNew\Owner\form:View` — скрывать лишние элементы при типе "Заявка на РКО": выбор сайта, описание вознаграждения, сумма вознаграждения за лид, тема отношений, форма обратной связи

**Страница "Заявки":**
- Доработать существующую страницу заявок в реф. системе сделок
- Из `ReferralProgram.Read` получать тип программы; если "Заявки на РКО" — скрыть кнопки создания заявки и копирования реф. ссылки
- Отображение заявок и обработка клика — по этапу "Создание заявок на основе участия в реф. программе"

**Страница "Партнёры":**
- Доработать существующую страницу Партнёры
- Из `ReferralProgram.Read` получать тип; если "Заявки на РКО" — скрыть лишние элементы и отключить возможность задавать индивидуальные вознаграждения

**Виджет "Мои программы":**
- Новый виджет для владельца и партнёра рефсети
- У владельца: все его программы; у партнёра: все программы, в которых он участвует
- Данные: `ReferralProgram.GetList`
- Элемент плитки: `LoyaltyOnline/ReferralLeadNew/tile:Container`
- По клику → стековая панель: `SabyPage/pageOpener:Opener`, параметр `owner-deals` / `participant-deals`
- В панель передаётся фильтр по периоду с главной страницы

### 6.1.2 Создание заявок на основе участия в реф. программе

**Создание заявки в зависимости от участия банков в рефпрограммах:**
- Перечень банков для РКО определяется участием Банка в реф. программе типа "Заявки на РКО" приложения SabyBank
- Мастер заявок получает перечень банков через метод "Получить банки-участники" (см. 6.3.2), вызываемый строго в акк. Тензора
- Если Банк не является участником — отправка в него невозможна

**Создание заявки в аккаунте банка:**
- Остаётся прежним (как в предыдущем проекте)
- Добавляется вызов асинхронного API-метода создания Корешка (см. 6.3.2)

**Изменение статуса заявки:**
- При изменении статуса (результат взаимодействия с API Банка) → вызов API-метода обновления Корешка (см. 6.3.2)
- Корешок — единственный источник данных для статистики в SabyBank

**Реестр "Заявки" на данных из корешков:**
- Новые методы BL (старые полностью изменятся по формату):
  - Списочный метод получения данных о заявках из корешков (фильтр: дата, статус, партнёр)
  - Метод получения данных об одной заявке из корешков
- Новые UI-компоненты:
  - Элемент списка на основе данных из корешка
  - Карточка просмотра: банк, ид заявки, дата создания, статус

**Назначение вознаграждений:**
- На странице "Заявки": действие над записью + кнопка ПМО
- Доработать `ReferralProgram.SetLeadPrice`: если тип "Заявки на РКО" — менять `ВидЦеныДокументБонусы`
- UI-компонент: `LoyaltyOnline/ReferralLeadNew/Owner/_deals/View/ChangePriceDialog`

### 6.1.3 Создание Корешков по существующим Заявкам

- Реф. система выходит позже проекта отправки заявок по API
- Требуется пакетная обработка уже существующих заявок
- Корешки создаются сервисно на основании данных о ранее отправленных заявках
- Используется метод "Создать корешок заявки" (вызов из ВНР)
- **Правило**: повторное создание по тому же `ИдентификаторЗаявки` → ошибка дублирования

### 6.1.4 Технический долг и доработки

**Выделить работу с реф. системой в отдельный СБИС-модуль:**
- Перенести backend реф. системы (`orx`, реализация) в новый модуль `PriceFormation.Referral`
- Ответственный: **[[Тимошенко А.А.|Тимошенко А.]]**
- `PriceFormation.Online` ставится в зависимость от нового модуля
- Аналогичная работа на уровне unit-тестов

**В реф. программах сделок перейти на корешки:**
- Отказаться от статистики по лидам из сервиса маркетинга
- Проверять, что `ВидЦеныДокумент` — не корешок, по наличию поля `ВидЦеныДокументТипСвязи` (NULL только не в корешках)

При создании заявки вызывать метод создания корешка в сценариях:
1. Из личного кабинета: `ReferralProgram.CreateLeadForPartner` (под владельцем) и `ReferralProgram.CreateLeadByPartner` (под партнёром)
2. Форма обратной связи — событие создания сделки (если источник проставляется асинхронно → нужна доработка API создания сделки)
3. Смена источника у сделки на реферальный → нужно событие смены источника

При изменении статуса заявки → вызов метода обновления корешка.

Добавить запись в историю реф. программы: при создании заявки и при смене источника у сделки на реф.

После ВНР во всех местах получения вознаграждений перейти на корешки:
- Реестр "Офферы"
- Реестр "Заявки"
- Реестр "Партнёры"
- Список периодов с вознаграждениями (на всех страницах детализации оффера)
- Виджеты реф. программ

## 6.2 Схема взаимодействия сервисов

Ключевой принцип: **две точки интеграции** между Реф. системой и банковскими заявками — методы создания и обновления Корешка.

```
Сервис Банковских интеграций
└── Очередь заявок / Обращение к Банку через API
        ↓
Сервис Online
├── Документ ←→ Документ ←→ Корешок (ВидЦеныДокумент) ←→ Реф. программа (ВидЦены:Тип)
│   ├── Исх.Банк.Заявка (ТипДокумента)        Участник (Карта) → ВидКарты
│   │   └── Регламент "Заявка на открытие счёта"
│   └── Вх.Банк.Заявка (ТипДокумента)
│       └── Регламент "Заявка на РКО"
│               Партнёр=Банк (Лицо)
│               Агентский договор (НашаОрганизация)
└── Бизнес-группа ← Сервис Соцсети

[Клиент Saby / акк. online] → [Партнёр=Банк / акк. online] ← [Владелец / акк. Тензора]
```

## 6.3 Описание API

### 6.3.1 Используемые API

Использование API других подсистем и сервисов не планируется.

### 6.3.2 Новые API

#### Метод "Получить банки-участники реф. программ SabyBank"

**Назначение**: для мастера заявок на РКО — получить список банков, в которые допустимо отправлять заявку.

**Контекст вызова**: строго в аккаунте Тензора.

**Результат**:
```json
{
  "ID_РефПрограммы": {
    "Тип": "<Заявки на РКО|...>",
    "Участники": [
      {"ИмяБанка": "...", "АккБанка": "..."}
    ]
  }
}
```

**Примечания:**
- Единственный источник истины для UI мастера заявок о доступных банках
- Если банк отсутствует в участниках программы типа "Заявки на РКО" — отправка в него недоступна

---

#### Метод "Создать корешок заявки"

**Назначение**: зафиксировать в реф. системе корешок (служебную запись) при создании Вх. банковской заявки.

**Контекст вызова**: строго в аккаунте Тензора. Вызывается **асинхронно** в момент создания Вх. банковской заявки.

**Входные параметры:**

| Параметр | Описание |
|----------|----------|
| `ID_РефПрограммы` | Идентификатор реф. программы |
| `АккБанка` | Аккаунт банка-партнёра |
| `ИдентификаторЗаявки` | UUID документа заявки |
| `МетаДанныеЗаявки` | Дата создания, текущий статус обработки, прочие служебные данные |

**Алгоритм:**
1. По `АккБанка` через API соцсети ищем партнёра (`@Лицо`) в бизнес-группе SabyBank
2. По `@Лицо` → находим `Карту` участника указанной реф. программы (реф. код)
3. Создаём Корешок (`ВидЦеныДокумент`) с полями: ссылка на реф. программу, ссылка на участника (Карту), ИдентификаторЗаявки, метаданные

**Ограничения и правила:**
- Повторный вызов с тем же `ИдентификаторЗаявки` → исключение (ошибка дублирования)
- Смена банка по одной заявке (маловероятна): удаление старого корешка + создание нового

---

#### Метод "Обновить корешок заявки"

**Назначение**: обновить метаданные корешка при изменении статуса обработки Вх. банковской заявки.

**Контекст вызова**: строго в аккаунте Тензора. Вызывается при обновлении статуса Вх. заявки (результат взаимодействия с API Банка).

**Входные параметры:**

| Параметр | Описание |
|----------|----------|
| `ИдентификаторЗаявки` | UUID документа заявки |
| `МетаДанныеЗаявки` | Текущий статус обработки, прочие изменяемые атрибуты |

**Алгоритм:**
1. По `ИдентификаторЗаявки` находим `ВидЦеныДокумент`
2. Обновляем метаданные, сохраняем

**Примечания по производительности:**
- Ожидаемые объёмы: тысячи, в пределе — десятки тысяч заявок
- Требуется проработка индексации/уникальности поиска корешка по `ИдентификаторЗаявки`

## 6.4 Схема хранения данных

### Таблица ВидЦеныДокумент (корешок)

| Поле | Смысл |
|------|-------|
| `ВидЦены` | Идентификатор реф. программы |
| `Карта` | Идентификатор карты участника реф. программы |
| `ДатаВремя` | Дата и время создания Заявки |
| `OperationID` | UUID документа Заявки в аккаунте Партнёра |
| `ТипСвязи` | 10 — в работе, 11 — успешно завершена, 12 — не успешно завершена |
| `EffectiveDate` | Дата и время завершения (успешно или нет) Заявки |
| `Бонусы` | Величина начисленных вознаграждений (при успехе), иначе `NULL` |
| `Атрибуты` | Мета-данные: акк. клиента Исх. Заявки, ИД Исх. Заявки, прочее для идентификации. **По этим данным поиск не производится** |

### Жизненный цикл корешка

| Поле | Заявка создана | Заявка на обработке | Заявка завершена | Заявка с вознаграждением |
|------|---------------|---------------------|------------------|--------------------------|
| `ВидЦены` | ID реф. программы | → | → | → |
| `Карта` | ID карты участника | → | → | → |
| `ДатаВремя` | Дата создания заявки | → | → | → |
| `OperationID` | UUID заявки в акк. партнёра | → | → | → |
| `ТипСвязи` | 10 | 10 | 11 / 12 | 11 |
| `EffectiveDate` | NULL | NULL | Дата завершения | Дата завершения |
| `Бонусы` | NULL | NULL | NULL | Кол-во вознаграждения |
| `Атрибуты` | Метаданные | → | → | → |

### Новые индексы

1. **`ВидЦеныДокумент (ВидЦены + ДатаВремя + ТипСвязи)`** — для фильтрации по статусу заявки (существующий индекс `(ВидЦены + ДатаВремя)` не покрывает `ТипСвязи`)
2. **Под вопросом**: новый индекс для выборки суммы вознаграждений по дате создания — текущий индекс `(ВидЦены + EffectiveDate + Бонусы <> 0)` использует дату завершения, а нужна дата создания (`ДатаВремя`)

### 6.4.1 Описание основных выборок

#### Узнать факт участия Партнёра в реф. программе
1. Через API соцсети в бизнес-группе SabyBank ищем `@Лицо` контрагента партнёра
2. По `@Лицо` → связанные `Карты` → оставляем ту, что связана с `ВидЦены` реф. программы (через `ВидЦеныВидКарты`, `ВидКарты`)

**Индекс**: существующий `Карта(Лицо)`  
*Аналог сценария "Получение факта участия покупателя во внешней системе лояльности (UDS, PremiumBonus)"*

---

#### Получить список заявок по реф. программе (все партнёры)
1. Выбираем `ВидЦеныДокумент`, связанные с `ВидЦены` (реф. программой), сортировка по `ДатаВремя`

**Индекс**: существующий `(ВидЦены + ДатаВремя)`  
*Аналог "Получение списка продаж по скидке/бонусной программе"*

Для фильтрации по статусу (`В работе | Успешна | Не Успешна`) — используем поле `ВидЦеныДокументТипСвязи`. Индекса по нему нет → **новый индекс** `(ВидЦены + ДатаВремя + ТипСвязи)`.

---

#### Получить список заявок партнёра по реф. программе
1. Через API соцсети по UUID бизнес-группы и ИД партнёра → `@Лицо` контрагента
2. По `@Лицо` → `Карта` участника конкретной реф. программы
3. `ВидЦеныДокумент`, связанные с Картой, сортировка по `ДатаВремя`

**Индекс**: существующий `(Карта + ДатаВремя)`  
*Аналог "Получение списка продаж по карте лояльности клиента"*

---

#### Обновить корешок заявки
- По UUID заявки (`ИдентификаторЗаявки`) ищем (и обновляем) запись по полю `ВидЦеныДокумент.OperationID`

**Индекс**: существующий по `OperationId`

---

#### Получить сумму вознаграждений за период по реф. программе
1. `ВидЦеныДокумент` в диапазоне дат (поле `ДатаВремя`), `Бонусы > 0`
2. Подсчитываем сумму по `Бонусы`

**Текущий индекс** `(ВидЦены + EffectiveDate + Бонусы <> 0)` — хранит дату завершения, а выборка нужна по дате **создания** → **возможен новый индекс**

---

#### Получить сумму вознаграждений за период по партнёру
1. Через API соцсети → `@Лицо` → `Карта` участника
2. `ВидЦеныДокумент (Карта + ДатаВремя)`, подсчитываем сумму по `Бонусы`

**Индекс**: существующий `(Карта + ДатаВремя)`  
*Аналог "Получение бонусного баланса/бонусных начислений по карте лояльности"*

## 6.5 Пользовательский интерфейс

### 6.5.1 Декомпозиция пользовательского интерфейса

В основном используются и дорабатываются существующие компоненты реф. системы сделок.

| Страница | Компонент |
|----------|-----------|
| Реестр "Офферы" | `LoyaltyOnline/ReferralLeadNew/Owner/registry:View` |
| Карточка реф. программы | `LoyaltyOnline\ReferralLeadNew\Owner\form:View` (скрываем лишние элементы по типу) |
| Страница "Заявки" | `LoyaltyOnline\ReferralLeadNew\deals:View` (скрываем кнопки создания и копирования ссылки) |
| Страница "Партнёры" | `LoyaltyOnline/ReferralLeadNew/master:View` + `LoyaltyOnline/ReferralLeadNew/statsRegistry:View` (скрываем лишние элементы) |

### 6.5.2 Описание источников данных в интерфейсе

#### Страница "Офферы"
*Выполняется на сервисе представления. Методы параллельно:*
- `PriceFormation.GetPersonBGRoleInAgents` — роль текущей персоны в бизнес-группе (редактирование только администраторам/модераторам)
- `ReferralProgram.GetList/4` — списочный метод для реестра офферов

#### Карточка создания / редактирования реф. программы
*Выполняется на клиенте. Методы параллельно:*
- `ReferralProgram.Read/3` — чтение информации реф. программы
- `PriceFormation.GetPersonBGRoleInAgents` — роль персоны в бизнес-группе

#### Страница "Заявки"
*Выполняется на сервисе представления. Методы параллельно:*
- `ReferralProgram.Read/3` — название, описание, тип
- `ReferralProgram.GetLeadPeriodList/4` — данные для фильтра по периоду

*1-я цепочка (последовательно):*
- `PriceFormation.GetPersonBGRoleInAgents`
- `ReferralProgram.GetLeadList/4` — реестр заявок

#### Страница "Партнёры"
*Выполняется на сервисе представления. Методы параллельно:*
- `ReferralProgram.Read/3` — название, описание, тип
- `ReferralProgram.GetLeadPeriodList/4` — фильтр по периоду

*1-я цепочка (последовательно):*
- `PriceFormation.GetPersonBGRoleInAgents`
- `ReferralProgram.GetPartnerList/4` — реестр партнёров
