# Патчи

Патч — zip-архив, создаваемый разработчиком для модификации версии дистрибутива. Применяется когда нужно срочно заменить файлы, а сборка нового дистрибутива нецелесообразна (невозможна, требует слишком много времени).

Применение патча: Хоттабыч заменяет указанные файлы в существующем дистрибутиве → создаёт новый дистрибутив с добавлением номера патча к версии → обновляет приложение.

> Патч применяется на **все сервисы** внутри приложения, использующие изменённый модуль — выбрать конкретный сервис нельзя.

## Что можно патчить

| Тип | Файлы |
|-----|-------|
| Бизнес-логика (без компиляции) | `.orx`, `.py` |
| Интерфейс | `.js`, `.ts`, `.tsx`, `.tmpl`, `.wml`, `.css`, `.less`, картинки (`.png`, `.jpg` и т.д.), текстовые файлы |
| Описание модулей | `.smod` |

## Что нельзя патчить

- Бинарные файлы, требующие бинарной совместимости с другими файлами дистрибутива
- Скомпилированные библиотеки: `.so`, `.dll`, `.dylib`
- Файлы описания БД: `.dicx`, `.cnv`, `.trg`, `.dpack`

> Исключение: если бинарные файлы никем другим не используются — патч возможен, создаётся по инструкции для патча на интерфейсный модуль.

## Тип 1: Патч на модуль интерфейса (UI)

Архив **не создаётся вручную** — Jenkins собирает его автоматически по MR.

### Шаг 1: Получить ревизию

Ревизия = хэш коммита, соответствующий состоянию репозитория на момент сборки дистрибутива. Находится в файле `metainfo` (JSON-файл в дистрибутиве).

**Два способа получить `metainfo`:**
1. Через Jenkins: открыть сборку нужного дистрибутива (GET-сборка) → Артефакты сборки → `metainfo`
2. Через Хоттабыч: реестр «Дистрибутивы» → нужный дистрибутив → клик на версию и сборку → скачать `metainfo`

**Структура `metainfo`:**
```json
{
  "application": {
    "repos": {
      "build_scripts/git_tools": {
        "branch": "master",
        "rev": "bac03c28ee3329cee2acf419da875aec7e79b3d0",
        "url": "git@git.sbis.ru:sbis-ci/devops/git-tools.git",
        "url_http": "https://git.sbis.ru/sbis-ci/devops/git-tools.git"
      }
    }
  }
}
```

Поиск ревизии: Ctrl+F по адресу репозитория (SSH → ключ `url`, HTTP → ключ `url_http`). Ревизия — в ключе `rev` строкой выше.

### Шаг 2: Переключить репозиторий

```bash
git checkout bac03c28ee3329cee2acf419da875aec7e79b3d0
# Создать ветку по правилам tensor-flow:
git checkout -b rc-24.NNNN/bugfix/ivanov/
# Внести правки в исходный код, отправить изменения
```

> Подтягивать изменения из релизной ветки (`git pull`) **не нужно**.

### Шаг 3: Создать Merge Request на GitLab

MR создаётся на **`git.sbis.ru`**, **НЕ на `online.sbis.ru`**. Из созданной ветки в rc-ветку, по которой собран дистрибутив. Если есть конфликты — выбрать в пользу новых изменений.

### Шаг 4: Создать ВНР

На `online.sbis.ru` оформить задачу «Выполнить на рабочем» (ВНР). Тип работы: **«Патч на интерфейс»**. Приложить ссылку на MR из GitLab (не архив).

> После запуска ВНР с патчем статики **нельзя вносить изменения в MR**. Для добавления изменений — откатить ВНР и запустить заново.

### Шаг 5: Добросить в репозиторий

1. Дождаться выполнения патча (фаза «Выполнение» в ВНР)
2. Администраторы сборки создадут специальную веху для патчей и включат задачу в неё
3. Создать MR на `online.sbis.ru` из этой задачи по указанной вехе
4. MR должен содержать изменения для **текущей и всех последующих версий**

## Тип 2: Патч на модуль бизнес-логики (BL)

Архив создаётся **вручную**.

### Шаг 1: Подготовить файлы

**Способ 1: С сетевого диска**
- Windows: `file:\\sbis-dev.corp.tensor.ru\PROGRAMMERS\СБиС_3\Тестирование\`
- Linux: `smb://sbis-dev.corp.tensor.ru/PROGRAMMERS/СБиС_3/Тестирование/`

Скачать дистрибутив → распаковать → внести изменения → удалить неизменённые файлы (оставить структуру папок).

**Способ 2: Из git-репозитория (рекомендован)**
1. Склонировать репозиторий, переключиться на релизную ветку
2. Внести изменения в нужные файлы
3. Создать папку `Модули бизнес-логики` и скопировать изменённые файлы (с сохранением структуры)
4. Удалить неизменённые файлы

### Структура архива BL-патча

```
p_timoshenkoaa_12345678.zip
└── Модули бизнес-логики
    └── Accounting
        └── main.py
```

```
p_фиопользователя_номервнр.zip
└── Модули бизнес-логики
    └── ReleaseManager
        ├── doc
        │   ├── doc_api.py
        │   └── doc_read.py
        └── plan
            ├── status.py
            └── update_work_status.py
```

**Имя архива:** `p_{фиопользователя}_{номервнр}.zip`

### Требования к архиву

- Расширение: `.zip`
- Кодировка имён файлов: **CP866** (по умолчанию в дистрибутивах на сетевом диске)
- Инструмент: **7-Zip** (стандартные средства Windows могут зависнуть)
- Linux: правый клик → «Сжать»

### Шаг 2: Создать ВНР

На `online.sbis.ru` оформить ВНР. Тип работы: **«Патч на бизнес-логику»**. Прикрепить переименованный архив.

### Шаг 3: Добросить в репозиторий

Аналогично UI-патчу: дождаться выполнения → веха от администраторов → MR на `online.sbis.ru` с изменениями для текущей и всех последующих версий.

## Техническая реализация (внутри Хоттабыча)

При применении патча:
1. Создаётся копия дистрибутива в Хранилище дистрибутивов (ссылки, реального копирования файлов нет — быстро)
2. Хоттабыч скачивает оригинальный дистрибутив и патч на `test-fs`
3. Распаковывает оба архива
4. По пути файлов в патче определяет имя модуля и тип (БЛ или интерфейс)
5. Скачивает только изменённые модули (по модульной структуре дистрибутива)
6. С помощью open-source `zip`: удаляет старые файлы из архива модуля → добавляет новые
7. Загружает изменённые архивы в копию дистрибутива
8. Патчит `sdistr` (конфиг дистрибутива в корневых архивах) — добавляет номер патча к версии
9. Применяет обновление с новым дистрибутивом

**Для интерфейсных патчей:** после замены модулей вызывается jinnee для перестройки корневых файлов в `resources/` (`contents.json`, справочники и их минифицированные версии).

**Нельзя применить патчем:** закэшированные браузером файлы (версия файлов не меняется в браузерном кэше).
