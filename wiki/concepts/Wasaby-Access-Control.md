# Wasaby — Система прав доступа

Три уровня: **Участок системы** → **Роль** → **Пользователь**.

## Участок системы (Subject Area, .uax)

Набор объектов и методов БЛ, относящихся к одному use-case. Описывается в `.uax` файле через Genie.

### Уровни доступа к методу

| Уровень | access | Описание |
|---------|--------|----------|
| Denied | 0 | Запрещён на участке (может быть включён в другом) |
| Access | 2 | Readonly-методы: при уровне "Просмотр"; остальные: при "Просмотр и изменение" |
| Read and Modify | 4 | Требует "Просмотр и изменение" даже если флаг Readonly |
| Full Control | 8 | Требует полного доступа |

### Уровни доступа пользователя к участку

- **Запрещён** — никакие методы недоступны
- **Только для чтения** — только Readonly методы
- **Просмотр и изменение** — все методы участка
- **Полный с настройками** — все методы + настройки реестра

### Служебные участки

Скрыты из настроек пользователя. Используются для:
1. Участков, доступных всем (Главная страница) → роль "Пользователь системы"
2. Участков, права которых приходят от родительских участков

### Наследование участков

Визуально — граф. Если X1 включает X2, все разрешённые объекты X2 доступны в X1.

### Расчёт прав (политики)

По умолчанию — **разрешающая политика** (объединение ограничений).
- Исключение: если хотя бы на одном участке ограничений нет → "Без ограничений".

Запрещающая политика: пересечение (максимальные ограничения).

### Метаданные .uax

```xml
<action identifier_action="Банк" name="Банк" is_service="0" responsible="..." last_changed="...">
  <comment>{"ForRead":"...", "ForModify":"...", "ForFullControl":"..."}</comment>
  <base name="РодительскийУчасток"/>
  <privilegy access="2" object="Объект.Метод"/>
</action>
```

## Роли

Роль — группа пользователей с одинаковым набором участков системы и областей видимости.

### Типы ролей

| Тип | Описание |
|-----|----------|
| Служебная | Описывается разработчиком (`Services="true"`), скрыта из интерфейса |
| Системная (инсталляционная) | Из метаданных сервиса (`*.rlx`), загружается при обновлении |
| Пользовательская | Создаётся администратором на "боевом" приложении |

**Роль "Пользователь системы"** — базовая служебная роль, назначается всем без другой роли.

### Виды ролей

- **Аспектная (роль-аспект)** — один use-case системы
- **Должностная (роль-актор)** — несколько use-case, часто = должность сотрудника

### Наследование ролей

- Граф зависимостей (не дерево)
- Все роли неявно наследуют "Пользователь системы"
- Объединение: максимальный доступ (если в одном участке метод запрещён, в другом разрешён → разрешён)

### Создание роли

1. Согласовать создание
2. Создать в `*.rlx` через Genie
3. Добавить перевод в словарь переводов (обязательно для неслужебных ролей)
4. Разместить в папке (одноимённые папки из разных модулей объединяются)

### Метаданные роли

| Поле | Значение |
|------|----------|
| Name | Бухгалтер |
| Identifier | UUID |
| Message for user | Описание use case для клиента |
| Service role | true/false (служебная) |
| Cloud admin | true/false (администраторы облака) |
| Clients of system | true/false (обычные клиенты) |
| Demo clients | true/false |
| Guest | true/false |

### Реестр ролей

[cloud.sbis.ru/page/roles](https://cloud.sbis.ru/page/roles) — список ролей и пользователей.

## Ограничения (Access Area)

Дополнительная настройка к участку системы: разрешить/запретить действия над **конкретным множеством объектов**.

### Типы ограничений

- **Область видимости** — фильтрация данных по условию (по компании, по подразделению)
- **Работа над записями** — разрешить/запретить конкретные операции

### Настройка ограничений

1. В `*.orx` метода БЛ: атрибут `Access areas` — декларирует, что ограничение влияет на метод
2. В `*.rlx` роли на участке: Subject area permissions (запрещено / только чтение / запись / администрирование)

### Наследование ограничений

При включении дочернего участка в родительский — все ограничения дочернего доступны на родительском.

`Disabled Scopes` (`Scope settings`) — запрещает показ настройки ограничения на конкретном участке (не влияет на итоговый расчёт!).

> [!warning] Ограничение Disabled Scopes
> Если настроить Disabled Scopes на участке с методом — ограничение зафиксируется как "разрешено" для этого участка, независимо от остальных.

### Итоговое значение ограничений

Суммируется по всем участкам → пользователю даются **максимальные права** из всех назначенных.

### Использование в коде

```python
# Способ 1: через sbis.Session.Rights()
rights_state = sbis.Session.Rights()
if rights_state is not None and 'myOfficeDocs' in rights_state:
    # работа с ограничением

# Способ 2: через sbis.CheckRights.AccessAreaRestrictions
rights_state = sbis.CheckRights.AccessAreaRestrictions("myOfficeDocs", "Проверки")
rights_state = (rights_state.Get('Restrictions') or sbis.Record()).Get('Write', False)
```

### Алгоритм вызова

Метод ограничения вызывается если:
1. На метод назначена область видимости
2. На участок системы, включённый в роль, назначена область видимости
3. Область видимости назначена индивидуально в настройках пользователя
