# Акции — Архитектура и алгоритмы расчёта скидок

Navigation: [[index]] | [[Акции-Subsystem-Overview]] | [[Акции-Info-Model]]

## Порядок расчёта на продаже

```
1. Подарки (проверка условий)
2. Скидки (конкурс + суммируемые)
   — на подаренные позиции скидка не распространяется
3. Списание бонусов
4. Наценка (максимальная из доступных)
5. Скидка округления
```

## Конкурс скидок

**Конкурсные скидки**: Процентная, Суммовая на весь чек, Суммовая на позицию, Спеццена.

Делятся на **суммируемые** и **несуммируемые**. Каждая скидка имеет приоритет: высокий / средний / низкий.

### Алгоритм конкурса

1. Рассматриваются **несуммируемые** скидки, начиная с сегмента "высокий"
2. В каждом сегменте несуммируемые конкурируют → отбирается **наилучшая**
3. Конкурс на позицию → самая выгодная
4. Если в сегменте хоть одна скидка сработала — более низкие сегменты не рассматриваются
5. После завершения конкурса несуммируемых — проверяем **суммируемые** (та же схема)

### Позиционные vs. чековые

**Скидки на весь чек**: Суммовая на весь чек, Процентная на весь чек (считается как позиционная на каждую позицию).

**Позиционные**: делятся на **комплектные** и **некомплектные**. Комплектная всегда побеждает некомплектную на одной позиции.

Механизм расчёта наилучшей «Суммовой на чек»: сумма раскидывается пропорционально стоимости позиций (кроме изменённых вручную). Остаток после распределения вычитается из самой дорогой позиции.

## Суммируемые скидки

> [!key-insight]
> Суммируемая скидка **не участвует в конкурсе** — если подходит, применяется всегда. Все суммируемые одного приоритета применяются вместе, а не конкурируют.

- Суммируемых скидок может быть несколько — применяются в алфавитном порядке
- Стартуют с самого высокого приоритета заново после конкурса несуммируемых
- Суммирование в рамках одного приоритета
- Если сработал хоть один сегмент — более низкие не рассматриваются
- Суммируемая может быть настроена для срабатывания **только совместно с конкретными скидками** (СрабатываетСоСкидками)
- Каждая последующая уменьшает позицию с учётом предыдущих

## Накопительные скидки

Скидка, связанная с дисконтной картой + «накопительные пороги» + период накопления.

- Накопление только в рамках дисконтной карты, только по завершённым продажам
- В offline розницы — только при наличии связи

**Условия для участия в конкурсе** (все должны выполниться):
- Скидка Используется
- Если задано ограничение по типу продажи — продажа должна быть розничной/СУ
- Если задан период — дата/время продажи должна удовлетворять
- Если задано ограничение по покупателю/региону (только СУ) — проверяется
- Если задано ограничение по ролям — проверяется
- Если задано ограничение по ТП — проверяется
- Если задано ограничение по типу оплаты — проверяется
- Карта не заблокирована (для автоматической с привязкой к ДК)

## Комплектные скидки

Условие срабатывания = присутствие «триггерных» товаров в заданном количестве/сумме.

**Правила**:
- Проверяются в **алфавитном порядке** — применяется **первая** сработавшая (не лучшая)
- На одну триггерную номенклатуру — только одна комплектная скидка
- На документ может быть несколько комплектных скидок
- Не поддерживает «запрещённые товары»
- Нельзя задать порог «кол-во одного товара»

**Настройки**: размер комплекта, способ сравнения (`>=` / `=`), тип комплекта (все/любой из списка).

**Пример**: «При покупке 3 ручек — скидка 20% на тетради». 5 ручек + 5 тетрадей → скидка на 3 тетради. 6 ручек → 2 комплекта → скидка на 5 (все имеющиеся) тетради.

## Спеццена

Вариант суммовой скидки на позицию: величина = разница между ценой по прайсу и ценой из настроек.

- Срабатывает только если цена в настройках **ниже** цены по прайсу
- Может быть настроена как комплектная скидка

## Скидка округления

Автоматическая, неконкурсная, техническая (создаётся системой при включении настройки).

**Алгоритм**:
1. Определяется величина округления (до 0.1р, 0.5р, 1р и т.д.) с учётом порогов. Сумма — с учётом всех применённых скидок и бонусов
2. Рассчитывается сумма, которую нужно «отбросить»
3. Если сумма < 1р — «отбрасывание» с позиций с копейками (от самой дорогой), потом без копеек
4. Если сумма >= 1р — «отбрасывание» с позиции с самой большой суммой

## OOP-архитектура системы расчёта

```
Discount (abstract)  ← TriggerEntity
├── PositionalDiscount
│   ├── PercentDiscount
│   ├── PositionalSumDiscount
│   └── SpecialPrice
├── SaleSumDiscount
├── SetDiscount          (комплектная)
└── GiftDiscount         (подарочная)

Вспомогательные компоненты:
- DiscountCondition, DiscountTarget, DiscountValue
- SetSettingsParams, ThresholdCondition, NomListHolder
- sale::SalePos, CalcResult
```

Паттерны: **Стратегия** + **Композиция** — добавление новых типов скидок без изменения существующего кода.
