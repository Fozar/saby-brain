# Report Prefetch Service

Платформенный механизм кэширования отчётов (`report-prefetch-service`) — сервис Wasaby для предварительного получения и хранения больших отчётов постранично. Позволяет клиенту получать страницы отчёта мгновенно без повторных дорогостоящих запросов к источнику данных.

---

## Два основных сценария

1. **Создание кешированного отчета** — полный цикл от запроса до сохранения всех страниц в БД сервиса.
2. **Получение данных кешированного отчета** — загрузка уже закешированной страницы из БД сервиса без обращения к источнику.

---

## Порядок выполнения Prefetch.List

1. **Проверка прав** для метода, который хочет закешировать пользователь.
2. **Создание сессии** (`PrefetchSrv.CreateSession`) — если пользователь не передал идентификатор сессии; иначе отчёт привязывается к переданной сессии.
3. **Вызов метода с большим лимитом навигации** — единый запрос к источнику данных.
4. **Разбиение результатов на страницы** и сохранение в БД через `PrefetchSrv.SaveReportBlocks` (одна строка в БД = одна страница).
5. **Возврат клиенту** запрошенной страницы + признак успешности кэширования + дата актуальности.

---

## Конвейер обработки отчёта

Набор операций над отчётом определяется **фильтром** запроса `Prefetch.List`. По фильтру строится **конвейер обработки** — цепочка узлов. Порядок определяет **строитель конвейера**. Каждый узел:
- Читает нужные поля из фильтра.
- Удаляет «свои» поля из фильтра.
- Вызывает строитель конвейера для следующего узла.

### Узлы обработки

| Узел | Назначение |
|---|---|
| **SimpleHandler** | Запрашивает страницу из prefetch-сервиса; если нет — запрашивает источник с большим лимитом, разбивает на страницы, сохраняет. Строит индексы если в фильтре есть `PrefetchIndices`. |
| **BigLimitHandler** | Склеивает множество страниц малого лимита в одну страницу большого лимита. |
| **FlatHierarchyPageExtractor** | Извлекает нужный раздел и страницу из иерархического отчёта с разворотом. |
| **HierarchyLevelsBuilder** | Разбивает отчёт с разворотом на отдельные уровни иерархии; сохраняет каждый уровень в prefetch. Строит индексы при наличии `PrefetchIndices`. |
| **Search** | Поиск в отчёте. Если для полей поиска есть индексы — поиск по индексам. |
| **Filter** | Фильтрация отчёта. Если для поля фильтрации есть индекс — фильтрация по индексу. |
| **Sort** | Сортировка отчёта. Если для поля сортировки есть индекс — сортировка по индексу. |
| **AsyncSave** | Синхронный запрос малого лимита + асинхронный запрос большого лимита с последующей сериализацией и сохранением в prefetch. |
| **AsyncSaveAll** | Если сессия ещё не создана — асинхронное сохранение всего отчёта в кэш. |
| **Selection** | Отбор выделенных записей. |
| **EntryPathBuilder** | Строит `ENTRY_PATH` по переданному списку entries. |

> [!note] Индексы и поиск/сортировка
> Поиск, фильтрация и сортировка работают по индексам только при выполнении двух условий:
> - В фильтре задан `PrefetchIndices`.
> - Поле, по которому производится операция, входит в индекс.

---

## Индексы

Индекс — отсортированный «срез» отчёта из 1 или нескольких столбцов, в котором каждая запись содержит **адрес записи полного отчёта** (номер/хэш страницы + номер записи на странице).

- Формируется по 1 или нескольким столбцам отчёта.
- При достижении порогового размера разбивается на **главы**: страница 0 = оглавление, последующие = страницы индекса.
- Предусмотрено расширение: например, тип адреса `RecAddrWithSecSize` дополнительно хранит размер поддерева каждой записи (для иерархических отчётов).

## Публичное API — Prefetch.List (параметры фильтра)

> [!note] Отличие от кэширования методов
> [[Wasaby-BL-Advanced]] кэшира только **одну страницу** (Redis). При навигации к следующей странице требуется полный вызов. Prefetch решает именно эту проблему.

### Быстрый старт

```python
@staticmethod
def Example.List(ДопПоля, Фильтр, Сортировка, Навигация):
    Фильтр.AddString('PrefetchMethod', 'Example.ListWithoutCaching')
    Фильтр.AddInt32('PrefetchPages', 100)
    return Prefetch.List(ДопПоля, Фильтр, Сортировка, Навигация)
```

На UI: `Layout/reportPopup:Browser` — компонент управляет микросессиями сам (опция `prefetchParams`).

### Термины

| Термин | Определение |
|--------|-------------|
| Первичка | БЛ-метод — источник данных отчёта |
| Малый лимит | Лимит навигации одной страницы |
| Большой лимит | `малый лимит × PrefetchPages` — объём единого запроса к первичке |
| Микросессия | UUID-идентификатор набора кэшированных данных; определяет TTL |

### Входные параметры фильтра

| Поле | Тип | Описание |
|------|-----|----------|
| `PrefetchMethod` | String | Метод-первичка |
| `PrefetchEndpoint` | String | Системное имя сервиса первички |
| `PrefetchPages` | Int32 | Кол-во страниц за запрос; дефолт: 50 / 500 для иерархии |
| `PrefetchCursorFields` | ArrayString | Поля курсора (обязательно при курсорной навигации) |
| `PrefetchCursorFormat` | Record | Точный формат курсора; исправляет несовпадение типов |
| `PrefetchSessionId` | UUID | ID микросессии; если не задан — создаётся новая |
| `PrefetchSessionLiveTime` | TimeInterval | TTL микросессии |
| `PrefetchKeepSessionId` | Boolean | При истечении сессии пересоздать с тем же ID |
| `PrefetchOnlyCache` | Boolean | Только из кэша |
| `PrefetchThrowErrors` | Boolean | Бросать ошибки вместо fallback |
| `PrefetchAsyncSave` | Boolean | Синхронный малый лимит + асинхронное сохранение большого |
| `PrefetchAsyncSaveAll` | Boolean | Асинхронное сохранение всего отчёта при первом запросе |
| `PrefetchForceAsyncSave` | Boolean | Всё сохранение асинхронно |
| `PrefetchIgnoredInHash*` | Any | Поля, не влияющие на хэш фильтра (имя начинается с этой строки) |

> [!warning] Иерархия: обязательный фильтр
> Все вызовы `Prefetch.List` с иерархическими отчётами **должны** содержать `PrefetchPreSort`.

### Параметры иерархии

| Поле | Тип | Описание |
|------|-----|----------|
| `PrefetchPreSort` | Boolean | Предварительная сортировка разворота для строгой иерархии |
| `PrefetchHierarchyColumn` | String | Поле родителя; дефолт `"Раздел"` |
| `PrefetchIdColumn` | String | Поле ID записи; дефолт `"DBId"` |
| `PrefetchLevels` | Int32 | Кол-во уровней; `-1` = все |
| `PrefetchPathFields` | ArrayString | Поля, включаемые в path уровней |
| `PrefetchSpredHierarchy` | Boolean | Режим разворота из кэша всего отчёта |
| `PrefetchMultinavigation` | Boolean | Имитировать мультинавигацию обычными запросами |

### Параметры сортировки

| Поле | Тип | Описание |
|------|-----|----------|
| `PrefetchDefaultSortRows` | Boolean | Стандартная сортировка на Prefetch |
| `PrefetchCustomeSortRows` | String | Имя кастомного метода сортировки |
| `PrefetchSortGroup` | String | Поле приоритета группы (порядок групп не меняется при смене направления) |
| `PrefetchIndices` | String (JSON) | Индексы: `[["Количество","Наименование"],["Сумма"]]` |
| `PrefetchMaxIndexPageSize` | Int64 | Макс. размер страницы индекса |
| `PrefetchHierarchyIndex` | Boolean | Индекс иерархии с разворотом |

### Параметры поиска

| Поле | Тип | Описание |
|------|-----|----------|
| `PrefetchSearchField` | String | Поле поиска |
| `PrefetchSearchFields` | ArrayString | Поля поиска |
| `ФильтрСтрока` | String | Строка поиска |
| `PrefetchSearchType` | String | Тип сравнения; `BaseVector` = любое из слов |
| `PrefetchTranslitSearch` | Boolean | Учёт неверной раскладки |
| `PrefetchSearchAddChildren` | Boolean | Добавлять детей найденных записей |

### Суммирование и выбор записей

| Поле | Тип | Описание |
|------|-----|----------|
| `PrefetchSelection` | Record | Выделенные записи для суммирования |
| `PrefetchPageCount` | Int64 | Лимит навигации отчёта (при PrefetchSelection) |
| `PrefetchCustomeOutcome` | String | Метод постобработки итогов |
| `PrefetchOriginalOutcome` | Boolean | Не суммировать |
| `PrefetchSelectionLimit` | Int64 | Макс. возвращаемых записей; дефолт 19 950 |

> [!note] Sum.ByMethod
> При `Sum.ByMethod` — передавать записи и в `selection`, и в `PrefetchSelection` (платформа не передаёт selection в первичку).

### Выходные поля (в итогах результата)

| Поле | Тип | Описание |
|------|-----|----------|
| `PrefetchDataExist` | Boolean | Данные существуют |
| `PrefetchDataCreated` | DateTime | Дата создания сессии |
| `PrefetchDataValidUntil` | DateTime | До какой даты кэш валиден |
| `PrefetchSessionId` | UUID | Передавать в следующий вызов для работы с тем же кэшем |

### Обработка ошибок

При недоступности сервиса / ошибке (без `PrefetchThrowErrors`):
- В `outcome` — нулевая сессия
- Выполняется прямой вызов метода (fallback без ошибки пользователю)

## Prefetch.Data — кэширование виджетов

Кэширует `Record Func(Record filter)`. Дефолты: обновление 1 сутки, TTL 1 неделя.

**Алгоритм:** данные < 1 дня → мгновенный ответ; 1–7 дней → ответ из кэша + фоновое обновление; > 7 дней → синхронное построение.

> [!warning] Одна сессия = один метод первички. Смешивание ломает механизм обновления.

---

## Prefetch.AppendBatch — ручное наполнение

Для сценариев без списочного метода (данные порциями из внешнего источника). Только курсорная навигация. Обязательные фильтры: `PrefetchSessionId`, `PrefetchMethod="Prefetch.AppendBatch"`, `PrefetchBatch` (RecordSet).

---

## Блокировка сессии

`PrefetchSessionLock=true` в `Prefetch.List` + удаление через `Prefetch.DiscardSessionLock`. Защита от race при параллельных вызовах. Держать с осторожностью: блокировка живёт всё время выполнения первички.

---

## Курсорная навигация

Соглашения:
1. Курсор всегда в одном формате (состав и типы не меняются)
2. Обязателен `PrefetchCursorFields`
3. Начальный курсор всегда `null`

Поиск по индексу с курсором: формат `{поле_индекса: ..., IndexOffset: ...}`.
Поддержана только навигация вниз (ndFORWARD).
