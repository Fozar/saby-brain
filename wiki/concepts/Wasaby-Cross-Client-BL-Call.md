# Wasaby: Вызов БЛ-метода под другим клиентом и пользователем

Паттерн для ситуации: основной метод работает в аккаунте 1 под пользователем 1, нужно вызвать подметод в аккаунте 2 под пользователем 2 (та же персона, но другой user_id в другой схеме).

---

## Архитектурный принцип

> [!important] Фундаментальное ограничение онлайна
> В онлайне пользователь имеет доступ **только к схеме своего аккаунта**. Если нужно работать в другом аккаунте — необходимо найти соответствующего пользователя в этом аккаунте и авторизоваться под ним явно. Одного ID персоны недостаточно — нужен конкретный `userID` в целевой схеме.

---

## Почему `CreateMultitenantEndpointByClientId` не подходит

`CreateMultitenantEndpointByClientId` переключает только клиент (схему), но оставляет текущего пользователя. В итоге выполнение идёт под пользователем, который может не иметь правильного контекста в целевой схеме.

---

## Рабочие подходы

### Подход 1: AuthByClientAndUserId (предпочтительный)

```python
# Шаг 1: найти user_id нужного пользователя в целевом аккаунте
# (поиск по ID персоны в схеме аккаунта 2)

# Шаг 2: авторизоваться под конкретным пользователем
auth_data = sbis.AuthByClientAndUserId(client_id_2, user_id_2)
end_point = sbis.EndPoint('online', auth_data=auth_data)
result = end_point.MyObject.MyMethod(params)
```

Применяется командой «Подготовка к отправке и печать отчётности» (по состоянию на 10.07.25).

### Подход 2: CreateMultitenantEndpointByClientId + Session.Set

```python
# Шаг 1: войти в схему клиента 2
end_point = sbis.CreateMultitenantEndpointByClientId(client_id_2)

# Шаг 2: подменить sid на sid пользователя 2 в целевом аккаунте
end_point.Session.Set(icsSESSION_ID, target_sid)

# Шаг 3: вызвать метод
result = end_point.MyObject.MyMethod(params)
```

> [!warning] Осторожно с именем константы
> Использовать **`icsSESSION_ID`** (элемент перечисления), а не строку `'sid'`. (Бойцов Е., 11.07.25)

## Запланированный новый API

Обсуждается добавление метода `CreateMultitenantEndpointByClientAndUserId` — аналог `CreateMultitenantEndpointByClientId`, но с одновременной подстановкой нужного пользователя. Задача выписана по результатам обсуждения (10.07.25, Бойцов Евгений / Черемисин Илья).

---

## Петля вызовов (Call Loop) — как избежать

При межклиентских вызовах через `EndPoint` возникает петля, если:
- Целевой сервис и текущий сервис — **та же группа серверов + тот же аккаунт**
- Неправильно настроены параметры авторизации (`auth_data`)
- Используется только `TenantContext` без явной смены `EndPoint`

**Признаки петли:** в профилировщике виден рекурсивный вызов того же метода.

**Решения:**
1. Убедиться, что целевой клиент действительно другой (другая схема → другой `clientID`)
2. Использовать `AuthByClientAndUserId` вместо переиспользования текущей сессии
3. Не передавать текущие `auth_data` при вызове на тот же сервис под другим пользователем без явной смены endpoint

## Бизнес-контекст

Типичный кейс — бухгалтерия: бухгалтер ОБ находится в аккаунте ОБ, но должен видеть данные клиентов (хранятся в их схемах). Он имеет доступ к аккаунтам клиентов (та же персона → находим user_id в схеме клиента), вызываем метод в правильном контексте.
