# Конвертация БД — подсистема Системы обновлений

Сервис `database-converter` — часть приложения **Управление версиями** (Хоттабыч). Выполняет операции над базами данных в процессе обновления приложений Saby/Wasaby.

## Назначение

Подсистема отвечает за следующие типы операций над БД:

| Тип операции | Описание |
|-------------|---------|
| **Конвертация БД** | Перевод схемы БД из состояния старого дистрибутива к новому |
| **Разворот данных** | Трансформация существующих данных под новую схему |
| **Конвертация эластиков** | Операции над Elasticsearch-индексами |
| **Население индексов эластиков данными** | Наполнение ES-индексов после их пересоздания |
| **Построение отчётов анализа БД** | Диагностические отчёты по состоянию базы данных |

Для выполнения задач конвертации используется утилита **jinnee**. Для построения отчётов анализа БД — утилита **cloud-update-report**.

## Архитектура сервиса

Сервис работает в **Kubernetes**, самостоятельно масштабируется. Не имеет собственной БД — работает исключительно с **Redis** для хранения оперативных данных.

### Компоненты

```
database-converter (Kubernetes)
├── 1. API-слой               — приём задач, добавление в очередь Redis, мониторинг
├── 2. Очередь задач          — Redis List (<ServiceName>_<TaskId>) + Redis Hash (параметры, SessionId, X-Uniq-ID)
│                               BLPop — блокирующее извлечение
├── 3. CustomEntryPoint       — извлечение из очереди, десериализация, передача в пул
├── 4. Служебный пул          — потоков = App.Ресурсы.Предельные.CPU + 1
│                               лимит конкурентности: ЧислоПроцессов × ЧислоПотоков × КоэфДопустимойЗагрузки
│                               выполняет: подготовку/запуск jinnee, скачивание дистрибутивов,
│                               построение отчётов, отправку результатов
├── 5. Модуль отчётности      — отдельный поток, пакетная отправка отчётов с заданным интервалом
├── 6. Мониторинг задач       — планировщик, периодическая проверка зависших задач в Redis
└── 7. Автомасштабирование    — сравнение планируемой нагрузки с текущим числом Pod;
                                scale-команды в datacenter; блокировка во время уменьшения
```

> [!key-insight] Stateless-архитектура
> Сервис не имеет собственной базы данных. Всё оперативное состояние (очереди, контексты, отчёты, блокировки) хранится в Redis. Это позволяет горизонтально масштабировать количество Pod без координации на уровне БД.

## Схема взаимодействия сервисов

```
version-manager
  │  постановка задач (JSON-RPC)
  ▼
Redis ◄──────────────────────── database-converter Pods
  │  добавление задач в очередь       │
  │  публикация отчётов               ├── Pod 1 → jinnee 1..N → db-host 1..N
  │  хранение активных задач          ├── Pod 2 → jinnee   → db-host N+1
  ▲  планируемая нагрузка             └── Pod M → jinnee   → db-host M*N
  │
  └── получение задач / отчёты ──► version-manager

database-converter
  ├── → cloud control api     (блокировка/разблокировка клиентов при посхемных обновлениях;
  │                            уведомление готовых клиентов)
  ├── → file-transfer          (выгрузка/загрузка чанков логов конвертации)
  │     ├── FileTransfer.Upload  (чанки во время конвертации, сжатые)
  │     └── FileTransfer.Download (сборка после завершения → zip-архив)
  └── → distribution storage  (готовые архивы логов)
        → fs-public            (добавление к файлам обновления)
```

**Виды взаимодействия:** JSON-RPC, Запросы к Redis, IPC (jinnee → сервис), Конвертация БД.

### Логирование конвертации

В процессе конвертации jinnee передаёт сервису логи через **IPC**:
1. Логи буферизируются в памяти (размер буфера задаётся облачным параметром)
2. При заполнении буфера — чанки сжимаются и выгружаются в file-transfer (`FileTransfer.Upload`)
3. По завершении конвертации — чанки скачиваются (`FileTransfer.Download`), перепаковываются в zip-архивы, выгружаются в distribution storage

## Организация кода

- **Базовый шаблон:** `basic_platform_no_db`
- **Собственный модуль:** `DatabaseConverter`
- **Зависимости:** `DiskClient`, `EntryPoint`, `Offline DB Converter`, `PermissionChecker`, `UserServiceCloud`
- **Сторонние библиотеки:** не используются
- **Репозиторий:** `https://git.sbis.ru/orchestration/admin.git`
- **Точка входа задачи:** `DatabaseConverter.ExecuteTask`

## Облачные параметры (Управление версиями → Конвертеры)

| Параметр | Описание |
|---------|---------|
| `Задержка после масштабирования` | Время охлаждения после успешного autoscaling (изменения числа Pod) |
| `Использовать Redis для отчётов` | Публикация прогресса конвертации через Redis вместо callback `Product.AgentReports` |
| `Потоковая передача логов конвертации` | Включение стриминга логов чанками в file-transfer; zip-пересборка после конвертации |
| `Процент использования воркеров` | Доля воркеров служебного пула, отведённая под задачи конвертации БД |
| `Размер буфера логов` | Размер буфера в МБ для потоковой передачи логов (отдельный буфер на каждую БД) |
| `Таймаут автоматического масштабирования` | Таймаут выполнения операции scale (изменения числа Pod) |

**Параметр в разделе Обновления → Логи:**

| Параметр | Описание |
|---------|---------|
| `ПодробныеСообщенияДжинна` | Транслировать ли в журнал обновления подробные сообщения jinnee |

## Задачи планировщика

Документация явных именованных задач планировщика не содержит. Единственная периодическая операция — **мониторинг зависших задач**: планировщик внутри модуля мониторинга периодически проверяет отсутствие выполняющихся методов и удаляет «зависшие» записи из Redis.

## Связь с Хоттабычем

Подсистема «Конвертация БД» — один из ключевых элементов [[Хоттабыч-System|Системы обновлений Хоттабыч]]. В терминологии Хоттабыча «Конвертация БД» — это перевод схемы БД из состояния старого дистрибутива к новому с помощью утилиты jinnee. Сервис `database-converter` реализует эту операцию как масштабируемый Kubernetes-сервис.
