# Хоттабыч — Система обновлений

Сервис: [vm.sbis.ru](https://vm.sbis.ru/) | Путь в cloud.sbis.ru: Оркестратор облака → Управление версиями

Хоттабыч — облачное приложение для управления обновлениями, дистрибутивами и применения патчей/скриптов на Wasaby Framework.

## Назначение

- Автоматическое обновление приложений на тестовых и рабочих серверах
- Хранение и ротация дистрибутивов
- Анализ истории обновлений и ошибок
- Применение патчей и скриптов
- Выпуск релизов из нескольких приложений

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| Дистрибутив приложения | Именованное файловое представление приложения, подлежащего установке |
| Версия дистрибутива | Конкретный экземпляр в хранилище (zip-архивы + xml метаданные) |
| Патч | zip-архив для быстрого изменения файлов версии дистрибутива. Создаёт новую версию |
| Скрипт | zip-архив с .orx методом для выполнения произвольного кода на сервисе |
| [[UpdateSystem-DBConversion\|Конвертация БД]] | Перевод БД из состояния старого дистрибутива к новому через jinnee-utility; реализована как Kubernetes-сервис database-converter |
| Jinnee-utility | Утилита конвертации БД, разворота данных, регистрации параметров |
| Персональное приложение | Multitenancy-приложение: данные клиентов в разных PG-схемах |
| Неперсональное приложение | Приложение, хранящее данные в одной схеме БД |
| Управление облаком | Сервис с описанием всех обновляемых приложений, серверов, настроек |
| Утилита обновления | Набор исполняемых файлов/библиотек с логикой обновления, доставляется агентом с хранилища дистрибутивов |
| Сервер конвертации | Выделенный сервер с агентом обновления, запускающий jinnee-utility для конвертации БД |
| Анализ изменений БД | Отчёт: сравнивает текущее состояние БД с версией дистрибутива, показывает различия с детализацией критичности (для «лёгкой конвертации») |

## Архитектура

```
Хоттабыч
├── Управление версиями — запуск/контроль обновлений, история
├── Планы выпуска — релизы из нескольких приложений, один на несколько облаков
├── Хранилище дистрибутивов — FileStorage: дистрибутивы + логи
├── Агенты обновления — демон (Linux) / служба (Windows), 1 агент на сервер
├── datacenter — обновление приложений в Kubernetes
└── apps — обновление внешних приложений (мобильных, оффлайн)
```

## Типы обновлений

**Лёгкие** (без простоя):
- Лёгкое обновление
- Лёгкая конвертация

**Блокирующие** (ограничивают работу сервиса или клиента):
- Полное обновление
- Посхемное обновление
- Мастер-реплика

## Фазы обновления

Доставка → Подготовка → **Обновление** → Откат → Подтверждение → Очистка папок

Переход между фазами: автоматически или кнопкой в интерфейсе.

Дополнительная фаза: **Отложенная регистрация** — запускается после Подтверждения; для планов выпуска обновление в этой фазе считается завершённым. Метод: `MetadataRegistratorDelayedRegistration`.

## Составляющие обновления

| Тип | Механизм | Описание |
|-----|----------|---------|
| BL-узел (VM) | агент обновления | Доставляются только изменённые модули нового дистрибутива |
| Контейнер (K8s) | datacenter service | Хоттабыч взаимодействует с Kubernetes через сервис datacenter |
| RPM-пакет | YUM package manager | Для не-платформенных приложений; имя: `sbis-<сервис>-<версия>-<билд>.<ОС><арх>.rpm` |
| База данных | jinnee-utility | Конвертация БД + разворот данных (отдельная задача в МАО) |
| Статика | — | Сервисы статики; загрузка модулей всех UI-приложений |
| Регистрация в облаке | metadata-registrator | Разбор дистрибутива и отправка метаданных в целевые сервисы |
| update.sbis.ru | cdn upload | Загрузка офлайн-приложений на CDN |
| Магазины приложений | — | Выкладка в App Store / Google Play |

> При откате: для BL/статики — восстановление файлов из бэкапа. Откат конвертации БД **невозможен** технически.

## Серверные события обновления

На каждой фазе (подготовка, обновление, подтверждение) публикуется серверное событие в начале и конце.

**Формат имени:** `versionmanager.[фаза][действие]`
- `[фаза]`: `prepare`, `update`, `confirm`
- `[действие]`: `started`, `complete`

**Примеры:** `versionmanager.updatecomplete`, `versionmanager.preparestarted`

**Поля события (Record):**

| Поле | Тип | Описание |
|------|-----|---------|
| `Product` | String | Имя обновляемого приложения |
| `Distribution` | String | Имя дистрибутива |
| `UpdateType` | String | Тип обновления (см. ниже) |
| `IsRollback` | Bool | Только на фазе отката при utINSTALL |
| `OldVersion` | String | Версия до обновления |
| `NewVersion` | String | Версия после обновления |
| `DateTime` | DateTime | Время публикации |
| `Groups` | ArrayInt | ID сервисов/tenant-групп из облака |
| `ВидЛогикиМассив` | ArrayString | Типы обновляемых серверов |
| `Services` | ArrayString | Системные имена обновляемых сервисов |
| `Personal` | Bool | Является ли приложение multitenancy |

> При откате — события `updatestarted`/`updatecomplete` с инвертированными `OldVersion`/`NewVersion`.

**UpdateType коды:**

| Код | Тип |
|-----|-----|
| `utLITE` | Лёгкое |
| `utFULL` | Полное |
| `utPER_SCHEMA` | Посхемное |
| `utLITE_CONVERT` | Лёгкая конвертация |
| `utMASTER_REPLICA` | Мастер-реплика |
| `utSCRIPT` | Скрипт |
| `utUPLOAD` | Загрузка на update.sbis.ru |
| `utDEPLOY` | Разворот пакетов данных |
| `utINSTALL` | Установка сервиса |
| `utUNINSTALL` | Деинсталляция сервиса |
| `utLITE_PERSCHEMA` | Лёгкое посхемное (устарело) |

Подписка: сервис подписывается на одно или несколько событий и фильтрует по `Product`/`Distribution`.

## Настройки обновления приложения (облачные параметры по типу)

Путь в Хоттабыче: Обновления → контекстное меню → «Настройка обновления».

**Раздел Особенности:**
- `Запрет полного обновления` — запретить тип «Полное»
- `Только легкое обновление` — запретить все обновления с конвертацией
- `Останавливаться только на ошибках` — не останавливаться между фазами (авто-завершение)
- `Разворачивать dpack по умолчанию` — флаг «Разворот dpack» включён по умолчанию в диалоге запуска

**Конвертация:**
- `Шаблон развертывания БД` — шаблон для сверки конфигурации; при расхождении обновление не запустится
- `Число потоков` — интенсивность конвертации (на хост БД)

**Обратный вызов** — URL, вызываемый по завершении обновления (автотесты).

**Облачные параметры (Управление версиями → Обновления):**

| Тип | Параметр | Описание |
|-----|---------|---------|
| Лёгкое | `ПоследовательныйРазворотДанных` | Разворачивать dpack последовательно внутри сервиса |
| Лёгкое | `РазворотДанных` | Включить разворот данных при лёгком обновлении |
| Лёгкая конвертация | `ПроверятьДублирущиеСервера` | Проверка дублирующих серверов (по умолчанию включена) |
| Полное | `ЧислоЗадержекРазблокировкиПользователей` | Сколько интервалов ждать перед ошибкой если % разблокированных не меняется |
| Посхемное | `ИнтервалУведомлений` | Интервал между уведомлением клиента и началом конвертации |
| Посхемное | `КоличествоУведомляемыхКлиентов` | Кол-во клиентов на хост БД за один интервал |
| Посхемное | `ТаймаутОжиданияМетодовКлиента` | Время ожидания завершения методов клиента после блокировки |
| Посхемное | `УведомлятьКлиентов` | Отсылать ли уведомление об обновлении с возможностью отложить |
| Посхемное | `ЧислоПотоковКонвертераНаХост` | Потоков конвертера на хост при посхемной конвертации |
| Мастер-реплика | `ДополнительныйТаймаутНормализацииЛага` | Доп. время после конвертации мастера для нормализации лага реплики |
| Разворот данных | `Таймаут запуска` | Время для ответа сервиса о старте разворота (иначе ошибка) |

**Шаблон пути загрузки (внешние приложения):**
`${distr_name}` / `${zip_name}` / `${app_name}` / `${version}` / `${build_number}` / `${build}`

## Совместимость SDK и дистрибутива

Из файла `Схема сборки дистрибутива.s3distr` в корне дистрибутива:
```xml
version_sdk="26.2000"          // совместимая версия SBIS SDK
build_date="10.03.2026"        // дата создания
build_number_sdk="311"         // номер сборки
```

> [!key-insight] Версия дистрибутива
> Формат: `<версия>-<сборка>.<патчи>`. Пример: `24.5220-60` = версия 24.5220, 60-я сборка.
