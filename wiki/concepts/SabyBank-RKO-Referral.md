# SabyBank RKO Referral — Рефералка в Заявках в банк на РКО

Navigation: [[price-formation/_index]] | [[ReferralProgram-Module]] | [[ReferralDeals-System]]

Проект внедряет реферальную систему на Заявки, отправляемые в банк на открытие расчётного счёта (РКО) по API. Цель — Тензор получает статистику по заявкам и возможность начислять вознаграждения.

> [!key-insight] Ключевое отличие от стандартной рефералки
> В SabyBank отсутствуют: ссылка на продукт, виджет обратной связи, тема сделки, фиксированная величина вознаграждения. Вознаграждение назначается вручную. Банки-партнёры не используют SabyBank — они работают в своих системах.

---

## Бизнес-процесс

1. **Тензор** создаёт в SabyNet приложение SabyBank.
2. Добавляет банки (Альфа-Банк, ТБанк) как партнёров в бизнес-группу SabyBank.
3. Создаёт реферальную программу с типом **"Заявки на РКО"** и добавляет банки-партнёры.
4. Клиент Saby видит список банков-партнёров и оставляет заявку на РКО (выбрав конкретный банк).
5. При отправке заявки: в аккаунте Тензора автоматически создаётся **корешок** (stub) заявки.
6. Статусы корешков обновляются по мере обработки банком.
7. Менеджеры Тензора в SabyBank вручную назначают вознаграждение по успешным заявкам.
8. Статистика по заявкам и вознаграждениям видна Тензору в реестрах и виджетах.

**Потенциальные пользователи**: только Тензор (менеджеры мониторинга и начисления). Менеджеры Банков не используют.

## Концептуальное решение

**Сценарий "Корешок"**: вся статистика строится на основе записей `ВидЦеныДокумент`. Корешок — это запись, в которой зафиксированы ссылка на реф. программу, банк (Карта участника), UUID заявки, метаданные и статус.

**Поиск партнёра**: по `АккБанка` через API соцсети находится `@Лицо` в бизнес-группе SabyBank, по `@Лицо` — `Карта` участника реф. программы. Затем создаётся корешок.

**Вознаграждение**: ручное. Менеджер Тензора редактирует поле `Бонусы` у `ВидЦеныДокумент` (в т.ч. через ПМО).

## API методы (новые, вызываются строго в акк. Тензора)

### `GetBankParticipants()`
Возвращает банки-участники реф. программ SabyBank:
```json
{"ID_РефПрограммы": {"Тип": "Заявки на РКО", "Участники": [{"ИмяБанка": "...", "АккБанка": "..."}]}}
```
Используется мастером заявок — определяет доступные банки.

### `CreateLeadStub(program_id, bank_account, app_id, metadata)`
Создаёт корешок заявки.
- По `АккБанка` → находит `@Лицо` (API соцсети) → находит `Карта` участника
- Создаёт `ВидЦеныДокумент` с `ТипСвязи=10`
- Повторный вызов с тем же `app_id` → исключение (дублирование)
- Смена банка = удаление + создание нового корешка

### `UpdateLeadStub(app_id, metadata)`
Обновляет метаданные корешка при смене статуса обработки заявки банком.
- По `ИдентификаторЗаявки` находит корешок (`ВидЦеныДокумент`)
- Обновляет метаданные и `ТипСвязи`

## Схема хранения корешка (ВидЦеныДокумент)

| Поле | Смысл |
|------|-------|
| `ВидЦены` | ID реферальной программы |
| `Карта` | Карта участника (Банк-партнёр) |
| `ДатаВремя` | Дата и время создания заявки |
| `OperationID` | UUID документа заявки в акк. банка |
| `ТипСвязи` | 10 = в работе, 11 = успешно, 12 = неуспешно |
| `EffectiveDate` | Дата завершения (NULL пока не завершена) |
| `Бонусы` | Начисленное вознаграждение (NULL если не назначено) |
| `Атрибуты` | Метаданные (акк. клиента, ID заявки, прочее) |

> [!note] Уникальность
> Повторное создание корешка с тем же `OperationID` запрещено. Нужна проработка индексации для поиска корешка по `ИдентификаторЗаявки`.

**Жизненный цикл корешка:**

| Состояние | ТипСвязи | EffectiveDate | Бонусы |
|-----------|----------|---------------|--------|
| Создана | 10 | NULL | NULL |
| В работе | 10 | NULL | NULL |
| Завершена успешно | 11 | Дата завершения | NULL→заполняется вручную |
| Завершена с вознаграждением | 11 | Дата завершения | Кол-во |
| Завершена неуспешно | 12 | Дата завершения | NULL |

## Архитектура создания корешков (2026-05-28)

> [!key-insight] Принцип: корешок создаётся целевым документом
> Сформулировано Свешниковым А. 2026-05-28. Подробнее: [[ReferralStub-TargetAction-Pattern]]

**Два сценария**, которые обсуждались:

**Сценарий 1 (устаревший подход):** партнёр нажимает «Заявка» → создаётся заявка + корешок одновременно.
— Работает только если оффер не привязан к ЦРМ и не имеет определённого целевого документа.

**Сценарий 2 (правильный):** любая заявка в ЦРМ (или другой системе) с источником реф. программы сама создаёт свой корешок.

**Итоговая позиция:** кнопка «Заявка» в кабинете партнёра должна создавать заявку с правильным источником — **а корешок создаётся уже этой заявкой** (как и любой другой заявкой с таким источником).

Это универсальный паттерн: [[ReferralStub-TargetAction-Pattern]]

## Выборки БД — основные сценарии

> [!note] Новые индексы
> Создаётся новый составной индекс `ВидЦеныДокумент (ВидЦены + ДатаВремя + ТипСвязи)` для фильтрации по статусу заявки.  
> Под вопросом: нужен ли новый индекс для суммы вознаграждений по дате создания (существующий `EffectiveDate` хранит дату завершения, не создания).

### Узнать факт участия Партнёра в реф. программе
1. Через API соцсети в бизнес-группе SabyBank ищем `@Лицо` контрагента партнёра
2. По `@Лицо` → связанные `Карты` → только та, что связана с `ВидЦены` реф. программы

**Индекс**: существующий `Карта(Лицо)` (аналог: участие в UDS/PremiumBonus)

### Получить список заявок по реф. программе (все партнёры)
- Выбираем `ВидЦеныДокумент` по `ВидЦены`, сортировка по `ДатаВремя`
- **Индекс**: существующий `(ВидЦены + ДатаВремя)` (аналог: список продаж по скидке)
- Фильтр по статусу (`ТипСвязи`): **новый индекс** `(ВидЦены + ДатаВремя + ТипСвязи)`

### Получить список заявок партнёра по реф. программе
1. API соцсети → `@Лицо` партнёра → `Карта` участника конкретной реф. программы
2. `ВидЦеныДокумент` по `Карта + ДатаВремя`

**Индекс**: существующий `(Карта + ДатаВремя)` (аналог: продажи по карте лояльности)

### Обновить корешок заявки
- По `ИдентификаторЗаявки` (= `ВидЦеныДокумент.OperationID`) находим и обновляем запись

**Индекс**: существующий по `OperationId`

### Получить сумму вознаграждений за период по реф. программе
- `ВидЦеныДокумент` в диапазоне дат (`ДатаВремя`), `Бонусы > 0`
- **Осторожно**: текущий индекс `(ВидЦены + EffectiveDate + Бонусы <> 0)` использует дату завершения — нужна проработка (выборка должна быть по дате **создания**)

### Получить сумму вознаграждений за период по партнёру
1. API соцсети → `@Лицо` → `Карта` участника
2. `ВидЦеныДокумент (Карта + ДатаВремя)`, сумма `Бонусы`

**Индекс**: существующий `(Карта + ДатаВремя)` (аналог: бонусный баланс по карте)
