# Bonus.GetSaleTotals — таймаут на больших аккаунтах

## Симптом

При серверном построении страницы реестра «Бонусы/Покупки» (переход по прямой ссылке / сброс кэша) не подгружается заголовок — итоговые суммы и счётчики не отображаются. Методы `Bonus.GetSaleTotals/1` и `Bonus.GetClientListWithStatsTotals/1` не вызываются или завершаются с ошибкой.

Из логов: фронт устанавливает `X-TIMEOUT: 5000` (5 секунд). Сервер получает `ExecutionInterruptedException` — SQL прерван по сигналу клиентского disconnect.

## Root cause

`Bonus.GetSaleTotals` перед поиском данных за нужный месяц вызывал вспомогательную функцию `_get_first_sale_date`, которая запрашивала `MIN(EffectiveDate)` по таблице `ВидЦеныДокумент` **без фильтра по дате**.

**EXPLAIN ANALYZE:**
```
Aggregate (actual time=17137ms)
  Buffers: shared hit=13265 read=611460   ← ~5 ГБ с диска

  Nested Loop (loops=7, rows=428108/loop → 3M rows total)
    Hash Join: ВидЦеныРасширение ⋈ BonusPriceEntities → 7 бонусных ВидЦены
    Index Scan via AutoGeneratedIndexVidTseny(ВидЦены)
      → для каждого из 7 ВидЦены сканирует все 428K строк
      → actual time ~1561ms/loop
```

Запрос занимал **17 секунд** — в 3.4 раза больше таймаута.

## Почему индекс не помогает

Единственный полезный индекс для этого запроса — `AutoGeneratedIndexVidTseny` (только `ВидЦены`). Он не сортирован по дате → PostgreSQL обязан сканировать все строки для каждого из 7 бонусных видов цен, чтобы найти MIN(EffectiveDate).

**Полный список индексов `ВидЦеныДокумент` с EffectiveDate:**

| Индекс | Колонки | Predicate | Подходит? |
|---|---|---|---|
| `ВидЦены` (composite) | `(ВидЦены, ДатаВремя)` | — | Нет — вторая колонка `ДатаВремя`, не `EffectiveDate` |
| `BonusesEffectiveDate` | `(@ВидЦеныДокумент DESC, EffectiveDate DESC, ВидЦены)` | `Бонусы IS NOT NULL` | Нет — leading: PK, не дата |
| `PromoCodeEffectiveDate` | `(EffectiveDate::date, SalePoint)` | `PromoCode IS NOT NULL` | Нет — наши строки имеют ВидЦены, не PromoCode |
| `CardOnSaleEffectiveDate` | `(EffectiveDate)` | `ВидЦены IS NULL` | Нет — противоположное условие |

## Варианты исправления

### Вариант 1 — Новый индекс (рекомендуется)

```xml
<!-- PriceFormation.Online.dicx -->
<index method="btree" name="BonusSaleEffectiveDate" null_keys="1" service="0" unique="0">
  <column name="ВидЦены" sorting="ASC"/>
  <column name="EffectiveDate" sorting="ASC"/>
  <predicate>
    <body>"EffectiveDate" IS NOT NULL AND ("Sale" IS NOT NULL OR "Документ" IS NOT NULL)</body>
  </predicate>
</index>
```

PostgreSQL делает 7 index-seek'ов (по одному на каждый бонусный ВидЦены) и сразу получает минимальную дату. Миллисекунды вместо 17 секунд. Логика метода остаётся полностью корректной — поиск назад до первой продажи за всё время.

**Минус:** изменение схемы БД, требует сборки и согласования.

### Вариант 2 — Ограниченный поиск назад (без изменения схемы)

Убрать `_get_first_sale_date` полностью. Вместо поиска до «первой продажи за всё время» — искать назад не более N месяцев. Каждый шаг — запрос за конкретный месяц с фильтром по дате, который работает быстро.

```python
_MAX_LOOKBACK_MONTHS = 24  # или больше

for _ in range(_MAX_LOOKBACK_MONTHS):
    totals = _get_totals_for_period(filters, cur_start, cur_end)
    if (totals.Get('TotalSalesCount') or 0) > 0:
        return _make_result_record(totals, cur_start, cur_end)
    cur_start, cur_end = _prev_month_range(cur_start)

return empty_result(original_start, original_end)
```

**Минус:** если последние продажи старше N месяцев — метод вернёт нули вместо реальных данных.
