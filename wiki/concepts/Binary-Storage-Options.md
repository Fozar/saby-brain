# Хранение бинарных данных: выбор хранилища

## Сравнение трёх хранилищ

| | SabyDisk | FileStorage | file-transfer |
|---|---|---|---|
| **Назначение** | Пользовательские файлы | Внутренние файлы сервисов | Временные файлы |
| **Доступ** | Пользователи + внутренний | Только внутри облака | Внутренний + внешний |
| **Резервирование** | Да | Да | Нет |
| **TTL** | Нет | Нет | Да (авто-удаление) |
| **Иерархия** | Да (папки, ссылки) | Нет | Нет |
| **Права доступа** | Гранулярные | По ссылке без авторизации | По ID |
| **Версионирование** | Да | Нет | Нет |
| **Подходит для** | ЭДО, соцсеть, диск | Дистрибутивы, импорт данных | LRS-результаты, Huge Payload |

## Когда что использовать

**SabyDisk**: пользовательские документы, ЭДО, соцсеть, новости.

**FileStorage**: долгосрочное хранение файлов _сервисов_ (не пользователей); доступ по ссылке без авторизации.

**file-transfer**: 
- результат длительной операции для клиента
- Huge Payload Protocol (большие аргументы async-вызовов)
- любые временные файлы с коротким TTL

## Критерии выбора

При оценке выбора хранилища рассматривать:
- Количество файлов и их объём
- Время жизни (TTL нужен → file-transfer)
- Частота доступа (горячий → FileStorage hot-cache, редкий → SWIFT cold)
- Способы доступа (внешний с авторизацией → fs-auth, без → fs-public)
- Критичность потери данных (нет резервирования → file-transfer только для временных)
- Бюджет

> [!note] Небольшое количество файлов малого объёма с долгим хранением → СУБД с репликацией (PostgreSQL) может быть оптимальнее всех трёх.
