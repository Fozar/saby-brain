# Планы выпуска — Release Plans

Планы выпуска — инструмент Хоттабыча для автоматизации и контроля ежедневных выпусков приложений СБИС/Saby. Сервис: [vm.sbis.ru](https://vm.sbis.ru/) → Планы выпуска.

Каждый день эксплуатации запускаются десятки (при релизах — сотни) обновлений и скриптов. Планы выпуска описывают весь порядок работ, их ограничения и зависимости, а затем исполняют план автоматически — снижая влияние человеческого фактора.

## Термины

| Термин | Определение |
|--------|-------------|
| **План выпуска** | Описание релиза версии, выполняемого за несколько дней (н-р, с пятницы по воскресенье) |
| **Работа** | Минимальная единица плана: обновление сервиса, скрипт, патч, разворот данных, регистрация параметров, пользовательская задача |
| **Этап** | Именованная папка-контейнер для работ; имеет статусы и может быть вложен в другой этап |
| **Задача** | Экземпляр запуска работы на конкретном Хоттабыче-облаке; история задач = история запусков на стендах |
| **Стенд** | Окружение: pre-test, test, fix, fix-kz, prod, prod-kz, prod-uz |
| **Ограничение запуска** | Условие, при котором работа не стартует: зависимость, время, кол-во параллельных задач, интервал |
| **Приоритет работы** | Очерёдность при одновременной готовности нескольких работ: низкий / нормальный / высокий |
| **Волна обновления** | Порядок обновления приложений одного дистрибутива: 1-я волна (главные сервисы) и 2-я волна (прочие) |

## Структура плана

### Шаблоны

**Шаблон ХФ (хот-фикс)** — основной шаблон для ежедневных выпусков. Содержит:
- Пункты согласования плана и журнал работ ЦОД
- Стандартные этапы (папки), разбитые по времени старта
- Папка `на 03:00` — обновление `help` (ежедневное)
- Папки `на 03:00` и `на 04:00` — этапы по типам обновлений приложений
- Папки `на 04:00` и `на 05:00` — скрипты по основным/прочим сервисам и прочие работы

> [!important] Для создания нового плана выпуска обязательно копировать шаблон ХФ.

**Шаблон мажорных работ** — пункты для мажорных выпусков по всем волнам (окружениям). При подготовке мажора нужные пункты копируются в мажорные планы.

### Текущие планы

Создаются из шаблона ХФ. Структура совпадает с шаблоном. При планировании работ выпускающий добавляет пункты в папки `на ОС` (Обновление сервиса) и `на ВНР` (Выполнить на рабочем); в день выпуска распределяет их по временным папкам.

### Мажорные планы

Размещены в корне Планов выпуска в папках по версиям. Каждый мажор — несколько планов:
- Четверг — окружение 1, волна 1
- Пятница — окружение 2 (волна 1: админка; волна 2: прочие сервисы)
- Суббота — волна 1: основные; волна 2: основные и прочие
- Воскресенье — прочие волна 2, ХФ все волны

### Архив

Хранит старые планы со статусом «Выполнен». Срок хранения задан параметром `Управление версиями.Планы выпуска.Срок хранения архивных планов` (сейчас 730 дней / 2 года). Видны только при фильтре «Все» (по умолчанию «Все кроме завершенных»). При ошибочном создании плана — переименовать в «Создан ошибочно <дата>», переместить в архив, поставить статус «Выполнен».

## Создание нового плана выпуска

Роль «Релиз-менеджер» или «Релиз-менеджер расширенный» обязательна. Без неё редактирование планов недоступно.

**Способ 1** — через реестр: нажать «+», выбрать тип работы, заполнить поля, в поле «Дата» указать дату несуществующего плана → план создаётся автоматически.

**Способ 2** — автоматически при переносе: изменить поле «Дата» у существующей работы на несуществующую дату — работа попадёт в новый план.

**Способ 3** — вручную через «+» на главной странице планов. Правила именования:
- Название: «Обновление ддммгг»
- Версия: версия выпускаемого ХФ основных сервисов в скобках (или пустое)
- Период: даты выпуска на бой
- Ответственный: вечерний выпускающий по графику дежурств

**Удаление** не предусмотрено — только перемещение в архив.

## Алгоритм работы с планами (операции над работами)

### Создание работ

Через контекстное меню ПКМ на нужном этапе или кнопку «три точки» в конце строки.

При добавлении работы указывается ссылка на ВНР/ОС — происходит автозаполнение. Для работ по **всем основным сервисам** поля заполняются автоматически. Если скрипт по **списку клиентов**:
- Если кол-во id/номеров аккаунтов > N: работу заводим по всем дистрибутивам основных сервисов
- Если ≤ N: проверяем, на каких приложениях клиенты, добавляем скрипт по дистрибутиву отдельно с указанием приложений

### Изменение даты работы (перенос в другой план)

Через ПМО → «Изменить дату». В зависимости от ситуации работа либо остаётся в исходном плане и клонируется в конечный, либо удаляется из исходного и добавляется в конечный.

> [!important] При создании и переносе работы в другой план у пункта плана по всем стендам всегда проставляется кирпич (запрет выполнения).

### Клонирование

Создаёт полный дубль элемента плана. Контекстное меню → «Клонировать» → открывается карточка клона для редактирования.

### Перемещение внутри плана

- **Drag-and-drop** — перетаскивание мышью (только один элемент)
- **ПМО** → «Перенести» → выбрать целевой этап (только внутри одного плана)

### Удаление

- Через контекстное меню — удаляет одну выбранную работу или пустой этап
- Через ПМО — удаляет все выбранные работы и пустые/непустые этапы

## Типы работ

| Тип | Описание |
|-----|----------|
| **Обновление** | Обновление сервиса/дистрибутива |
| **Разворот пакетов данных** | Разворачивает пакет(ы) данных; в работе задаётся имя пакета; Хоттабыч берёт дистрибутив по имени пакета из облака |
| **Загрузка на update.sbis.ru** | Выгрузка дистрибутива во внешнее хранилище |
| **Скрипт** | Запуск .orx-скрипта; список сервисов — из файла `services` внутри архива |
| **Патч** | Применение патча из ВНР + лёгкое обновление на пропатченную версию |
| **Регистрация значений параметров** | Регистрация до 10 параметров из документа ВНР; значения берутся в рантайме |
| **Пользовательская** | Выполняется сотрудником эксплуатации вручную согласно инструкции |
| **Внешнее приложение** | Обновление внешних (необлачных) приложений |

### Работа типа "Включение нового функционала"

Работает **только на БОЮ** (на тестовых стендах включается вручную через сервис управления функционалами).

Алгоритм:
1. Тестировщик создаёт ВНР → задаёт идентификатор функционала и условия выполнения
2. Тестировщик в сервисе управления функционалами настраивает переключатель (для каких клиентов/групп/параметров)
3. Выпускающий заводит работу типа «Включение нового функционала» с ссылкой на ВНР
4. В ходе плана plan вызывает: `feature-ctrl.FeatureCommit(feature_id)` → переключатель включается автоматически

Минимальный набор настроек карточки: название (идентификатор функционала), ссылка на ВНР, стандартные ограничения.

### Работа типа "Обновление кластеров"

Специфичные настройки:
- **Кластеры** — выбор папки с однотипными кластерами (обновляются все входящие в папку по критериям) либо одиночные кластеры
- **Волна** — обновить только кластеры выбранной волны; волны задаются эксплуатацией на точках доступа согласно опросникам разработчиков; облако определяет волну кластера по **максимальной волне** среди его точек доступа

При запуске работы в Хоттабыче создаётся одно обновление, содержащее все выбранные кластеры.

### Алгоритм патча

1. ReleaseManager вызывает метод в Хоттабыче для последовательного применения патчей
2. Хоттабыч проверяет, применялся ли патч к текущей версии (если да — пропускает)
3. Хоттабыч выкачивает архивы из ВНР в папку «Патчи» хранилища дистрибутивов, запускает применение
4. При ошибке — работе присваивается «Ошибка запуска» с расшифровкой
5. Хоттабыч запускает лёгкое обновление на пропатченную версию (для prod — с флагом последовательного обновления логик)

### Алгоритм скрипта

Список сервисов задаётся разработчиком в файле `services` внутри архива скрипта. Файл содержит системные имена сервисов.

При добавлении ссылки в поле работы происходит подтягивание выполненных задач на стендах + загрузка/актуализация файла скрипта в хранилище (сравнение по имени архива и md5).

Если скрипт не успел выполниться до часа завершения работ — переносится на следующий день. При повторном запуске выполняется только на тех клиентах, где ещё не выполнялся.

## Ограничения запуска

Планировщик (раз в минуту) анализирует ограничения работы и всех её надэтапов. Если ни одно не срабатывает — работа запускается.

| Тип ограничения | Описание |
|----------------|----------|
| **Время запуска** | Работа стартует не раньше заданного времени |
| **Флаг «Вручную»** | Работа не запустится в авторежиме; ждёт действия эксплуатации |
| **Зависимости** | Работа ждёт завершения всех указанных зависимостей и зависимостей всех надэтапов |
| **Кол-во параллельных задач** | Максимум активных задач в поддереве элемента; освобождение — при завершении задачи |
| **Интервал запуска** | Разброс времени запуска для работ по дистрибутивам, чтобы не перегружать диск |

### Контроль нагрузки плана

Нагрузка активной задачи = сумма весов логик (серверов) × весовые коэффициенты по типу логики (задаются параметрами). Параметр предела нагрузки: `Управление версиями.Планы выпуска.Предел нагрузки плана`. Может быть переопределён в реальном времени. При достижении лимита новые работы не стартуют до освобождения.

## Статусы работы

Каждая работа имеет 2 вида статуса на каждом стенде:
- **Автоматический** — выставляется release-manager, не изменяется пользователем (можно только сбросить)
- **Ручной** — задаётся вручную, отображает статус проверки

Ключевые автоматические статусы: «Выполнено» (блокирует повторный запуск), «Ошибка запуска».

Цветовая индикация версии на стенде: зелёный — совпадает; жёлтый — версия на стенде выше; красный — ниже.

## Элементы управления интерфейса

### Карточка плана

Панель отображает:
- Заголовок: название плана, ответственный выпускающий, статус
- Комментарий со ссылкой на Telegram-чат выпуска
- Прогресс-бар по стендам: Завершённые / Запущенные / Завершённые с ошибками / Не запущенные
- Строка поиска (по названию, ссылкам, ФИО ответственного)
- Иерархический справочник этапов и работ

### Статусы плана

| Статус | Возможности |
|--------|-------------|
| **Планируется** | Только изменение плана |
| **В работе** | Изменение плана + запуск работ |
| **Выполнен** | Изменение и запуск недоступны |
| **На паузе** | Запуск новых задач недоступен; исполняющиеся задачи завершают работу |

### Панель быстрого доступа

- **Покебол** — синхронизация дистрибутивов. Выгружает дистрибутивы с пользовательским статусом «Проверено» по стенду fix. Для Docker-контейнеров (содержат «-docker» в названии): строит образы и выгружает. Для rpm-работ с тегом «rpm»: запускает сборку.
  - Алгоритм jinnee_lite: при синхронизации анализируются `dependencies.xml` дистрибутивов → собирается множество требуемых версий jinnee_lite → копирование версии, если на рабочем отсутствует или хэш архива отличается
- **Пересчёт расхождений версий** — ручной пересчёт несоответствий билдов и версий в плане
- **Копировать ссылку** на план
- **Выгрузка в Excel** — для рассылок на тестирование
- **Свернуть/Развернуть все этапы**
- **Режим просмотра / редактирования** — в режиме редактирования клик на шеврон открывает диалог редактирования (минуя просмотр)
- **Комментарий к плану** — ссылка на Telegram-чат выпуска
- **Открыть план в новой вкладке**
- **Просмотр отчётов** (по приложениям / по времени)
- **Переключатель детального/упрощённого вида**

### Фильтр для эксплуатации

Специальная кнопка-индикатор отображает кол-во работ, ожидающих действий эксплуатации:
1. Задачи с ошибкой
2. Пользовательские работы в статусе «Можно выполнять» или «Выполняется»
3. Работы с флагом «Вручную» при подошедшем времени старта и снятых ограничениях

Telegram-бот подключается к чату выпуска и присылает уведомления при появлении ожидающих работ.

### ПМО (Панель массовых операций)

Активируется по клику на чекбокс пункта/этапа. Действия:
- Изменение даты (перенос в другой план)
- Перемещение этапов/пунктов внутри текущего плана
- Снятие / установка «кирпича» на выбранных пунктах по текущему стенду
- Изменение пользовательских статусов
- Удаление выбранных этапов/пунктов

### Теги

Позволяют отмечать работы и этапы. Видны всем пользователям (в отличие от тегов сообщений, которые личные). Хранятся в отдельной таблице БД с полями: название, цвет, связь с пунктами плана, связь с планом.

### Разделение по стендам

Быстрый фильтр «Все» → раскрывает список стендов: pre-test, test, fix, fix-kz, prod, prod-kz, prod-uz. Стенд можно выбрать через фильтр или клик по прогресс-бару.

## Пересчёт расхождений версий

Включает определение:
1. Версий без явного билда
2. Несоответствия версии в работе и версии на стенде fix (стенд задаётся параметром `Управление версиями.Планы выпуска.СтендПроверкиВерсииДистрибутива`)
3. Наличия более свежего билда в хранилище (патчевые билды не учитываются)
4. Несоответствий значений полей работы с документом-основанием
5. Принадлежности элемента плана текущему облаку

Пересчёт запускается: ежедневно в 03:00 для планов «В работе», при запуске этапа/работы, вручную кнопкой.

Для обновления значений всех полей из документа: кнопка «Обновить значение всех полей в карточке работы». Для конкретного поля — нажать у поля с треугольником уведомления.

> [!important] При открытой карточке и параллельном изменении данных в документе необходимо переоткрыть карточку работы для актуализации.

## Синхронизация дистрибутивов

- **Точечная** — через ПМО, одна или несколько работ/этапов → «Синхронизировать дистрибутивы»
- **Массовая** — через кнопку на панели быстрого доступа (для всех элементов плана)

## Архитектура продукта

### Сервисы и модули

```
release-manager (основной сервис)
└── БЛ модуль ReleaseManager
    ├── release-manager — хранит структуру плана в БД, исполняет план
    └── задача планировщика — раз в минуту анализирует готовность к запуску

release-external (интерфейсное приложение, 2 сервиса)
├── release-ps — интерфейс планов выпуска; внешний модуль подключается в Хоттабыч
└── release-bl-proxy — проксирование вызовов БЛ от Хоттабычей к release-manager
```

**Ключевой принцип**: plan выпуска — единый документ для всех стендов. Боевой и тестовые Хоттабычи обращаются к одному рабочему сервису release-manager через release-external.

Код: `git@git.sbis.ru:sbis/admin.git`

### Публичное API

Объект `ReleaseManagerAPI` (4 метода):
1. `GetWorkDependencies/2` — список зависимостей работ по стенду
2. `PlanList/4` — список планов
3. `UpdateWork/1` — изменение работы
4. `WorkList/4` — получение структуры плана

Документация: https://wi.sbis.ru/docs/bl/release-manager/ReleaseManager/ReleaseManagerAPI

### Связывание по ссылке

Каждая работа содержит поле «Ссылка» на документ ВНР/ОС. При добавлении ссылки в Хоттабыч — анализируются незакрытые планы, при нахождении работы — обновление привязывается к ней как задача. То же при задании ссылки на работе — опрашиваются все Хоттабычи сервиса плана.

### Резервные сервисы

Обновляются цепочкой без участия плана, но план контролирует их:
1. Два пункта плана: основной и резервный сервис с одной ссылкой на ОС
2. Запуск работы основного сервиса → Хоттабыч завершает его и автоматически обновляет резервный с той же ссылкой
3. Планы выпуска автоматически связывают работу резервного с обновлением в Хоттабыче

## Признак необходимости исполнения работы в облаке

Хранится в `CloudWorkStatus` — по одному признаку для каждого облака, используемого планом. Заполняется автоматически:

1. По умолчанию — работа должна выполняться в каждом из облаков.
2. Если работа связана с приложением — проверяется наличие приложения в каждом облаке (при редактировании работы или нажатии «Перечитать расхождение версий»). Если приложение отсутствует — признак становится отрицательным.
3. Если работа не связана с приложением — анализируется документ-основание (ВНР): набор флагов по регионам. Если флаг для региона снят — признак для облака становится отрицательным.

Знания о том, какие облака принадлежат каким регионам, Хоттабыч запрашивает у Управления облаком через API.

**Правило пропущенных работ**: все зависимости, ведущие на исключённую (пропущенную) работу, считаются недействительными и игнорируются при исполнении плана. Транзитивность не работает — если требуется цепочка A→B→C, где B пропущена, нужна явная связь A→C.

## Автономные планы — синхронизация с мастер-планом

Для изолированных регионов, где разворачивается собственный набор сервисов release-manager, реализовано клонирование плана:

- Структура клона (работы, этапы, зависимости) должна быть **полностью идентична** мастер-плану.
- Клон **не копирует**: настройки, специфичные для облаков которых нет в целевом релиз-менеджере; статусы; задачи.
- Клонированный план хранит признак «склонирован» и ссылку на источник.
- Кнопка **Синхронизация** (доступна если у плана выставлен признак клона) — актуализирует структуру (работы, этапы, настройки), не трогая статусы уже выполненных работ и задачи.

## Реестр работ

Реестр строится по таблице `Work`. По умолчанию показывает только невыполненные работы на боевых облаках (без флага принудительного выполнения).

Иерархия реестра:
- Уровень 1 — тип работы
- Уровень 2 — версия

При создании работы из реестра в статусе «В работе»: обязателен **Дата** → по дате находится/создаётся план → работа привязывается к нужному этапу согласно типу работы. Уникальность даты начала среди планов приложения обязательна.

**Время выполнения** = суммарное время задач по работе в рамках условий фильтрации.

## Отчёты по плану

### Отчёт по приложениям

Иерархический справочник (семейства → приложения → работы). Показывает только участвующие в плане объекты. Критичные сервисы помечены красным флагом. Для пользовательских работ — опциональная привязка к приложению.

### Отчёт по времени

Все работы в хронологическом линейном порядке с фазами и предполагаемым временем конвертации. Шапка: столбчатая диаграмма по интервалам в 1 час, разбитая по типам работ. Клик по столбцу — фильтр по часу. Фильтрация по часовым периодам, типам работ, наличию ошибок.

Алгоритм хронологии: если на фазе указано время — берётся как база; анализируются надэтапы и зависимости; если зависимость конвертируется — прибавляется предполагаемое время конвертации.

### Фильтр по ошибкам

В рамках отчёта по времени. Проблемы добавляются эксплуатацией из интерфейса. Поля проблемы: время обнаружения/ликвидации, дежурный ЦОД, затронутые работы, описание, ссылка на аварию, предпринятые меры, анализ.

### Выгрузка в Excel

Используется релиз-менеджерами для формирования файлов рассылок на тестирование.

## Рабочий процесс тестировщика

Тестировщики работают с планами на [vm.sbis.ru](https://vm.sbis.ru/). Необходима роль «Тестировщик прикладных сервисов».

### Ежедневный процесс

1. Релиз-менеджер создаёт план на каждый день заранее
2. Тестировщик до 09:00 указывает версию и **финальный билд** на выпуск
3. Снять «кирпич» для нужного сервиса (установить фильтр по стенду fix → найти сервис → снять запрет)
4. Зафиксировать **билд** (нажать замок в окне работы по стенду fix) — только если версия уже есть на стенде
5. Поставить **статус «Проверено»** по fix или fix-kz

> [!important] БИЛД фиксируется один раз для всех стендов. Статус «Проверено» по FIX — если приложение есть в РУ-облаке. По FIX-KZ — если отсутствует в РУ-облаке, но есть в других региональных.

Билд/статус нужны только по **обновлениям**, **разворотам данных** и **разворотам новых сервисов** — для автоматической выгрузки дистрибутивов в боевой репозиторий.

### Снятие для хот-фиксов / мажорных выпусков

- Хот-фикс: релиз-менеджер снимает кирпич перед заездом версии на fix
- Мажор: тестировщик снимает кирпич после рассылки от релиз-менеджера в чате

### Автоматическое обновление сервисов на fix

Сервисы обновляются раз в час с 09:00 до 19:00 (включительно 18:00, последнее — в 19:00).

**Условия автообновления**:
- Сервис находится в Плане выпуска на этот день
- Нет «кирпича» на работе
- Не установлен статус «Проверено» по fix
- Если в поле «Версия» нет конкретного билда — обновляется на свежий при наличии на test

**Стоп-лист** (не обновляются автоматически): cloud-ctrl, sharded-logs, rights-mng, scheduler, config-ctrl, admin-repo, version-manager, filestorage, SBISDisk, диспетчер, нехоттабычные сервисы, новые сервисы, основные сервисы, необлачные продукты, пакетные данные.

**Отключение автообновления**: поставить тег `#noauto` в комментарии к пункту плана. После исправления ошибки — удалить тег.

## Интеграция с Хоттабычем

[[Хоттабыч-System]] — ядро системы обновлений. Планы выпуска работают поверх Хоттабыча:
- release-external проксирует BL-вызовы в Хоттабыч
- Работы типов «Патч», «Скрипт», «Разворот данных» инициируют методы Хоттабыча
- При запуске работы release-manager вызывает метод запуска обновления/скрипта в нужном облаке через Хоттабыч
- Задача планировщика раз в минуту опрашивает Хоттабыч о состоянии запущенных обновлений

Для [[Wasaby-Patches|патчей]]: ReleaseManager → Хоттабыч (применение) → лёгкое обновление.

## Схема БД (ключевые таблицы и индексы)

**Таблицы:**
- `План выпуска` — общая информация и статус плана
- `Пункт плана` — иерархическая единица (этапы и работы)
- `Работа` — описание сценария задачи релиза
- `Зависимости работ` — зависимости между работами (переносятся между планами)
- `Зависимости пунктов плана` — внутриплановые зависимости (с этапами)
- `Статус работы по облаку` (`CloudWorkStatus`) — статусы в разрезе каждого облака
- `Задача` (`Task`) — экземпляр запуска работы (у одной работы может быть несколько задач)
- `WorkRelationship` — зависимости между элементами плана

**Индексы:**
```
Plan(Uuid)
Plan(Status)
Work(Link)
Work(GuidFromLink)
Work(Uuid)
Work(Plan, Hierarchy, Order)
Task(UpdateId, CloudAlias)
Task(Work, CloudAlias)
CloudWorkStatus(Work, CloudAlias)
WorkRelationship(Plan, Target)
```
