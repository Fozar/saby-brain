# Планы выпуска — Release Plans

Сервис: [vm.sbis.ru](https://vm.sbis.ru/) → аккордеон «Планы выпуска»

Планы выпуска — инструмент контроля обновления и тестирования сервисов Saby/SBIS. Хранит всю информацию о обновляемых сервисах и работах для конкретного дня выпуска. Разворачивается один на несколько облаков, взаимодействует с сервисами Управления облаком и Управления версиями из других облаков.

## Роли участников

| Роль | Обязанности |
|------|-------------|
| Релиз-менеджер | Создаёт план на каждый день; перед заездом хот-фиксов снимает кирпич |
| Выпускающий | Выгружает дистрибутивы в боевой репозиторий при проставленных билде И статусе |
| Тестировщик | Проставляет билд и статус «Проверено» для обновлений/разворотов/новых сервисов |
| Дежурный сборщик fix | Помогает при незасвязке обновления с пунктом плана |

> Роль «Тестировщик прикладных сервисов» запрашивается через «Задание дежурному СЭ».

## Стенды и их семантика

| Стенд | Назначение |
|-------|-----------|
| pre-test | Ранние тесты |
| test | Основной тестовый стенд |
| fix | Продакшн РУ-облако (Россия) |
| fix-kz | Продакшн без РУ-облака (другие регионы) |
| prod | Боевой продакшн |

**Правило статуса:**
- `Статус FIX` — приложение есть на бою в РУ-облаке (обязательно включая РУ)
- `Статус FIX-KZ` — приложение отсутствует в РУ-облаке, но присутствует в других региональных

## Проставление билда и статуса

1. Найти нужный план (поиск по дате или ссылка в рассылке выпускающего)
2. В строке поиска плана найти работу (по имени приложения / семейства / ФИО / ссылке)
3. Кликнуть на название приложения
4. Нажать иконку **замка** у fix или fix-kz — фиксирует версию и билд со стенда
   - Иконка замка появляется только если версия из работы уже есть на стенде
5. Поставить статус **«Проверено»** напротив fix или fix-kz

> [!warning] Важно
> - БИЛД фиксируется **один раз** для всех стендов в пределах одной работы
> - Не устанавливайте билд и статус заранее — это заблокирует авто-обновление в день выпуска

## Альтернативные способы фиксации билда

- Кликнуть на номер версии у работы → дописать билд вручную
- Открыть карточку обновления на редактирование → выбрать билд из справочника (горизонтальные полоски в конце строки; содержит все версии фиксового репозитория)
- Если билдов нет в справочнике — дописать вручную

## Автоматическое обновление сервисов на fix

Обновления проходят **раз в час** с 10:00 до 19:00 (в 10:00, 11:00, ..., 19:00).

Последнее автоматическое обновление — 19:00. При прохождении авто-тестов в 8:00–9:00 (online) автоматических обновлений нет.

### Условия для авто-обновления

- Сервис находится в плане выпуска на данный день
- Если в поле «Версия» не указан конкретный билд — обновляется на свежий билд при условии:
  - Сервис обновлён на test на этот билд
  - В плане не стоит «кирпич» на работе
- Статус «Проверено» по стенду fix **отключает** авто-обновление
- Если указаны версия И билд — сервис не обновится выше этого билда автоматически

### Стоп-лист (не обновляются автоматически)

```
cloud-ctrl, sharded-logs, rights-mng, scheduler, config-ctrl, admin-repo,
version-manager, filestorage, SBISDisk, диспетчер, нехоттабычные сервисы,
новые сервисы, основные сервисы, необлачные продукты,
обновление пакетных данных
```

### Отключение авто-обновления для одного сервиса

Поставить тег `#noauto` в комментарий к пункту плана. После исправления критичной ошибки — удалить тег.

## Механизм «кирпич»

«Кирпич» (блокировка) запрещает обновление сервиса на стенд fix/fix-kz.

**Снятие кирпича:**
- Для **хот-фиксов** — релиз-менеджер снимает перед заездом версии на тестовый стенд
- Для **мажорных выпусков** — тестировщик ответственного участка снимает после рассылки от релиз-менеджера

Процедура снятия:
1. Установить фильтр по стенду fix
2. Найти нужный сервис
3. Навести курсор или отметить галкой → кнопка «Разрешить запуск обновления»

## Новые сервисы

- **Новое приложение** — указать статус, версию И билд
- **Новый сервис в существующем приложении** — указать только статус

## Расписание выполнения работы на бою

В режиме фильтра по стенду у каждой работы отображается:
- **Синий** — время запуска работы
- **Зелёный** — время запуска фазы обновления

Если есть **время запуска фазы обновления** — это примерное время начала выполнения.
Если есть только **время запуска работы** — это примерное время.

> У скриптов и разворотов данных время запуска фазы обновления может отсутствовать.

## Проверка статуса обновления

В плане → выбрать интересующий стенд в прогресс-баре или фильтре → у пункта плана будет показан статус выполнения работы.

## Изменение ответственного тестировщика

Изменение в текущем плане недостаточно — план создаётся из **шаблона**. Нужно оформить поручение администратору (Абакова С.) для внесения изменений в шаблон.
