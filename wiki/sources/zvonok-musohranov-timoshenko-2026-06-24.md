## Тема 1 — MR CreateLead → CreateStub (задача 06152561)

**Контекст:** Тимошенко добавил автоматическое создание корешка при создании CRM-сделки в `ReferralProgram.CreateLead`. Мусохранов делает ревью.

**Вопрос по тестированию:**  
Мусохранов не увидел явных сценариев тестирования в задаче/MR. Тимошенко пояснил, что тест есть — создание заявки через оффер (`create_lead.py`, 7 тестов). Мусохранов ответил, что сценарий непонятен из описания задачи: «не воспринял, почитал — ничего толком проверить нельзя, только из-под кода».

**Действие:** Написать сценарии в явном виде в MR/задаче: какой оффер, что вызывается, какой результат ожидается.

**Ключевой риск — старые рефералки:**  
Исторически корешок (`ВидЦеныДокумент`) создавался только при *завершении* сделки. Теперь он создаётся при *создании* сделки (статус 10 = в работе). Мусохранов опасается, что:

- Виджеты или реестры старой рефералки фильтруют только «выполненные» корешки, и новый корешок в статусе 10 «вылезет» там, где не должен
- Сделка не прошла все этапы, а корешок уже есть — возможны разъезды в расчётах и отображении

**Сценарии для покрытия (обязательны):**
1. Создали корешок → сделка дошла до финального статуса → убедились, что корешок не повредил старую рефералку (виджеты, реестры)
2. Корешок в статусе 10 → проверить, что он не отображается там, где ожидаются только завершённые корешки

**Итог по коду:** замечаний по коду нет; нужно дописать сценарии и на ревью отправить снова.

## Тема 2 — MR GetLeadPeriodList: LeadCount (задача 04307081)

**Контекст:** Добавление колонки «Количество лидов» в список периодов мастера рефпрограммы. Текущая реализация: для каждого источника (партнёра) идём в marketing-API за статистикой в цикле.

**Проблема производительности:**  
Мусохранов: «10 межсервисных взаимодействий — это не взлетит. Даже если каждый метод 200 мс, итого ~2 с. Это шляпа. Нам такое не нужно. В цикле — это догма, нельзя.»

**Принятое решение:**

> Считать лиды и вознаграждения ТОЛЬКО из `ВидЦеныДокумент` (по корешкам), без обращения к marketing-API в цикле.

Разбивка по типу программы:
- **SabyBank / табмод-программы** → ВидЦеныДокумент (уже есть данные)
- **Стандартные программы** → маркетинг (один вызов, не в цикле по партнёрам)

**Оптимизация запроса:**  
Текущий код делал два отдельных подзапроса к ВЦД (один для лидов, один для вознаграждения). Мусохранов: «ровно один запрос к ВидЦеныДокумент — по нему же считаешь и количество корешков, и SUM(бонус)».

**Что вырезается:** блок кода, получающий идентификаторы источников всех партнёров + цикл по источникам → статистика из маркетинга. Остаётся единый источник для корешков.

**Будущее:** когда сделки (лиды) перейдут на корешки, маркетинговая интеграция вернётся — но уже без цикла.

**Итог:** поправить тест → отправить на сборку.
