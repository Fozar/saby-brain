# Loyalty Iterative Loading — ТД и предложение «Общие решения»

Техническая документация механизма итеративной (порционной) загрузки реестров лояльности по пункту
плана **594287653** (поручение №05145004, проект [[Loyalty-React-Migration-Project]]). Описать механизм
нужно в четырёх подсистемах: Промокоды, Дисконтные карты, Бонусы, Реферальная система.

## Решение по размещению

Механизм **один и тот же** во всех четырёх подсистемах, поэтому (по итогам совещания 10.06.2026):
описать его **один раз** и из остальных подсистем **ссылаться**, без дублирования.

**Мастер-описание — подсистема Бонусы, документ «Алгоритмы и процессы».** Остальные подсистемы получают
короткий раздел со ссылкой на Бонусы + свои особенности (метод, класс, курсор, параметры `GlobalParams`):

| Подсистема | Документ БЗ | Роль |
|------------|-------------|------|
| Бонусы | Алгоритмы и процессы | **Мастер** (полное описание + схемы) |
| Дисконтные карты | Алгоритмы и процессы | Ссылка + особенности |
| Промокоды | API подсистемы Промокоды | Ссылка + особенности |
| Реферальная бонусная программа | API подсистем | Ссылка + особенности |

**Почему мастер — Бонусы:** общий движок реестров покупок принадлежит программам лояльности; реферальный
реестр наследуется прямо от бонусного (`ReferralBonusSaleListWithCursorIterative(BonusSaleListIterative)`);
React-компонент реестра покупок Бонусов переиспользуют Промокоды, Реферальная и Акции. Бонусы — корневая
точка механизма.

**Артефакт:** `price-formation/docs/loyalty/_ТД Итеративная загрузка/` — мастер-md, 3 ссылочных md,
README с картой вставок в БЗ и 3 схемы в формате **drawio** (архитектура, цикл загрузки, иерархия
классов). Стиль документации выдержан под существующие PDF БЗ (H1/H2, прозовое изложение, callout-боксы,
таблицы, диаграммы-картинки).

## Фактическое состояние механизма на ветке 4100

Документация написана под **rc-26.4100**, где фича-флаг `loyalty_it_nav` уже выпилен
(см. [[Feature-Flag-Removal-LOYALTY-IT-NAV]]). Поэтому в ТД флаг не упоминается — итеративная загрузка
описана как штатный и единственный режим. Изменения иерархии классов относительно 3211:

- Не-итеративные базовые классы удалены: `BonusSaleListWithCursor`, `ReferralBonusSaleListWithCursor`,
  `PromoCodeSaleListWithCursor` больше не существуют — остались только итеративные классы.
- `BonusSaleListIterative(IterativeBlockSizeEmaMixin, ListWithCompositeCursor, SaleListWithCursor)`.
- `ReferralBonusSaleListWithCursorIterative(BonusSaleListIterative)` — теперь поверх бонусного итеративного.
- `PromoCodeSaleListIterative(IterativeBlockSizeEmaMixin, ListWithCompositeCursor, SaleListWithCursor)`.
- `DiscountCardListWithCursorIterative(IterativeBlockSizeEmaMixin, ListWithCompositeCursor)`.
- `PromotionSaleListWithCursor(IterativeBlockSizeEmaMixin, SaleListWithCursor)` — Акции (вне области ТД).
- **`Bonus.GetClientListWithStats` отказался от итеративности** — реестр клиентов перешёл на обычную
  курсорную навигацию через общий хелпер `get_list_by_cursor` (`priceformationcommon.core.cursor_navigation`),
  без адаптивного блока/EMA. В ТД этот реестр не упоминается; единственный итеративный реестр Бонусов —
  `Bonus.GetSaleList`.

`GlobalParams` (размер блока + EMA) заданы пометодно: `Bonus.GetSaleList.*`,
`DiscountCard.GetListWithStats.*`, `PromoCode.GetSaleList.*`, `ReferralBonus.GetSaleList.*`,
`Promotion.GetSaleList.*`.

## Предложение: раздел БЗ «Общие решения»

Кейс показал слабое место: сквозной механизм пришлось «прописать» в произвольно выбранную подсистему-владельца
(Бонусы) и линковать из остальных. Предложение Федько — завести отдельное место уровня **«Подсистемы
Лояльности / Общие решения»** (документ или папка) под механизмы, не принадлежащие ни одной подсистеме,
а переиспользуемые всеми; из подсистем — только ссылки.

**Выгода:** нет дублирования; нет случайного владельца; единый источник правды; проще найти.

**Кандидаты на вынос (обсуждается):**

*Точно сквозное:*
- Итеративная (порционная) загрузка реестров (текущий кейс).
- Курсорная навигация реестров (слой под итеративной, общий хелпер).
- Обогащение реестров покупок данными Розницы/СУ (общий движок реестра покупок).
- Скрытие конфиденциальных данных в реестрах (ФИО/контакты покупателя и продавца).

*Под вопрос:*
- `ВидЦеныДокумент` как единое хранилище событий лояльности + `EffectiveDate` как маркер завершения.
- Синхронизация online↔offline через брокер.
- Фоновые длительные операции через DWC.

**Статус:** Федько ранее ответил «выделять или нет и куда размещать — приходи»; название/расположение
и финальный список выносимого определяются на встрече. Сдача доки по проекту — на Федько (либо эта, либо
итоговая от Ютман Элины).
